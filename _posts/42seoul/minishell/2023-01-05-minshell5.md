# minishell

> Shell을 만들어라.
> 

과제를 시작했을 때는 상당히 막연했다. “Shell을 직접 만들라니?” Shell이라고는 여지껏 터미널을 켜면  실행되는 당연한 기능 정도로만 생각했었다. 그렇기에 어떤 방식으로 동작해야 할지, 무슨 기능을 만들어야 할지 감도 잡히지 않았다. 다행히 같은 고민을 하던 스터디원들이 있었고 다른 카뎃이 열어준 지식 품앗이에 참여해 전체적인 로직과 주요 개념에 대해 배울 수 있었다.

시작하기에 앞서 과제에서 사용할 자료구조를 정하였다. “mandatory만 진행하려면 연결 리스트로, bonus까지 하려면 트리 구조로 구현하라”는 조언이 많았는데 우리팀은 mandatory만 할 것임에도 트리 구조를 선택했다. 이 선택의 이유가 몇 가지 있었는데, 우선 mandatory만 진행할 것임에도 트리 구조를 선택하는 것이, 과제를 마친 이후에 재도전을 하게 될  때 훨씬 확장성이 좋다고 생각했다. 또한 이전 프로젝트인 libft, get next line(나는 배열을 사용하긴 했다), push swap 등에서 이미 다루었던 연결 리스트 개념과는 다르게 트리 구조는 처음 접하는 개념이었고, 관련 개념을 익히기에 이만한 과제가 없다고 생각했었다.

파싱 순서는 한 가지로 정하는 것이 아닌, 과제를 해나가며 계속해서 변경하였다. 매뉴얼에도 정확한 순서가 나와있지 않아서 어떤 경우에 확장을 진행하고 안하는지, 따옴표 처리가 되는지 등의 케이스를 bash로 직접 확인해가며 새로운 케이스가 발견될 때마다 순서를 바꾸었다. 

생각해보면 여태 shell을 사용하면서 이 프로그램의 동작방식을 깊게 생각해본 적은 없었다. 그렇기에 이번 과제를 하면서 많은 고생을 했던 것 같다. 내가 구현한 shell 빌트인인 cd는 단순하게 경로를 이동하기만 하는 것이 아닌, path가 존재하는지, 디렉토리인지를 확인해서 이동하고, 환경변수 PWD, OLDPWD를 변경하는 등 많은 역할을 하고 있었다. 이처럼 지금껏 별 생각 없이 사용하던 cd, echo, export, exit 등의 빌트인 명령어가 어떤 방식으로 작동하는지를 살펴볼 수 있었다. 또한 파싱 과정의 변수 확장, 따옴표 제거, 실행 과정의 리다이렉션에 따라 stdin, stdout을 변경하고 프로세스를 생성하는 등 shell의 기본적인 기능과 동작 방식을 배울 수 있었다.

# 협업

이번 미니셸 과제는 두 명이 한 팀을 이루어 하는 과제였다. 이전 과제에서 팀을 이루어 스터디를 진행하긴 했지만 코드를 공유하며 같이 무언가를 만들어가는 것은 본과정 뿐 아니라 개발 공부를 시작한 이래로도 처음이었다. 그렇기 때문에 시작하기에 앞서 걱정도 많았지만, 동시에 팀원으로부터 새로운 것들을 배울 수 있겠다는 기대도 있었다.

팀원인 rhong님은 소프트웨어 전공생으로 개발 경력이 긴 카뎃이었다. 그렇기에 내가 헷갈려 했던 연결 리스트, 트리 등의 자료구조 관련 개념이나 Bash 명령어에 대해 많이 배울 수 있었다. 과제의 이름인 “미니”셸 때문에 내가 우리 프로그램의 불완전함 앞에서 타협하려 할 때마다, “구현할 기능의 수를 적게하되, 구현하는 기능에 대해서는 완전하게 하자”고 나를 설득해주었다.

우리 팀은 미니셸 전체를 같이 만드는 대신, 파싱 파트와 실행&빌트인 파트로 나누어 각 파트를 나와 팀원이 나눠서 구현하는 방식으로 분업을 했다. 또한 here-doc과 트리구조 파싱 부분과 자료구조 관리는  팀원이 맡아주었고 빌트인의 cd와 코드 정리는 내가 맡았다. 각각의 파트별로 브랜치를 생성해서 충돌을 방지했고 어느정도 구현될 때마다 서로에게 코드를 설명해주고 코드 리뷰를 하였다. 각자의 파트를 구현하는 초기에는 보통 슬랙을 통해 소통했고, 파트를 합치고 프로그램의 에러를 찾는 마지막 2주동안은 클러스터에서 만나서 직접 문제를 해결했다.

![Untitled](/assets/img/42seoul/minishell/mini5/Untitled%201.png)

한달 반 동안 과제를 하면서 우리는 다양한 문제에 직면했다. 우선 분업을 하다보니 서로의 코드를 이해하기가 쉽지 않았다. 심지어 pipex 과제를 하지 않았던 나로서는 실행 파트를 이해하는 것이 한층 더 어려웠다. Pipe, dup 등의 개념은 생소했고 here-doc의 임시파일과 식별자 관련 내용은 아무리 다시 보아도 이해가 되지 않았다. 다행히 허용 함수들을 짚어본 후에 전체적인 흐름을 잡고 bash로 다양한 테스트를 하니 이해할 수 있었다. 그리고 나 또한 팀원에게 설명을 해주기 위해 그림을 그려보니 코드에서 모호했던 부분을 확실하게 잡을 수 있었던 것 같다.

또한 팀원의 블랙홀이 임박했던 문제도 있었다. 애초에 팀원은 과제를 시작할 당시에도 팀원의 블랙홀이 그리 여유롭지는 않았다. 거기에 미니셸을 끝낼 것이라고 예상한 기간이 다 되도록 과제는 끝날 기미가 보이지 않았고, 결국 철학자 과제를 먼저 해결하느라 미니셸을 잠시 중단하였다. 그렇지만 나도 내 일이라는 생각으로 해당 과제를 도왔고 팀원도 밤을 새가며 철학자 과제에 매달린 덕에 4일만에 다시 미니셸을 재개할 수 있었다.

마지막으로 구현 과정에서도 어려움이 있었다. 몇 주간 작성했던 파싱 부분의 따옴표 처리(토큰화, 확장, 따옴표 제거) 로직이 완전히 잘못되었다는 것을 깨닫고 멘탈이 많이 흔들렸다. 기존의 방식을 조금 바꾸는 걸로는 제대로된 처리가 불가능하다는 것을 알았음에도 아까운 마음에 쉽게 코드를 날릴 수가 없었다. 하지만 joonhan님의 설명을 듣고 방향을 완전히 잡고 나니 몇 주가 걸렸던 부분을 고작 이틀만에 복구할 수 있었다.

# 개선 사항

미니셸은 여지껏 가려져있던 잘못된 습관들이 많이 수면위로 올라왔던 프로젝트였던 것 같다.

## 개발 방식의 문제

![https://blog.crisp.se/wp-content/uploads/2016/01/mvp.png](https://blog.crisp.se/wp-content/uploads/2016/01/mvp.png)

우선 처음부터 너무 방대한 계획을 세우고 이것저것 신경쓰다보니 실행 파일 생성이 너무 늦어졌다. 위 그림처럼 처음엔 조촐한 스케이트 보드를 만들고, 기능을 하나씩 추가해가며 킥보드, 자전거, 오토바이를 거쳐 자동차를 완성시켰어야 하는데 그러지를 못했다. 토큰화 파트를 구현하기도 힘든데 norminette을 따르느라 변수 5개 이하 선언, 함수 5개 이하 사용에 신경쓰느라 정작 중요한 기능 구현은 뒷전이 됐다. 스터디원인 joonhan님 팀의 경우, Makefile을 먼저 만들어두고 파싱 단계의 토큰화 파트를 구현한 뒤에 실행 파일을 생성해서 해당 파트를 체크하고, 변수 확장 파트를 구현한 뒤에 컴파일해서 해당 파트를 체크하는 식으로 진행했다. 그렇게 하다보니 우리 팀보다 훨씬 즐겁게 코딩하는 것 같았고 결과적으로는 파싱부를 우리 팀보다 일주일가량 먼저 완성시켰다. 실행 파일 컴파일을 늦게 하다보니 에러 체크를 할 시간도 부족했고 이러한 문제는 첫 번째 평가에서 쏟아지는 문제들로 돌아왔다. 대표적으로 공백문자로만 이루어진 입력을 걸러주지 못한 것이나 PATH가 unset됐을 때 segfault가 발생하는 문제는 테스트를 조금만 했어도 해결이 가능한 문제였다.

## 디버깅 실력의 부족

![Untitled](/assets/img/42seoul/minishell/mini5/Untitled%202.png)

또한 위와 같은 궤에 있는 문제인데, 작성한 코드에 대한 디버깅이 부족했다. 이 문제는 프로젝트의 크기가 작은 여태까지의 과제에서는 크게 드러나지 않았다. 지금까지의 과제에서는 문제가 발생하면 확인해야 하는 파일이 적으면 1개에서 많아봐야 20개가 채 되지 않았기에 printf 디버깅만으로도 충분히 문제를 파악해 해결할 수 있었다. 하지만 이번 과제의 경우 파일이 60개 가량이 되고 기능 각각에 대한 이해도도 전보다 부족했기 때문에 printf 만으로 문제를 파악하기란 여간 힘든일이 아니었다. 팀원 둘 다 디버거 사용에 미숙해서 이렇게 디버깅에 어려움을 겪고 printf로 찍어보다가 스터디 모임이 있는 날에 디버거를 잘 다루는 다른 스터디원들에게 도움을 받기를 반복했다. 결국 프로젝트 막판이 되어서야 스터디원들로부터 배운 lldb, vscode 디버거를 조금씩 다뤄가며 문제를 해결할 수 있었다.

## 코드 정리의 부재

내가 작성한 함수를 글이나 그림으로 그때그때 정리하지 않아서 생긴 문제도 있었다. 변수 확장 파트는 워낙 구현 방식이 복잡한데다가 구현 이후에 norminette 규칙에 맞춰서 함수를 쪼개느라 나중 가서는 나조차도 이해하지 못하는 코드가 되어 있었다. 나부터가 이해하지 못했으니 팀원에게 설명은 더더욱 할 수 없었다. 코드를 완벽하게 작성한 후에 한번에 정리하자고 생각했던 것이 화근이었다. 이제부터는 불완전한 코드여도 나중에 이해할 수 있도록 글이나 그림으로 반드시 기록을 남겨야할 것 같다.
