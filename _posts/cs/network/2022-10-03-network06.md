- 네트워크 계층의 데이터그램이 링크를 통해 전송되기 위해 링크 계층 프레임에서 어떻게 **캡슐화**될까?
- 통신 경로 상에서 **서로 다른 링크**는 다른 링크 계층 **프로토콜**을 사용할까?
- 브로드캐스트 링크에서의 **전송 충돌**은 어떻게 해결될까?
- 링크 계층에는 **주소 지정(addressing)** 과정이 있을까? 만약 있다면, 이전에 배운 **네트워크 계층**의 주소 지정과 어떻게 작동할까?
- 스위치와 라우터의 정확한 차이점은 뭘까?

**링크 계층 채널**에는 두 가지 종류가 있다.

- **브로드캐스트(broadcast) 채널**
    - 무선 LAN, 위성 네트워크, 하이브리드 광섬유 동축 케이블(hybrid fiber-coaxial cable, HFC) 등의 네트워크의 여러 호스트를 연결한다.
    - 여러 호스트가 동일한 브로드캐스트 통신 채널에 연결되어있기 때문에 프레임 통신을 조정하기 위해 **매체 접근 프로토콜(medium access protocol)**이 필요하다.
        - 중앙 제어장치(central controller)를 통해 통신을 조정
        - 호스트끼리 알아서 통신을 조정
- **점대점(point-to-point) 통신 링크**
    - ex. 장거리 링크로 연결된 두 개의 라우터, 사무실 컴퓨터와 근처 이더넷 스위치 사이의 링크

# Introduction to the Link Layer

**용어 설명**

- 노드 (node) : 링크 계층 프로토콜을 실행하는 모든 기기. 호스트, 라우터, 스위치 , WiFi 액세스 포인트 등이 있다.
- 링크 (link) : 인접한 노드를 통신 경로를 따라 잇는 통신 채널
    - 데이터그램이 출발지 호스트에서 목적지 호스트까지 이동하기 위해서는 종단간 경로의 **개별적인 링크**로 이동해야 한다.
    - (그림 6.1 참고)아래 회사 네트워크에서 무선 호스트가 서버에 데이터그램을 보낸다고 가정하자. 이때, 데이터그램은 여섯 개의 링크를 거쳐갈 것이다.

## The Services Provided by the Link Layer

**링크 계층 프로토콜**이 제공하는 서비스는 다음과 같다.

- **프레이밍 (Framing)**
    - 거의 모든 링크 계층 프로토콜은 **네트워크 계층 데이터그램**을 링크를 통해 전송하기 전에 **링크 계층 프레임** 안에 **캡슐화**한다.
    - 프레임은 데이터 필드(네트워크 계층 데이터그램이 들어감)와 다수의 헤더 필드로 구성된다.
- **링크 접근 (Link access)**
    - 매체 접근 제어(Medium Access Control, MAC) 프로토콜은 **프레임이 링크로 전송되는 규칙**을 지정한다.
    - 점대점 링크에서 : MAC 프로토콜은 링크가 비어있기만 하면 송신자가 보낼 수 있게 한다.
    - 브로드캐스트 링크에서 : MAC 프로토콜은 여러 노드에서의 프레임 전송을 조정한다.
- **신뢰할 수 있는 전송 (Reliable delivery)**
    - 각 네트워크 계층 데이터그램이 **링크에서 오류없이 이동**하는 것을 보장한다.
    - **확인 응답(acknowledgment)**과 **재전송(retransmission)**을 통해 이루어진다.
- **오류 검출과 수정 (Error detection and correction)**
    - 수신 노드는 비트 오류가 발생한 프레임을 받을 수 있다. (0이 1로 되거나, 1이 0으로 되는 경우)
    - 송신 노드는 프레임에 오류 검출 비트(error-detection bits)를 포함시키고, 수신 노드는 오류 체크를 한다.
    - 수신자는 비트 오류를 탐지하는데 그치지 않고 프레임의 어디에서 오류가 발생했는지도 찾는다. 그 후 오류를 수정한다.

## Where Is the Link Layer Implemented?

아래 그림 6.2는 전형적인 **호스트 구조**를 나타낸 것이다.

대부분의 경우 링크 계층은 **네트워크 어댑터**(**네트워크 인터페이스 카드(NIC)**라고도 불림)에 구현되어 있다.

네트워크 어댑터의 중심에는 **링크 계층 제어 장치**가 있다. 이건 주로 하나의 특별한 용도의 칩으로 다양한 서비스(프레이밍, 링크 접근, 오류 검출 등)를 제공한다.

### 동작 과정
- **송신 측** 
    1. 제어 장치는 생성되어 호스트 메모리에 저장되어 있는 데이터그램을 받는다.
    2. 데이터그램은 링크 계층 프레임 안에 캡슐화된다.
    3. 링크 접속 프로토콜에 따라 통신 링크에 프레임을 전송한다.
- **수신 측**
    1. 제어 장치는 전체 프레임을 받는다.
    2. 프레임에서 네트워크 계층 데이터그램을 추출한다.

# Error-Detection and -Correction Techniques

이 장에서는 비트 오류를 검출하고 수정하는 몇 가지 단순한 기술들을 검토할 것이다.

송신 노드에서, 데이터(D)는 비트 오류를 방지하기 위해 오류 검출&수정 비트(EDC)가 추가되어있다.

수신 노드에서, D’와 EDC’이 수신된다. 전송 중에 비트 플립(bit-flip, 비트가 바뀌는 일)이 발생해서 기존의 D, EDC와 다를 수 있다.

**수신자**의 역할은 수신된 D’와 EDC’을 통해 **D와 D’이 동일한지**를 확인하는 것이다.

오류 검출 비트가 있음에도 **탐지되지 않은 비트 오류(undetected bit errors)**가 있을 수 있다.

이러한 오류를 탐지하는 세 가지 방법은 parity checks, checksumming methods, cyclic redundancy checks이다.

![Untitled](/assets/img/cs/network/network06/Untitled%202.png)

## Parity Checks

오류 검출을 위한 제일 단순한 방법은 **하나의 패리티 비트(parity bit)**를 사용하는 것이다.

(그림 6.4)전송되어야 하는 정보가 d개의 비트가 있다고 하자.

- **짝수 패리티 방식 (even parity scheme)**
    - 송신자는 전체 d+1 개의 비트 중 1의 개수가 짝수인지에 대한 값을 추가적인 비트로 포함시키면 된다.
- **홀수 패리티 방식 (odd parity scheme)**
    - 전체 비트 중 1이 홀수 개인지에 대한 값

![Untitled](/assets/img/cs/network/network06/Untitled%203.png)

위 방식은 하나의 오류가 발생했을 때는 유효하지만, **짝수 개의 오류**가 발생하면 알 수 없다는 문제가 있다.

실제로 오류는 하나씩 발생하기보다는 여러 개가 한번에 터지는 경향이 있다.

아래는 **이차원 패리티 방식(two-dimensional parity scheme)**으로, 오류 검출 비트는 끝 쪽의 `i+j+1`개의 비트로 구성된다. 이 경우 오류가 발생하면 column과 row 각각의 parity bit를 통해 오류를 탐지할 수 있어서 비트 오류가 **여러 개인 경우**에도 탐지할 수 있다.

![Untitled](/assets/img/cs/network/network06/Untitled%204.png)

## Checksumming Methods

이 방식에서는 위의 그림 6.4의 d개의 비트가 일련의 k-비트 정수로 취급된다. 단순한 체크섬 방법은 이 k-비트 정수를 그냥 합해서 나온 결과를 오류 검출 비트로 사용하는 것이다.

## Cyclic Redundancy Check (CRC)

요즘에 이 방식을 기반으로 한 오류 검출 기술이 많이 쓰이고 있다.

CRC 코드는 다항식 코드 (polynomial code)라고도 알려져 있다. 전송될 비트열을 다항식으로 볼 수 있기 때문이다.

- 동작 과정
    1. 송신 노드에서 수신 노드로 전송하려는 d개의 비트로 이루어진 데이터 D가 있다고 하자.
    2. 송신자와 수신자 모두 우선 `r + 1` 비트 패턴(**generator**, G)에 동의해야 한다.
    3. 송신자는 r개의 추가적인 비트(R)를 골라서 D에 붙여 d+r 비트 패턴이 모듈러 2 연산에 의해 G로 나누어 떨어질 수 있게한다.

# Multiple Access Links and Protocols

앞에서 네트워크 링크에는 두 종류가 있다고 했다. 점대점 링크와 브로드캐스트 링크.

- 점대점 링크 : 점대점 프로토콜 (PPP), 고수준 데이터 링크 제어(high-level data link control, HDLC)가 있다.
- 브로드캐스트 링크 : 이더넷(ethernet), 무선 랜(wireless LAN)이 있다.

이 장에서는 링크 계층의 중심적인 문제인 **다중 접근 문제(multiple access problem)**에 대해 알아볼 것이다. 이 문제는 여러 송신자와 수신자 노드가 공유 브로드캐스트 채널에 접속하는 것을 어떻게 조정할지에 대한 것이다.

**다중 접근 프로토콜(multiple access protocol)**에서 각 노드는 자신들의 공유 브로드캐스트 채널로의 전송을 통제한다.

두 개 이상의 노드가 동시에 여러 프레임을 전송하게 되면 모든 노드가 여러 프레임을 동시에 수신하게된다. 이것을 “전송된 프레임이 **출돌했다(collide)**”고 한다. 충돌이 발생하면 관련된 모든 프레임이 손실되고 브로드캐스트 채널도 낭비된다.

이를 막기 위해 **active한 노드에서의 전송을 조정**하는 것이 필요하다. 이게 다중 접근 프로토콜의 역할이다.

여태까지 여러 프로토콜이 구현되었는데, 다음과 같이 세 가지로 분류할 수 있다.

- **channel partitioning protocols**
- **random access protocols**
- **taking-turn protocols**

전송률이 R인 브로드캐스트 채널에 대한 다중 접근 프로토콜은 다음과 같은 이상적인 특성을 가져야 한다.

1. 데이터를 보내야하는 노드가 하나뿐이라면, 노드는 R bps의 처리량을 갖는다.
2. 데이터를 보내야하는 노드가 M개라면, 각 노드는 R/M bps의 처리량을 갖는다.
    - 각 노드가 항상 R/M의 속도를 가질 필요는 없다.
    - 특정 시간 동안의 평균 전송률이 R/M이면 된다.
3. 프로토콜은 분산된다.
    - 마스터 노드가 없다.
4. 프로토콜은 단순하며 구현 비용이 저렴하다.

## Channel Partitioning Protocols

챕터 1에서, 브로드캐스트 채널의 대역폭을 여러 노드와 공유하기 위해 **TDM(time-division multiplexing)**과 **FDM(frequency-division multiplexing)**을 사용한다는 것을 배웠다.

N개의 노드가 있고 전송률이 R bps인 채널을 가정해보자. (그림 6.9 참고)

### TDM

TDM은 시간을 **시간 프레임(time frame)**들로 쪼개고, 각 시간 프레임을 N개의 **시간 슬롯(time slot)**들로 쪼갠다. 그 후에 각 시간 슬롯은 N개의 노드 중 하나에 할당된다. 만약 한 노드가 패킷을 전송하려면, 패킷의 비트들을 **할당된 시간 슬롯동안 전송**한다. 아래 그림은 노드가 네 개일 때의 예시다.

TDM은 충돌을 없애고 공평하다는 장점이 있다. 하지만 다음 두 가지 문제점이 있다.

1. 노드 중 패킷을 전송할 노드가 하나뿐일 때에도 전송률이 R/N으로 제한된다.
2. 유일하게 전송하는 노드일 때도 자신의 전송 순서를 기다려야 한다.

![Untitled](/assets/img/cs/network/network06/Untitled%206.png)

### FDM

R bps의 채널을 여러 주파수(각각의 대역폭은 R/N)로 나눈다. 즉, 하나의 큰 채널을 N개의 작은 채널들로 나눈다.

- **장점**
    1. 충돌을 없앤다.
    2. 대역폭을 N개의 노드간에 공평하게 나눈다.
- **단점**
    - 유일하게 전송해야하는 노드인 경우에도 대역폭이 R/N으로 제한된다.

### 코드 분할 다중 접속 (code division multiple access, CDMA)

- 동작 과정
    1. CDMA는 서로 다른 코드를 각 노드에 할당한다.
    2. 각 노드는 고유한 코드를 사용해서 자신이 보낼 데이터 비트를 인코딩(encoding)한다.

코드가 제대로 선택됐다면, CDMA 네트워크 상의 노드들은 동시에 전송하고 각각이 제대로 수신될 수 있다.(다른 노드에 의해 방해를 받더라도)

거의 무선 채널에서만 사용된다.

## Random Access Protocols

Random access protocol에서 전송하는 노드는 항상 채널의 최대 전송률로 전송한다.

만약 충돌이 발생한다면, 이와 연관된 노드들은 충돌이 없어질 때까지 자신의 **프레임(패킷)을 재전송**한다.

노드는 프레임을 즉시 재전송하는 것이 아니라, **임의의 시간동안 기다린(random delay)** 후에 재전송한다. 이를 통해 충돌을 피할 수 있다.

### Slotted ALOHA

가장 단순한 random access protocol 중 하나이다.

- **가정 사항**
    - 모든 프레임은 정확히 L개의 비트로 구성된다.
    - 시간은 L/R 초 크기의 슬롯으로 나뉜다. (슬롯의 시간=하나의 프레임을 전송하는 시간)
    - 노드는 슬롯이 시작될 때만 프레임을 전송한다.
    - 노드는 동기화되어 각 노드는 슬롯이 언제 시작될지 안다.
    - 두개 이상의 프레임이 슬롯에서 충돌하면, 모든 노드는 슬롯이 끝나기 전에 충돌을 감지한다.
- **동작 과정** (p : 확률, 0 ≤ p ≤ 1)
    - 노드에 새롭게 보낼 프레임이 있다면, 다음 슬롯이 시작될 때까지 기다렸다가 모든 프레임을 슬롯안에 전송한다.
    - 충돌 X : 노드가 프레임을 **성공적**으로 전송한 것이므로 재전송을 고려할 필요가 없다.
    - 충돌 O : 노드는 슬롯이 끝나기 전에 **충돌을 감지**한다. 노드는 p의 확률로 다음 슬롯에 프레임을 **재전송**한다.(프레임이 충돌없이 전송될 때까지)

그림 6.10은 active한 노드가 여러 개일 때의 예시이다.

이 때, 다음 두 가지를 고려해야한다.

1. 여러 active한 노드가 있을 때, 특정 슬롯에 충돌이 발생하여 버려질 것이다.(wasted)
2. 다른 특정 슬롯은 확률적 전송 정책의 결과로 전송을 자제하기 때문에 비어있을 것이다.(empty)

### ALOHA

slotted ALOHA는 모든 노드가 슬롯이 시작될 때 전송이 시작되도록 동기화할 것을 요구했다.

초기 ALOHA는 unslotted이며 완전히 분산된 프로토콜이다. (pure ALOHA)

- **동작 과정**
    - 프레임이 처음 도착하면, 노드는 즉시 프레임 전체를 브로드캐스트 채널로 전송한다.
    - **충돌이 발생**하는 경우
        - p의 확률로 즉시 프레임을 재전송하거나,
        - 프레임 전송 시간동안 기다린다.
            - 기다리고 난 후, p의 확률로 프레임을 전송하거나, 1-p의 확률로 또 다른 프레임 시간동안 기다린다.

### Carrier Sense Multiple Access (CSMA)

slotted와 pure ALOHA 모두에서, 한 노드의 전송 결정은 브로드캐스트 채널의 다른 노드와 독립적이었다.

사람 사이의 정중한 대화를 위한 두 가지 **규칙**은 다음과 같다.

- 말하기 전에 듣기
    - 네트워크 용어로 **carrier sensing**이라고 한다.
- 어떤 사람이 동시에 말하기 시작하면, 말하는 것을 멈추기
    - 네트워크 용어로 **collision detection**이라고 한다.
    - 전송하는 노드가 전송하는 동안 채널을 듣는다. 만약 다른 노드가 전송하는 것을 감지하면 전송을 멈추고 임의의 시간동안 기다린다.

위의 사항들은 **carrier sense multiple access (CSMA)**와 **CSMA with collision detection (CSMA/CD)**에 포함되어 있다.

이제 CSMA와 CSMA/CD의 가장 중요하고 본질적인 특성을 알아볼 것이다.

1. 왜 모든 노드가 carrier sensing을 하는 경우, 처음에 충돌이 발생할까?
    - 노드는 다른 노드가 전송하는 것을 감지하면 전송을 자제할 것이다.
    - (그림 6.12 참고)하나의 선형 브로드캐스트 버스에 네 개의 노드가 붙어있는 경우를 가정하자.
    - $t_0$에, B는 채널이 비어있다는 것을 감지하고 전송을 시작한다.
        - 전송된 비트는 양방향으로 브로드캐스트 매체를 따라 전파된다.
    - $t_1$에, D에 전송할 프레임이 존재한다.
        - 같은 시간에 노드 B가 전송중이지만, 아직 D에 해당 비트가 도달하지 않았다.
        - 따라서 D는 채널이 비어있다고 감지한다.
        - D는 프레임 전송을 시작한다.
    - 이를 통해 **종단간 채널 전파 지연(end-to-end channel propagation delay)**에 따라 성능이 결정된다는 것을 알 수 있다.
        - 지연이 길어질수록, carrier-sensing하는 노드가 다른 노드에서 전송했다는 사실을 감지하지 못할 확률이 증가한다.

### Carrier Sense Multiple Access with Collision Dection (CSMA/CD)

위의 CSMA 방식은 충돌 탐지 기능이 없었다.

노드가 충돌 탐지 기능이 있다면, **충돌을 감지**하자마자 **전송을 중단**해야한다.

아래 그림 6.13을 보면 두 노드가 충돌을 감지하고 잠시 뒤에 전송을 중단하는 것을 볼 수 있다.

이를 통해 의미없고, 손상된 프레임이 통째로 전송되는 것을 방지할 수 있다.

![Untitled](/assets/img/cs/network/network06/Untitled%209.png)

- **동작 과정**
    1. 어댑터는 네트워크 계층에서 데이터그램을 가져와 링크 계층 프레임을 준비하고, 프레임 어댑터 버퍼(frame adapter buffer)를 배치한다.
    2. 어댑터가 채널이 비어있다는(채널에서 어댑터로 아무 신호 에너지도 들어오지 않는 경우) 것을 감지하면, 프레임을 전송하기 시작한다.
        1. 어댑터가 채널이 바쁘다는 것을 감지하면, 어떠한 신호 에너지(signal engergy)도 감지되지 않을 때까지 기다린다.
        2. 그 후에 프레이을 전송하기 시작한다.
    3. 전송하는 동안, 어댑터는 브로드캐스트 채널을 사용하는 다른 어댑터에서 오는 신호 에너지의 존재를 관찰한다.
    4. 어댑터가 다른 어댑터의 신호 에너지를 감지하지 않고 프레임을 다 전송하면 어댑터는 해당 프레임과 함께 종료된다.
        1. 감지했다면, 전송을 종료한다.
    5. 어댑터는 임의의 시간동안 기다린 후에 **2단계**로 돌아간다.

## Taking-Turns Protocols

이전에 살펴본 다중 접근 프로토콜의 두 가지 이상적인 특성은 다음과 같다.

1. 하나의 노드만이 active하면, 해당 노드의 처리량은 R bps이다.
2. **M개의 노드가 active**하면, 각각의 active 노드는 처리량이 약 R/M bps이다.

ALOHA와 CSMA는 1번 특성은 만족하지만 **2번**은 만족하지 못했다.

이를 만족하기 위해 만들어진 것이 **taking-turns protocol**이다.

여러 변형판이 있는데, 여기서는 **polling protocol**과 **token-passing protocol**을 알아볼 것이다.

### Polling protocol

1. 노드 중 하나를 마스터 노드로 지정한다.
2. 마스터 노드는 각 노드를 round-robin 방식으로 선정한다.
3. 마스토 노드는 노드1에 메시지를 보내서 **최대 몇개의 프레임까지 전송할 수 있다는 것을 알려준다.**
4. 노드1이 프레임을 보내고 나서, 마스터 노드는 노드2에게 비슷한 방식으로 알려준다.
5. 이 과정은 마스터 노드가 각 노드를 순환하면서 계속된다.

- **장점**
    - 충돌을 제거한다.
    - 임의 접근 프로토콜을 방해하는 슬롯을 비운다.

- **단점**
    - 폴링 지연(polling delay, 노드에게 전송할 수 있다는 것을 알려주는데 드는 시간)이 생긴다.
    - 마스터 노드가 고장날 경우, 전체 채널이 작동하지 않게된다.

### **Token-passing protocol**

1. 고정된 순서로 노드들끼리 **토큰(특수한 목적을 갖는 작은 프레임)을 교환**한다.
2. 토큰을 받은 노드는 **전송할 프레임**이 있는 경우에만 **토큰을 쥐고**있는다.
    - 만약 전송할 프레임이 없으면 즉시 토큰을 다음 노드에게 포워딩한다.
3. 노드는 **최대 프레임의 개수**만큼 전송을 한 후에 토큰을 다음 프레임에 포워딩한다.

- **장점**
    - 분산된 방식이다.
    - 매우 효율적이다.

- **단점**
    - 한 노드의 고장으로 인해 전체 채널이 고장나게 된다.
    - 노드가 실수로 토큰을 내려놓지 않는 경우, 토큰을 다시 돌려놓기 위한 복구 절차가 수행되어야 한다.

## Link-Layer Addressing and ARP

호스트와 라우터는 네트워크 계층 주소 뿐만 아니라 링크 계층 주소도 갖고있다.

### MAC Address

실제로는 **호스트와 라우터**가 아닌, 이들의 **어댑터(네트워크 인터페이스)**가 **링크 계층 주소**를 갖는다. 따라서 한 호스트와 라우터에 네트워크 인터페이스가 여러 개 있는 경우, 링크 계층 주소도 여러 개 존재한다.

링크 계층 스위치에는 호스트와 라우터와 연결된 **인터페이스**에 대한 **링크 계층 주소가 없다.** 해당 스위치의 역할은 둘(호스트, 라우터) 사이에 데이터그램을 전달하는 것이기 때문이다.

링크 계층 주소는 **LAN 주소(LAN address)**, **물리적 주소(physical address)**, **MAC 주소(MAC address)** 등으로도 불린다.

MAC 주소는 대부분 6바이트 길이를 갖기 때문에 2^48개가 존재할 수 있다. (그림 6.16)

어떤 어댑터도 동일한 MAC 주소를 갖지 않는다.

어떤 회사가 어댑터를 생산하려면 2^24개의 주소가 포함된 주소 공간 뭉치를 IEEE로부터 구매한다.

그 후 MAC 주소의 처음 24비트를 고정해서 주소 뭉치를 할당하고, 회사가 뒤의 24비트를 조합하도록 한다.

![Untitled](/assets/img/cs/network/network06/Untitled%2011.png)

어댑터의 MAC 주소는 **평평한 구조**이고 어댑터가 어디로 가던간에 **변하지 않는다.** 예를 들어 이더넷 인터페이스에 연결된 노트북은 어디로 노트북이 이동하더라도 MAC 주소가 동일하다.

어댑터가 특정 목적지 어댑터에 프레임을 전송하려면, 목적지 어댑터의 MAC 주소를 프레임에 넣고 LAN에 전송한다.

### Address Resolution Protocol (ARP)

네트워크 계층 주소(ex. **IP 주소**)와 링크 계층 주소(ex. **MAC 주소**)가 모두 있기 때문에 이들 간에 변환이 필요하다. 이러한 일을 하는게 **Address Resolution Protocol (ARP)**이다.

아래 그림과 같은 네트워크가 있다고 하자. 여기서 스위치는 모든 프레임을 브로드캐스트 한다고 가정해보자.

C가 A에게 **IP 데이터그램을 전송**하는 과정은 다음과 같다.

1. 출발지 C는 어댑터에게 IP 데이터그램뿐만 아니라 A의 **MAC 주소**도 준다.
2. 송신 어댑터는 목적지의 MAC 주소가 포함된 링크 계층 **프레임**을 만든다.
3. 해당 프레임을 LAN에 보낸다.

![Untitled](/assets/img/cs/network/network06/Untitled%2012.png)

이 때, 송신 호스트는 어떻게 **A의 MAC 주소**를 알까? 여기에 사용되는게 **ARP**이다.

송신 호스트가 같은 LAN 상의 **IP 주소를 입력**하면, **대응되는 MAC 주소를 반환**해준다.

### 서브넷 밖으로의 데이터그램 전송 (Sending a Datagram off the Subnet)

1. 송신 호스트는 어댑터에게 데이터그램을 전달한다.
    1. 보낼 때는 목적지의 MAC 주소도 적어야 한다.
    2. 이 때의 MAC 주소는 라우터 인터페이스 111.111.111.110의 어댑터의 주소이다. (= 첫 번째 hop 라우터의 MAC 주소)
2. 서브넷 1(왼쪽)의 라우터 어댑터가 링크 계층 프레임을 받는다.
    1. 라우터의 네트워크 계층으로 프레임을 전달한다.
3. 라우터는 포워딩 테이블을 사용해서 데이터그램이 222.222.222.220으로 포워딩되어야 한다는 것을 안다.
    1. 인터페이스는 데이터그램을 자신의 어댑터에 보내고 새로운 프레임으로 캡슐화하여 서브넷 2로 보낸다.
    2. 이 때의 MAC 주소는 실제 최종 목적지의 MAC 주소와 동일하다.

## Ethernet

현재 LAN 시장에서 가장 지배적인 위치를 차지한다.

- **이더넷의 성공 요인**
    1. 이더넷은 **최초**로 널리 배치된 고속 LAN이다.
        1. 초기에 배치됐기 때문에 네트워크 관리자들에게 친숙했다.
        2. 이들은 다른 LAN 기술로 옮겨가기 꺼려했다.
    2. 다른 기술들(토큰 링, FDDI, ATM)은 이더넷보다 더 복잡하고 비쌌다.
    3. 다른 기술들이 내세우는 강점인 높은 전송률을 이더넷은 버전을 올려가며 따라잡았다.
    4. 이더넷의 인기 덕에 이더넷 하드웨어는 필수품이 되었고 다른 것들에 비해 훨씬 저렴해졌다.
- 변천사
    - **초기 (1970년대 중반)**
        - **bus topology**
        - Broadcast LAN : 모든 전송된 프레임은 버스에 연결된 모든 어댑터에 의해 처리된다.
    - **1990년대 후반**
        - **hub-based star topology**
        - 호스트들은 twisted-pair copper wire에 의해 **허브에 직접 연결**된다.
    - **2000년대 초반**
        - **star topology**
        - 가운데에는 허브 대신 **스위치**를 사용했다.

### 이더넷 프레임 구조 (Ethernet Frame Structure)

다음은 이더넷 프레임의 필드 여섯 개이다.

![Untitled](/assets/img/cs/network/network06/Untitled%2015.png)

- 데이터 필드 (Data field) (46~1500 바이트)
    - IP 데이터그램을 담는 필드
    - 이더넷의 MTU(maximum transmission unit)는 1500 바이트이다.
        - IP 데이터그램이 1500 바이트를 초과하면 여러 조각으로 나눠야한다.
    - 최소 크기인 46 바이트보다 작으면 채워넣어서 크게 만들어야 한다.
        - 이후에 네트워크 계층이 길이 필드를 사용해서 채워넣은 것(stuffing)을 제거한다.
- 목적지 주소 (Destination address) (6 바이트)
    - 목적지 어댑터의 MAC 주소를 담는 필드
- 출발지 주소 (Source address) (6 바이트)
    - 프레임을 LAN에 보내는 어댑터의 MAC 주소를 담는 필드
- 타입 필드 (Type field) (2 바이트)
    - 이 필드는 이더넷이 네트워크 계층 프로토콜을 다중화(multiplex)하는 것을 허용한다.
- 순환 중복 검사 (Cyclic redundancy check) (4 바이트)
    - 수신 어댑터가 프레임의 비트 오류를 감지하도록 해준다.
- 프리앰블 (Preamble) (8 바이트)
    - 서두, 전문이라는 의미
    - 이더넷 필드의 시작 부분의 8 바이트
    - 처음 7바이트는 `10101010` , 마지막 바이트는 `10101011`
    - 수신 어댑터를 깨우고 송신자와 수신자의 시간을 동기화해준다.

## Link-Layer Switches

스위치의 역할은 링크 계층 프레임을 받아서 나가는 링크에다가 포워딩해주는 것이다.

스위치는 서브넷 내의 호스트와 라우터에게 투명인간(transparent) 취급된다. 호스트/라우터는 프레임에 대해 스위치가 아니라 다른 호스트/라우터로 주소를 지정한다. 즉, 스위치가 해당 프레임을 받아서 포워딩해준다는 것을 모른다.

### 포워딩과 필터링 (Forwarding and Filtering)

**포워딩** : 프레임이 어디로 가야할 지를 결정하는 스위치의 기능

**필터링** : 프레임이 포워딩 될지 버려질지를 결정하는 스위치의 기능

이 두 기능은 **스위치 테이블**에 의해 이루어진다.

- **스위치 테이블(switch table)**

여기에는 LAN 상의 호스트와 라우터 몇 개의 엔트리가 있다.

**엔트리**에는 다음 내용이 있다. (그림 6.22)

1. MAC 주소
2. 해당 MAC 주소로 연결되는 스위치 인터페이스
3. 엔트리가 테이블에 놓여진 시간

![Untitled](/assets/img/cs/network/network06/Untitled%2016.png)

### 자가 학습 (Self-Learning)

스위치는 네트워크 관리자나 구성 프로토콜의 개입 없이 테이블이 자동으로, 동적으로 구축되는 좋은 특성을 갖는다. 이러한 특성을 **자가 학습**이라고 한다.

- **동작 과정**
    1. 스위치 테이블은 초기에 비어있다.
    2. 인터페이스에서 받는 각각의 프레임에 대해 스위치는 다음 내용을 테이블에 저장한다. LAN 상의 모든 호스트가 프레임을 보냈다면 테이블이 채워질 것이다.
        1. 프레임의 출발찌 주소 필드의 MAC 주소
        2. 프레임이 들어온 인터페이스
        3. 현재 시각
    3. 일정 시간(aging time)동안 특정 주소에서 프레임이 들어오지 않는다면 테이블에서 주소를 삭제한다.

스위치는 네트워크 관리자나 사용자로부터 어떠한 간섭도 받지 않는 **plug-and-play 기기**이다.

### 링크 계층 스위칭의 속성 (Properties of Link-Layer Switching)

- 스위치의 장점
    - 충돌의 제거 (Elimination of collisions)
        - 스위치는 프레임들을 버퍼에 저장해두고 한 시점에 하나의 프레임만을 전송한다.
    - 서로 다른 링크 (Heterogeneous links)
        - 스위치는 링크들을 서로 떨어뜨린다.
        - 이로 인해 기존 장비와 새 장비를 혼합하는데에 이상적이다.
    - 관리 (Management)
        - 네트워크의 관리를 용이하게 해준다.
            - 만약 한 어댑터가 오작동하여 이더넷 프레임을 계속해서 전송한다면 스위치가 해당 어댑터를 끊을 것이다.
        - 대역폭 사용량, 충돌률, 트래픽 유형에 대한 통계를 네트워크 관리자가 사용할 수 있는 정보로 만들어준다.
            - 이 정보는 문제를 디버깅하고 향후 LAN이 어떻게 발전할지에 대한 계획을 세우는데 사용될 수있다.

### 스위치 vs 라우터 (Switches Versus Routers)

- **사전 지식 (그림 6.24)**
    - **라우터**
        - store-and-forward 패킷 스위치
        - 패킷을 **네트워크 계층 주소**를 사용해 포워딩한다.
        - 3계층 (layer-3) 패킷 스위치
    - **스위치**
        - store-and-forward 패킷 스위치
        - 패킷을 **MAC 주소**를 사용해 포워딩한다.
        - 2계층 (layer-2) 패킷 스위치

![Untitled](/assets/img/cs/network/network06/Untitled%2017.png)

- **각각의 장단점 (그림 6.25)**
    - **스위치**
        - 장점
            - plug-and-play 방식이기 때문에 네트워크 관리자의 부담을 덜어준다.
            - 필터링, 포워딩 비율이 상대적으로 높다.
                - 라우터에 비해 한 계층 덜 처리해도 된다.
        - 단점
            - 브로드캐스트 프레임의 순환을 막으려면, 토폴로지가 spanning tree로 한정된다.
            - 큰 switched 네트워크는 ARP 테이블도 커야하기 때문에 상당한 ARP 트래픽과 처리를 요한다.
    - **라우터**
        - 장점
            - 패킷이 spanning tree로 한정되지 않고, 출발지와 목적지 사이에 있는 최고의 경로를 이용할 수 있다.
            - 2 계층의 브로드캐스트 스톰(broadcast storm)을 막아주는 방화벽이 제공된다.
                - 브로드캐스트 스톰 : 한 패킷이 여러 수신자에 의해 끊임 없이 브로드캐스팅되어 발생하는 폭주 상태
        - 단점
            - plug-and-play 방식이 아니다.
                - 연결되는 호스트들은 각각의 IP 주소를 가져야한다.
            - 스위치에 비해 패킷별 처리시간이 더 길다. (3계층이기 때문)

## Virtual Local Area Networks (VLANs)

앞에서는 기관의 LAN은 계층적으로 구성된다고 했다. 이러한 계층은 각 부서가 자신의 switch된 LAN을 갖고, 이들이 스위치로 다른 그룹과 연결된다.

이러한 구성은 실제로는 몇가지 문제가 있다. (그림 6.15)

- **트래픽 고립의 부족 (Lack of traffic isolation)**
    - 계층을 통해 대부분의 그룹 트래픽은 각 스위치로 나뉘어진다.
    - 하지만 **브로드캐스트 트래픽**은 전체 기관 네트워크를 통과해야한다.
    - 트래픽의 고립은 그림 중앙에 위치한 스위치를 라우터로 대체하여 만들 수 있다.
- **비효율적인 스위치의 사용 (Inefficient use of switches)**
    - 그룹의 수가 많아지고 각 그룹의 사람이 적은 경우, 10개의 1단계 스위치가 필요하다.
    - 이보다는 한 개의 96-포트 스위치가 더 효율적이다.
        
        ![Untitled](/assets/img/cs/network/network06/Untitled%2019.png)
        
        - 대신 이러한 스위치는 트래픽 고립을 지원하지 않는다.
- **사용자 관리 (Managing users)**
    - 직원이 그룹 간에 이동하는 경우, 연결되어 있던 물리적 케이블을 다른 스위치에 연결해야한다.
    - 두 그룹에 속한 경우 더 복잡해진다.

이러한 문제들은 **virtual local area networks (VLANs)**을 지원하는 스위치를 통해 해결될 수 있다.

VLAN을 지원하는 스위치는 **하나의 물리적인 LAN**에서 **여러개의 가상 LAN**을 정의하도록 해준다.

(그림 6.25)아래 예시를 보면, 하나의 스위치에 16개의 포트가 있는 것을 볼 수 있다. 포트 2~8은 EE VLAN에 속해있고, 포트 9~15는 CS VLAN에 속해있다.

이를 통해 위의 **문제점들을 해결**할 수 있다.

- **트래픽 고갈 문제**
    - EE와 CS VLAN은 **서로 독립**되어 있다.
- **비효율적인 스위치 사용 문제**
    - (그림 6.15)기존에 두 개였던 스위치가 **한 개**로 줄었다.
- **사용자 관리 문제**
    - 포트 8번 사용자가 CS 부서로 이동한다면 **VLAN 소프트웨어를 재설정**하기만 하면 된다.

# Data Center Networking

구글, 마이크로소프트 등의 인터넷 회사들은 **데이터 센터**를 짓는다. 이러한 데이터 센터는 자신의 호스트끼리 연결하고 데이터 센터와 인터넷을 연결하는 **데이터 센터 네트워크**를 갖고있다.

- 데이터 센터에서의 **호스트** (hosts in data center)
    - 데이터 센터에서 호스트들은 다양한 일을 한다.
        - 콘텐츠(웹 페이지, 비디오 등)를 제공한다.
        - 이메일과 문서를 저장한다.
        - 광범위하게 분산된 계산들을 모아서 처리한다. (검색 엔진에서의 분산된 인덱스 계산)
    - 이들은 블레이드(blade)라고 불린다.
    - CPU, 메모리, 디스크 저장소 등을 포함한 범용 호스트(commodity host)이다.
    - 랙(rack, 선반) 하나당 20~40개 정도의 호스트들이 쌓여있다. (그림 6.30 참고)
    - 각 랙의 상단에는 **Top of Rack (TOR) 스위치**가 있다.
        - 랙 안의 호스트들을 연결해주고 다른 스위치와 연결한다.

![Untitled](/assets/img/cs/network/network06/Untitled%2022.png)

- **트래픽의 종류**
    - 외부 클라이언트와 내부 호스트 사이의 트래픽 플로우 (traffic flowing between external clients and internal hosts)
        - 이를 위해 **경계 라우터(border router)**가 필요하다.
            - 데이터 센터 네트워크를 공공 인터넷(public internet)에 연결한다.
    - 내부 호스트 사이의 트래픽 플로우 (traffic flowing between internal hosts)

데이터 센터 네트워크는 **랙끼리** 연결하고 **랙을 경계 라우터와 연결**한 형태이다. (그림 6.30 참고)

### 로드 밸런싱 (Load Balancing, 부하 분산)

외부 클라이언트의 요청을 받기 위해서 각 애플리케이션은 public하게 볼 수 있는 IP 주소를 가져야 한다.

- **동작 과정**
    - 데이터 센터 내에서, 외부의 요청은 우선 **로드 밸런서(load balancer)**에게 간다.
        - 이들은 호스트에게 온 요청을 분산해준다.
    - 로드 밸런서는 **4계층 스위치(Layer-4 Switch)**라고 여겨진다.
        - 이들은 목적지 IP 주소 뿐만 아니라 목적지 포트 번호(4계층)를 기반으로 결정을 내리기 때문이다.
    - 로드 밸런서가 특정 애플리케이션에 대한 요청을 수신하면 관련된 호스트 중 하나로 요청을 전달한다.
    - 호스트는 요청을 다 처리하고 나면, 자신의 응답을 로드 밸런서에게 보낸다.
    - 이 응답은 외부 클라이언트에게로 간다.
- 이들은 부하를 분산할 뿐 아니라 NAT와 유사한 함수를 제공한다.
    - 외부의 공공 IP 주소를 내부 IP 주소로 변환한다.
    - 클라이언트에게로 되돌아가는 패킷의 IP 주소를 다시 변환한다.

### 계층 구조 (Hierarchical Architecture)

데이터 센터가 작은 경우에는 네트워크는 경계 라우터, 로드 밸런서, 열 개 정도의 랙이 하나의 이더넷 스위치로 연결된 형태면 충분하다.

하지만 데이터 센터가 큰 경우에는 라우터와 스위치를 계층적으로 구성한다. (그림 6.30)

- 각 액세스 라우터의 아래에는 **세 단계의 스위치**가 있다.
- 각 액세스 라우터는 1단계 스위치와 연결되어 있다.
- 각 1단계 스위치는 여러 개의 2단계 스위치와 로드 밸런서와 연결되어 있다.
- 각 2단계 스위치는 3단계 스위치인 TOR 스위치를 통해 여러 랙과 연결한다.

DHCP 서버가 라우터 안에서 실행중이라고 가정하자.

우선 밥은 노트북으로 **DHCP 프로토콜**을 실행해서 DHCP 서버로부터 **IP 주소**와 기타 정보를 받아와야 한다.

1. 밥의 노트북의 OS는 DHCP 요청 메시지를 생성해서 UDP 세그먼트에 넣는다.
    1. 이 때, 목적지 포트는 67번(DHCP 서버), 출발지 포트는 68번(DHCP 클라이언트)이다.
    2. 이 UDP 세그먼트는 이후에 IP 데이터그램에 놓여진다.
        1. 목적지 IP 주소 : `255.255.255.255` (브로드캐스트)
        2. 출발지 IP 주소 : `0.0.0.0` (밥의 노트북은 아직 IP 주소가 없기 때문)
2. DHCP 요청 메시지를 포함한 **IP 데이터그램**은 **이더넷 프레임**에 놓여진다.
    1. 목적지 MAC 주소 : `FF:FF:FF:FF:FF:FF` (스위치에 연결된 모든 기기에 브로드캐스팅)
    2. 출발지 MAC 주소 : `00:16:D3:23:68:8A` (그림에서 밥의 MAC 주소)
3. DHCP 요청을 담은 브로드캐스트 이더넷 프레임은 밥의 노트북에서 이더넷 스위치로 보내는 첫 번째 프레임이다.
    1. 스위치는 프레임을 받아서 모든 나가는 포트로 브로드캐스팅한다. (연결된 라우터도 포함)
4. 라우터는 해당 프레임을 MAC 주소가 `00:22:6B:45:1F:1B`인 인터페이스로 받아서 IP 데이터그램을 추출한다.
    1. 해당 데이터그램의 페이로드(UDP 세그먼트)는 위쪽의 UDP로 역다중화(demultiplexed)된다.
    2. 해당 세그먼트에서 DHCP 요청 메시지가 추출된다.
    3. 이렇게 DHCP 서버는 DHCP 요청 메시지를 갖게 된다.
5. 라우터 안에서 실행되는 DHCP 서버가 CIDR 블록 `68.85.2.0/24`에서 IP 주소를 할당한다고 가정하자. 이때, 학교에서 사용되는 모든 IP 주소는 컴캐스트(Comcast)의 주소 블록에 속한다. DHCP 서버가 밥의 노트북의 주소를 `68.85.2.0.101`로 할당했다고 하자.
    1. DHCP 서버는 IP 주소가 포함된 DHCP ACK 메시지를 생성한다.
    2. 이 메시지에는 DNS 서버의 IP 주소도 포함된다.
    3. DHCP 메시지는 UDP 세그먼트 안에, 이건 IP 데이터그램 안에 들어간다.
    4. 이더넷 프레임에는 출발지 MAC 주소(라우터 인터페이스에서 홈 네트워크로)와 목적지 MAC 주소(밥의 노트북)가 있다.
6. DHCP ACK를 포함하는 **이더넷 프레임**은 라우터에서 스위치로 보내진다.
    1. 스위치는 받은 프레임을 밥의 노트북으로 가는 유일한 포트인 `00:16:D3:23:68:8A` 로 포워딩해야한다는 것을 안다.
    2. 스위치는 자가학습을 하고 이더넷 프레임을 이전에 밥의 노트북으로부터 받았기 때문이다.
7. 밥의 노트북은 DHCP ACK를 포함하는 **이더넷 프레임**을 받는다.
    1. 그 후, 해당 프레임에서 **IP 데이터그램**을 추출하고,
    2. 해당 데이터그램에서 **UDP 세그먼트**를 추출하고,
    3. 해당 세그먼트에서 **DHCP ACK 메시지**를 추출한다.
    4. 밥의 DHCP 클라이언트는 받은 메시지에 자신의 IP 주소와 DHCP 서버의 IP 주소를 기록한다.
    5. 또한 IP 포워딩 테이블에 기본 게이트웨이 주소를 기입한다.
    6. 여기까지 하면, 밥의 노트북은 자신의 네트워크 구성요소를 초기화하고 웹 페이지를 받아올 준비를 마친 상태가 된다.

	> **아직도 시작 중 : DNS and ARP**

	밥이 웹 브라우저에 [www.google.com](http://www.google.com)을 입력하면, 구글 홈페이지가 웹 브라우저에 뜨기까지 긴 과정이 시작된다. 밥의 웹 브라우저는 **TCP 소켓**을 생성하고 이를 통해 **HTTP 요청**을 [www.google.com](https://www.google.com) 에 보낸다. 소켓을 생성하기 위해서 밥의 노트북은 구글의 IP 주소를 알아야 한다. 여기서 사용되는 것이 **DNS 프로토콜**이다.

8. 밥의 노트북의 운영체제는 **DNS 쿼리 메시지**를 생성한다.
    1. 쿼리(Query) : 데이터베이스 등에서 원하는 정보를 검색하기 위해 요청하는 것
    2. 해당 DNS 메시지의 질문 섹션에 [www.google.com](http://www.google.com) 문자열을 넣는다.
    3. DNS 메시지는 목적지 포트가 53으로 설정되어 **UDP 세그먼트** 안에 놓여진다.
    4. UDP 세그먼트는 목적지 IP 주소가 `68.87.71.226` , 출발지 IP 주소가 `68.85.2.101`로 되어 **IP 데이터그램** 안에 놓여진다.
9. 밥의 노트북은 DNS 쿼리 메시지를 담은 데이터그램을 **이더넷 프레임** 안에 넣는다.
    1. 해당 프레임은 밥의 학교 네트워크의 게이트웨이 라우터로 보내진다.
    2. 밥의 노트북은 게이트웨이 **라우터의 MAC 주소**를 알기 위해 ARP 프로토콜을 사용한다. (현재는 라우터의 IP 주소만 아는 상황)
10. 밥의 노트북은 대상 IP 주소가 `68.85.2.1`(기본 게이트웨이)로 설정된 ARP 쿼리 메시지를 생성한다.
    1. 해당 메시지를 **이더넷 프레임** 안에 넣고 목적지 주소를 **브로드캐스트용**(`FF:FF:FF:FF:FF:FF`)으로 설정한 뒤 **스위치**에 보낸다.
11. **게이트웨이 라우터**는 ARP 쿼리 메시지가 담긴 프레임을 학교 네트워크 쪽 인터페이스에서 받는다.
    1. 이 메시지의 **대상 IP 주소**인 `68.85.2.1`이 해당 **인터페이스의 IP 주소**와 일치하는지를 확인한다.
    2. 라우터는 자신의 IP 주소 `68.85.2.1`에 대응되는 **MAC 주소**가 `00:22:6B:45:1F:1B`라는 **ARP 응답**을 **이더넷 프레임**에 담아서 밥의 노트북으로 보낸다.
12. 밥의 노트북은 해당 프레임에서 ARP 응답 메시지를 추출해 게이트웨이 라우터의 **MAC 주소**를 알아낸다.
13. 여기까지 하면 밥의 노트북은 DNS 쿼리가 담긴 이더넷 프레임을 게이트웨이 **라우터의 MAC 주소**로 주소 지정할 수 있게 된다.
    1. 하지만 해당 **프레임**의 목적지 주소가 **게이트웨이 라우터**의 MAC 주소(`00:22:6B:45:1F:1B`)인 반면, 이 내부의 **IP 데이터그램**의 목적지 IP 주소는 **DNS 서버**의 것(`68.87.71.226`)이다.
    2. 밥의 노트북에 의해 스위치로 보내진 프레임은 이후 게이트웨이 라우터로 보내진다.

	> **여전히 시작 중: Intra-Domain Routing to the DNS Server**

14. 게이트웨이 라우터는 프레임을 받고 DNS 쿼리가 담긴 IP 데이터그램을 추출한다.
    1. **게이트웨이 라우터**는 데이터그램의 목적지 주소(`68.87.71.226`)를 포워딩 테이블에서 찾아본 후에 그림 6.32의 컴캐스트 네트워크의 제일 왼쪽 라우터로 보내야 한다는 것을 알게된다.
    2. IP 데이터그램은 **링크 계층 프레임** 안에 놓여진 후에 링크를 통해 전송된다.
15. **컴캐스트 네트워크의 제일 왼쪽 라우터**는 이 프레임을 받고 **IP 데이터그램**을 추출한 뒤에, 목적지 주소(`68.87.71.226`)를 포워딩 테이블에서 확인해서 어느 인터페이스로 포워딩해야 할 지를 결졍한다.
    1. 이 포워딩 테이블은 컴캐스트의 intra-domain protocol(RIP, OSPF, IS-IS 등)과 인터넷의 inter-domain protocol, BGP에 의해 채워진다.
16. DNS 쿼리를 담은 IP 데이터그램이 DNS 서버에 도달한다.
    1. DNS 서버는 DNS 쿼리 메시지를 추출해서 자신의 **데이터베이스**에서 [www.google.com](http://www.google.com) 을 찾아본 후에 이에  대한 **IP 주소**(`64.233.169.105`)가 담긴 **DNS 자원 레코드(DNS resource record)**를 발견한다.
    2. DNS 서버는 위와 같이 호스트명-IP 주소의 매핑이 포함된 **DNS 응답 메시지**를 만들어 UDP 세그먼트에 넣고 다시 **IP 데이터그램**(밥의 노트북으로 주소 지정됨)에 넣는다.
    3. 이 데이터그램은 컴캐스트네트워크를 따라 학교의 라우터로 다시 포워딩되고, 이더넷 스위치를 거쳐 밥의 노트북으로 돌아갈 것이다.
17. 밥의 노트북은  DNS 메시지에서 [www.google.com](http://www.google.com) 의 **IP 주소를 추출**해낸다.

	> **웹 클라이언트-서버 간의 상호작용 : TCP and HTTP**

18. 이제 밥의 노트북은 [www.google.com](http://www.google.com) 의 IP 주소를 알고있으므로, **TCP 소켓**을 만들어서 HTTP GET 메시지를 www.google.com에게 보낸다.
    1. TCP 소켓을 만드는 과정은 밥의 노트북과 구글간의 **three-way handshake**를 통해 이루어진다.
    2. 밥의 노트북이 먼저 **TCP SYN 세그먼트**(목적지 포트 : 80(**HTTP**))를 생성하고, 해당 세그먼트를 **IP 데이터그램**(목적지 IP 주소 : `64.233.169.105`(**구글**))에 넣고, 이를 다시 **프레임**(목적지 MAC 주소 : `00:22:6B:45:1F:1B`(**게이트웨이 라우터**))에 넣어서 **스위치**로 보낸다.
19. 학교 네트워크, 컴캐스트 네트워크, 구글 네트워크 내부의 **라우터**들은 이 데이터그램을 포워딩 테이블을 활용해서 [www.google.com](http://www.google.com) 으로 **포워딩**한다.
20. 데이터그램을 포함한 **TCP SYN**이 [www.google.com](http://www.google.com) 에 도착한다.
    1. TCP SYN 메시지가 데이터그램에서 추출되어 80번 포트에 연결된 **welcome 소켓**으로 역다중화된다.
    2. 구글 HTTP 서버와 밥의 노트북 간의 TCP 연결을 위해 **연결 소켓**이 생성된다.
    3. **TCP SYNACK** 세그먼트가 생성되고 데이터그램 안에 놓여 밥의 노트북으로 주소 지정된다.
    4. 이 데이터그램은 [www.google.com](http://www.google.com) 에서 첫 홉 라우터로의 링크에 맞는 링크 계층 프레임 안에 놓여진다.
21. **TCP SYNACK** 세그먼트를 포함한 데이터그램은 구글, 컴캐스트, 학교 네트워크를 거쳐 포워딩되어 밥의 노트북의 이더넷 카드에 도착한다.
    1. 해당 데이터그램은 TCP 소켓으로 역다중화되고 **연결 상태**가 된다.
22. 이제 밥의 노트북은 [www.google.com](http://www.google.com) 으로 바이트를 보낼 준비가 되었기에, 밥의 브라우저는 fetch될 URL이 담긴 **HTTP GET 메시지**를 생성한다.
    1. HTTP GET 메시지가 소켓에 기록되고, TCP 세그먼트의 페이로드가 된다.
    2. 해당 세그먼트는 데이터그램에 놓여진 채 [www.google.com](http://www.google.com) 으로 보내진다.
23. [www.google.com](http://www.google.com) 의 HTTP 서버는 HTTP GET 메시지를  읽고 **HTTP 응답 메시지**를 생성한다.
    1. 요청받은 웹 페이지의 내용을 메시지의 body에 위치시킨다.
    2. 해당 메시지를 TCP 소켓으로 보낸다.
24. HTTP 응답 메시지를 담은 데이터그램이 구글, 컴캐스트, 학교 네트워크를 거쳐 밥의 노트북에 도착한다.
    1. 밥의 웹 브라우저는 HTTP 응답 메시지를 읽고, body에서 웹 페이지의 html을 추출한다.
    2. 그 후 웹 페이지를 화면에 표시한다.
