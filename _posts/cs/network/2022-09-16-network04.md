> Computer Networking: A Top-Down Approach (7th Edition)를 정리한 글입니다.

네트워크 계층은 프로토콜 스택에서 제일 복잡한 계층이라고 할 수 있다.

이 계층은 크게 **데이터 평면(data plane)**과 **제어 평면(control plane)**으로 나뉜다. 이 장에서는 **데이터 평면**에 대해 다룬다.

# Overview of Network Layer

아래는 두 호스트 H1, H2가 단순한 네트워크 상에 있는 모습을 나타낸 그림이다. H1이 H2에게 정보를 보내는 상황을 가정해보자.

- **H1**
    1. H1의 **네트워크 계층**이 **전송 계층**으로부터 **세그먼트**를 받는다.
    2. 각 세그먼트를 캡슐화해서 **데이터그램**에 넣는다.
    3. 데이터그램을 인근 **라우터(R1)**로 보낸다.
- **H2**
    1. H2의 네트워크 계층은 인근 **라우터(R2)**로부터 **데이터그램**을 받는다.
    2. 데이터그램을 전송 계층의 **세그먼트**로 추출한다.
    3. 전송 계층으로 세그먼트를 보낸다.

위 과정에서 각 **라우터**의 **데이터 평면**의 주 역할은 입력 링크(input link)의 **데이터그램을 출력 링크(output link)로 포워딩**해주는 것이다. **제어 평면**의 주 역할은 데이터그램이 이와 같이 end-to-end로 전송되도록 **로컬과 라우터별 포워딩 작업을 조정**하는 것이다.

## Forwarding과 Routing: 데이터와 제어 평면

네트워크 계층의 주 역할은 매우 단순하다—패킷을 송신 호스트로부터 수신 호스트로 옮기는 것이다. 이를 위해 네트워크 계층의 **두 가지 중요한 기능**이 정의된다.

- **포워딩(Forwarding)**
    - 패킷이 라우터의 **입력 링크(input link)**에 들어오면, 라우터는 패킷을 적절한 **출력 링크(output link)**로 옮겨야 한다.
    - 위 그림에서 H1의 패킷이 R1으로 도달하면, H2로의 경로 상의 다음 라우터로 포워딩 되어야 한다.
    - 라우터-로컬 간의 작업이다.
    - 굉장히 짧은 시간동안 일어난다. (주로 몇 나노초간)
- **라우팅(Routing)**
    - 네트워크 계층은 패킷이 송신자에게서 수신자에게로 날아갈 **경로(route)를 결정**해야 한다.
    - 이러한 경로를 계산하는 알고리즘을 **라우팅 알고리즘**이라고 한다.
    - 네트워크 계층의 **제어 평면**에 구현되어 있다.
    - 네트워크 전반적인 작업이다.
    - 훨씬 긴 시간동안 일어난다.(주로 몇 초간)

네트워크 라우터의 핵심 요소는 **포워딩 테이블(forwarding table)**이다.

- 라우터는 들어온 패킷의 헤더의 한 개 이상의 필드를 검사해서 패킷을 포워딩한다.
- 그 후, 해당 패킷의 헤더 값을 사용해 패킷을 포워딩 테이블에 인덱싱한다.
    - 위 값에 대해 **포워딩 테이블에 저장되어있는 값**은 패킷이 해당 라우터에서 **포워딩 될 링크 인터페이스**를 나타낸다.
- 예시 (그림 4.2 참고)
    - 헤더 필드의 값이 0110인 패킷이 라우터에 들어온다.
    - 라우터는 패킷을 포워딩 테이블에 인덱싱한다.
    - 해당 패킷의 **출력 링크 인터페이스**를 2번으로 정한다.
    - 라우터는 패킷을 2번 인터페이스로 포워딩한다.

### 제어 평면: 전통적인 접근법

![Untitled](/assets/img/cs/network/network04/Untitled%201.png)

- (그림 4.2 참고) **라우팅 알고리즘**은 라우터의 **포워딩 테이블**의 **내용들을 결정**한다.
- 한 라우터의 라우팅 알고리즘 기능은 **다른 라우터의 라우팅 알고리즘 기능과 소통**하며 포워딩 테이블의 값을 계산해낸다.
    - 이러한 소통은 **라우팅 프로토콜**에 따라 라우팅 정보가 포함된 **메시지를 교환**함으로써 진행된다.

### 제어 평면: SDN 접근법

위의 그림 4.2에서 보여지는 라우팅을 구현하는 접근법—각 라우터가 라우팅 요소를 갖고, 다른 라우터의 요소와 소통하는 방식—은 최근까지 사용되었다. 하지만 사람이 **수동**으로 **포워딩 테이블의 값을 설정**할 수 있다는 점 때문에 **제어 평면**이 데이터 평면의 포워딩 테이블의 내용을 결정할 다른 방법을 찾게 되었다.

- 라우터로부터 물리적으로 분리된 **원격 제어 장치(Remote Controller)**가 **계산**을 하고 각각의 라우터에 **포워딩 테이블을 분배**해준다.(아래 그림 참고)
- 전통적인 접근법(그림 4.2)과 비교했을 때, 데이터 평면의 구성 요소는 동일하지만, 제어 평면의 라우팅 기능은 라우터와 분리되어있다.
- 이러한 접근법은 **소프트웨어 정의 네트워킹(software-defined networking, SDN)**의 핵심이다.
    - **“소프트웨어 정의”**인 이유는 포워딩 테이블을 계산하고 라우터와 상호작용하는 것이 소프트웨어에 구현되어있기 때문이다.

## 네트워크 서비스 모델 (Network Service Model)

네트워크 계층의 데이터 평면을 자세히 살펴보기 전에, 네트워크 계층이 제공할 수 잇는 다양한 유형의 서비스를 먼저 알아보자.

- 송신 호스트의 전송 계층이 패킷을 네트워크에 보냈을 때, 이들은 **패킷이 목적지에 보내지는** 것에 대해 **네트워크 계층**에게 의존할까?
- 여러 패킷이 보내질 때, 수신 호스트의 전송 계층에게 보내진 순서대로 도착할까?
- 연속적인 두 패킷을 보낼 때의 시간차가 받을 때도 동일할까?
- 네트워크가 이들의 혼잡에 대해 피드백을 해줄까?

이러한 질문에 대한 답은 **네트워크 서비스 모델**에 의해 결정된다. 이들이 제공할 수 있는 서비스는 다음과 같다.

- **Guaranteed delivery** : 출발지(source) 호스트로부터 전송된 패킷이 목적지(destination) 호스트에게 **도착**하는 것을 보장한다.
- **Guaranteed delivery with bounded delay** : 패킷의 전달 뿐만 아니라, 특정한 호스트간 **딜레이 안에 전달**될 것을 보장한다.
- **In-order packet delivery** : 패킷이 보내진 **순서대로** 도착할 것을 보장한다.
- **Guaranteed minimal bandwidth** : **최소한의 전송률**을 특정 비트 레이트(1 Mbps 등)로 보장해준다.
- **Security** : 출발지에서 **데이터그램을 전부 암호화**하고 목적지에서 그것들을 **복호화**하여 전송 계층 세그먼트에 기밀성을 제공한다.

![Untitled](/assets/img/cs/network/network04/Untitled%203.png)

라우터의 구성 요소는 다음과 같이 정의될 수 있다.

- **입력 포트(Input ports)**
    - 여러 주요한 기능을 수행한다.
    - 라우터에서 **물리 링크를 종료**하는 물리 계층 기능을 사용한다. (그림 4.4의 입력 포트의 왼쪽 끝과 출력 포트의 오른쪽 끝 참고)
        - 물리 링크(physical link) : 두 기기 간의 전기적인 연결
    - 반대쪽의 들어오는 **링크와 링크 계층간의 상호작용**을 위해 필요한 링크 계층 기능을 사용한다.
    - 가장 중요한 점은 **검색 기능**을 사용한다는 것이다. (이 과정은 입력 포트의 오른쪽 끝에서 발생한다.)
- **스위칭 패브릭(Switching fabic)**
    - 라우터의 **입력 포트**와 **출력 포트**를 **연결**해준다.
    - 라우터 안에 전부 포함되어 있다. 네트워크 라우터 안에 있는 **네트워크**인 셈이다.
- **출력 포트(Output ports)**
    - 링크 계층과 물리 계층의 기능들을 사용해서 스위칭 패브릭으로부터 **받은 패킷**을 **나가는 링크로 전송**한다.
- **라우팅 프로세서(Routing processor)**
    - 제어 평면의 기능들을 수행한다.
    - 전통적인 라우터의 경우
        - 라우팅 프로토콜을 실행한다.
        - 라우팅 테이블 및 연결된 링크 상태 정보를 유지한다.
        - 라우터를 위해 포워딩 테이블을 계산한다.
    - SDN 라우터의 경우
        - 계산된 포워딩 테이블 엔트리를 받기 위해 원격 제어 장치(remote controller)와 소통한다.
        - 이 엔트리를 라우터의 입력 포트에 설치한다.
    - 네트워크 관리 기능을 수행한다.

### 목적지 기반 포워딩 (Destination-Based Forwarding)

들어온 패킷이 나갈 **출력 포트**가 **패킷의 목적지 주소**를 기반으로 하는 “가장 간단한” 경우를 생각해보자.

**32비트 IP 주소**에서 포워딩 테이블을 브루트 포스로 구현하면 각 가능한 목적지 주소에 대해 하나의 엔트리만을 갖게 될 것이다. 40억개의 가능한 주소들이 있기 때문에 이 방법은 불가능하다.

이러한 스케일의 문제를 해결하기 위해 라우터의 엔트리가 반드시 40억개일 필요는 없다. 다음과 같이 **포워딩 테이블**에 4개의 **엔트리**만을 가지면 된다.

```
          **Prefix**              **Link Interface**

11001000 00010111 00010            0
11001000 00010111 00011000         1
11001000 00010111 00011            2
         Otherwise                 3
```

- 이렇게 하면, 라우터는 패킷의 목적지 주소의 prefix와 테이블의 엔트리를 확인한다.
    - 만약 일치한다면, 라우터는 패킷을 일치하는 해당 링크로 포워딩한다.
- 예를 들어, 패킷의 목적지 주소가 `11001000 00010111 00010110 10100001` 라고 한다면 위의 포워딩 테이블의 첫 번째 엔트리의 21비트와 일치하기 때문에 라우터는 패킷을 0번 인터페이스로 포워딩한다.
- 만약 일치하는 엔트리가 없다면 3번 인터페이스로 포워딩한다.
- 예를 들어, `11001000 00010111 00011000 10101010` 과 같이 앞의 24비트가 두 번째 엔트리와 일치하면서 21비트가 세 번째 엔트리와 일치하는 경우가 있다.
    - 이처럼 **일치하는 엔트리가 여러 개**인 경우, 라우터는 **longest prefix matching rule**을 사용하는데, 가장 길게 일치하는 엔트리의 링크 인터페이스로 패킷을 포워딩하는 것이다.

## 스위칭 (Switching)

스위칭 패브릭은 **라우터에서 매우 중요한 부분**이다. 패킷이 실제로 이 패브릭을 통해 입력 포트에서 출력 포트로 스위칭되기 때문이다.

스위칭은 다양한 방법을 통해 이루어질 수 있다. (아래 그림 참고)

- **메모리를 통한 스위칭 (Switching via memory)**
    - 가장 간단하고 초기의 라우터는 전통적인 컴퓨터였다. 여기서 입력  포트와 출력 포트 간의 스위칭은 **CPU(라우팅 프로세서)의 직접적인 제어** 하에 이루어졌다.
    - 전통적인 운영체제에서 입력 포트와 출력 포트는 전통적인 I/O 장치와 같이 기능했다.
    - 입력 포트가 패킷을 받으면 **라우팅 프로세서**에게 **인터럽트**로 시그널을 보낸다.
    - 패킷은 그 후에 입력 포트에서 **프로세서 메모리로 복사**된다.
    - 라우팅 프로세스는 **헤더에서 목적지 주소를 추출**한다.
    - 포워딩 테이블에서 **적절한 출력 포트**를 찾아서 해당 포트의 **버퍼에 패킷을 복**사한다.
    - 이러한 과정은 한번에 하나씩만 처리될 수 있다. 공유 시스템 버스(shared system bus)에서는 한번에 하나의 메모리 읽기/쓰기만이 가능하기 때문이다.
- **버스를 통한 스위칭 (Switching via a bus)**
    - 입력 포트는 **공유 버스(shared bus)**를 통해 패킷을 출력 포트로 **직접 전송**한다.
        - 이 과정에서 라우팅 프로세서는 어떤 간섭도 하지 않는다.
    - 모든 출력 포트가 이 패킷을 받지만 해당 **라벨과 일치하는 포트**만이 패킷을 가진다.
        - 라벨은 그 후 버려진다.
    - 여러 패킷이 각기 다른 입력 포트로부터 라우터에 동시에 도착하면, **한 번에 한 패킷만** 버스를 통과할 수 있기 때문에 한 패킷을 제외한 모든 패킷은 **기다려야 한다.**
    - 스위칭의 속도는 **버스의 속도**로 제한된다.
- **상호접속된 네트워크를 통한 스위칭 (Switching via an interconnection network)**
    - 위와 같은 대역폭 한계를 극복하는 방법은 더 정교한 상호 연결 네트워크를 사용하는 것이다.
    - 크로스바 스위치는 2N개의 버스로 이루어진 상호연결된 네트워크로, N개의 입력 포트와 N개의 출력 포트를 연결하는 형태이다.
    - 각 수직 버스(vertical bus)는 수평 버스(horizontal bus)와 교차하며, 이 교차점은 스위치 패브릭 제어 장치(switch fabric controller)에 의해 언제든지 열리고 닫힐 수 있다.
    - 예를 들어 포트 A에서 포트 Y로 포워딩하고자 한다면, 제어 장치가 버스 A와 Y간의 교차점을 닫은 후에 A가 버스로 패킷을 전송한다.
    - 앞선 두 방법과 달리 한번에 여러 개의 포워딩이 가능하다.

# The Internet Protocol (IP): IPv4, Addressing, IPv6, and More

이 섹션에서는 오늘날의 인터넷과 유명한 인터넷 프로토콜(IP)에 대한 네트워크 계층의 주요 측면에 초점을 맞출 것이다.

오늘날 쓰이는 IP 버전에는 두 가지가 있는데, 우선 광범위하게 쓰이는 IP 프로토콜 버전 4(IPv4)를 살펴보고 이를 대체하기 위해 제안된 IP 버전 6를 알아볼 것이다.

## IPv4 데이터그램 형식 (IPv4 Datagram Format)

앞에서 이야기했다시피, 인터넷의 네트워크 계층 패킷은 **데이터그램**으로 불린다. 우선 IPv4 데이터그램의 문법과 의미를 알아보자. IPv4 데이터그램 형식은 아래 그림과 같다.

![Untitled](/assets/img/cs/network/network04/Untitled%207.png)

- **Version number**
    - 데이터그램의 IP 프로토콜 버전을 명시한다. 라우터는 이를 통해 IP 데이터그램을 어떻게 해석할지에 대해 결정할 수 있다.
- **Header length**
    - IPv4 데이터그램은 임의의 개수의 옵션을 지닐 수 있기 때문에 데이터그램의 어디서부터 **페이로드가 시작**되는지를 알려준다.
        - payload : 송신 측에서 보내고자 하는 **데이터**를 의미한다. **헤더와 구분**된다.
    - 대부분의 IP 데이터그램은 **옵션 필드**를 포함되지 않아서 **헤더가 20바이트**이다.
- **Type of service**
    - type of service(TOS) 비트는 다양한  종류의 IP 데이터그램을 구별할 수 있도록 해준다.
- **Datagram length**
    - IP 데이터그램의 전체 길이(헤더+데이터)를 바이트 단위로 측정한 것이다.
- **Identifier, flags, fragmentation offset**
    - IP 단편화와 관련되어 있다.
- **Time-to-live**
    - 데이터그램이 네트워크 상에서 영원히 순환하지 못하도록 방지한다.
    - 라우터에 의해 처리될 때마다 필드가 1씩 감소한다.
    - TTL 필드가 0이 되면 라우터는 해당 데이터그램을 버린다.
- **Protocol**
    - IP 데이터그램이 목적지에 도달했을 때에만 주로 사용된다.
    - 해당 필드의 값은 이 데이터그램의 **데이터 영역이 전해져야 할 특정한 전송 계층 프로토콜**을 지칭한다.
    - 예를 들어, 값이 6이면 데이터 영역은 TCP로 전해지고, 값이 17이라면 데이터는 UDP로 전해진다.
- **Header checksum**
    - 수신된 IP 데이터그램에서 라우터가 **비트 에러를 발견**하는 것을 돕는다.
    - 각 2 바이트를 숫자로 처리하여 1의 보수 계산을 통해 **값을 더한다.**
- **Source and destination IP addresses**
    - 출발지에서 데이터그램을 생성하면, 출발지 IP 주소 필드에 해당 주소를 기입하고, 최종적인 목적지를 목적지 IP 필드에 기입한다.
- **Options**
    - IP 헤더가 확장될 수 있도록 한다.
    - 거의 드물게 사용된다.
- **Data (payload)**
    - 대부분의 경우, 목적지에 전달될 **전송 계층 세그먼트**(TCP 또는 UDP)를 담고있다.
    - 하지만 ICMP 메시지와 같이 **다른 형식의 데이터**도 담을 수 있다.

## IPv4 데이터그램 단편화 (IPv4 Datagram Fragmentation)

모든 링크 계층의 프로토콜이 같은 크기의 네트워크 계층 패킷을 가질 수는 없다. 일부 프로토콜은 큰 데이터그램을 지닐 수 있고, 다른 프로토콜은 적은 데이터그램만을 지닐 수 있다.

- **Maximum Transmission Unit(MTU)** : 링크 계층 프레임이 지닐 수 있는 최대 데이터의 양
- 각 IP 데이터그램은 라우터 간에 이동을 위해 링크 계층 프레임 내부로 캡슐화된다.
    - 이 때문에 링크 계층 프로토콜은 IP 데이터그램의 길이에 엄격한 제한을 둔다.
    - 문제는 송신자와 목적지 사이의 경로에 있는 각 링크들이 **다른 링크 계층 프로토콜**을 사용할 수 있고, 이들의 **MTU가 다를 수 있다**는 점이다.

### 단편화 (Fragmentation)

- 예를들어, 내가 **라우터**라고 가정하자.
    1. 내가 한 링크로부터 IP 데이터그램을 받으면, 포워딩 테이블을 확인해서 나갈 링크를 결정한다.
    2. 만약 나갈 링크의 MTU가 **IP 데이터그램의 길이보다 작다면?**
        1. 데이터그램의 payload를 두 개 이상의 IP 데이터그램으로 **쪼개고**,
        2. 각각의 작은 IP 데이터그램들을 독립된 **링크 계층 프레임에다가 캡슐화**하여
        3. 이 프레임들을 나갈 링크를 통해 보낸다.
        4. 여기서 각각의 작은 데이터그램을 **조각(fragment)**이라고 부른다.

### 재조립 (Reassembly)

- 이 조각들은 목적지의 전송 계층에 가기 전에 **다시 합쳐져야** 한다.
    - 하지만 이는 IPv4의 성능을 저하시킬 수 있다.
    - 따라서 IPv4에서는 데이터그램의 재조립을 네트워크 라우터가 아닌 **종단 시스템(end-system)**이 하도록 하였다.

![Untitled](/assets/img/cs/network/network04/Untitled%208.png)

목적지 호스트가 일련의 데이터그램을 같은 출발지로부터 받았다면, 어떤 것이 큰 데이터그램의 조각인지를 알아내야 한다. 이를 위해 `identification`, `flag`, `fragmentation offset`과 같이 세 개의 필드를 사용한다.

- 데이터그램이 생성되면 송신 호스트는 출발, 도착 주소와 함께 **식별 번호(identification number)**를 기입한다.
    - 주로 식별 번호는 각 데이터그램 별로 증가한다.
- 라우터가 데이터그램을 쪼개야한다면, 각 조각들에는 출발 주소, 도착 주소와 함께 **원본 데이터그램의 식별 번호**가 기입된다.
- 목적지에서 일련의 데이터그램을 받으면 식별 번호를 통해 어떤 것이 **큰 데이터그램의 조각**인지를 알 수 있다.

## IPv4 주소 지정(IPv4 Addressing)

IP는 각 호스트와 라우터 인터페이스가 **고유한 IP 주소**를 갖도록 요구한다.

각 IP 주소는 **32비트 길이**이므로, 총 2^32개의 IP 주소가 존재할 수 있다. 이러한 주소들은 주로 **닷 데시멀 표기법**으로 작성된다.

- **닷 데시멀 표기법(dot-decimal notation)**
    - 주소의 각 바이트가 10진수 형태로 쓰여지며 점(.)으로 구분된다.
    - 예
        - 바이너리 표기법(binary notation)으로 나타낸 주소 :  `11000001 00100000 11011000 00001001`
        - 닷 데시멀 표기법으로 나타낸 주소 : `193.32.216.9`

모든 호스트와 라우터의 각 인터페이스는 고유한 IP 주소를 가진다.

이러한 주소들은 마구잡이로 정해지지 않고, 주소의 일부분이 **연결된 곳의 서브넷**으로 정해진다.

![Untitled](/assets/img/cs/network/network04/Untitled%209.png)

- 위 그림에서 왼쪽의 세 **호스트**와 이들의 **라우터 인터페이스**의 IP 네 개 모두 `223.1.1.xxx` 의 형태이다. (IP 주소의 왼쪽 끝 24비트가 동일하다는 의미)
- 이 네 개의 인터페이스들은 **서브넷(subnet)**을 형성한다.
- IP 주소 지정은 위 서브넷에 주소를 할당한다 : `223.1.1.0/24`
    - /24 표기법(slash-24 notation)은 **서브넷 마스크(subnet mask)**라고 부른다.
    - 이는 32비트 중 제일 왼쪽의 24비트가 **서브넷 주소로 정의된다**는 것을 의미한다.
- 아래 그림은 위 그림에서의 서브넷 주소를 나타낸 것이다.

![Untitled](/assets/img/cs/network/network04/Untitled%2010.png)

서브넷의 IP 주소가 반드시 호스트-라우터 간에 연결된 하나의 주소로 한정되지는 않는다.

아래 예시를 보자. `223.1.1.0/24`, `223.1.2.0/24`, `223.1.3.0/24`와 같이 호스트-라우터 간의 연결도 있지만, 라우터-라우터 간의 연결인 `223.1.7.0/24`, `223.1.8.0/24`, `223.1.9.0/24`도 존재한다.

![Untitled](/assets/img/cs/network/network04/Untitled%2011.png)

- **서브네팅 (subnetting)**
    - 큰 네트워크를 작은 서브넷(subnetwork)으로 **나누는** 것
- **슈퍼네팅 (supernetting)**
    - 작은 네트워크를 **합치는** 것

### CIDR (**Classless Interdomain Routing)**

인터넷의 **주소 할당 방식**이다. 이 방법은 서브넷 주소 지정과 마찬가지로, **32비트**의 IP 주소가 **두 부분**으로 나뉘고 `a.b.c.d/x` (x: 주소의 앞 부분의 비트의 개수)와 같은 닷 데시멀 형식를 가진다.

### prefix

a.b.c.d/x 형식의 주소에서 가장 중요한 **x개의 비트**로 **네트워크 영역**을 구성한다. 하나의 조직은 일반적으로 같은 prefix의 연속적인 주소를 할당한다. 조직의 네트워크 밖의 라우터들은 이 **prefix 비트만을 고려**한다.

**32-x개의 비트**는 조직 내에서 **기기들을 구별**하는데 사용된다. 다시 말하면, 조직 내에 있는 라우터가 패킷을 포워딩할 때 사용한다.

### 클래스 기반 주소 지정 (Classful Addressing)

CIDR이 채택되기 전의 방식

IP 주소의 **네트워크 영역**의 길이가 8, 16, 24 비트로 **한정**되었다.

8-, 16-, 24-비트 서브넷 주소를 가진 서브넷들은 각각 클래스 A, B, C로 불렸다.

- **이 방식의 문제점**
    - 작은 규모의 서브넷들이 점점 늘어나는 상황
    - 클래스 C(/24)는 2^8-2=254개의 호스트만을 수용할 수 있다.
        - 2개를 빼는 이유 : `x.x.x.0`은 네트워크 주소와 `x.x.x.255`는 브로드캐스트 주소로 사용되기 때문이다.
            - 브로드캐스트 주소(broadcast address) : 해당 주소로 데이터그램을 보낸다면, 같은 서브넷에 속한 모든 호스트가 해당 메시지를 받게된다.
    - 반면 클래스 B(/16)는 2^16-2=65634개의 호스트까지 수용할 수 있다.
    - 이 방법으로는 조직의 호스트의 수가 2000개일 때 클래스 B를 사용해야한다.
        - 이는 클래스 B의 급격한 감소와 할당된 주소 공간의 낮은 사용률을 초래한다.
        - 65634-2000=63634개의 주소는 다른 조직에서 사용될 수 없게 될 것이기 때문이다.

### 주소 블록의 획득 (Obtaining a Block of Addresses)

1. 네트워크 관리자가 ISP(Internet Service Provider)와 접촉한다.
2. ISP는 이전에 할당받은 주소 블록에서 주소를 제공한다. (아래 코드블록 참고)
    1. ISP가 할당받은 주소 블록이 `200.23.16.0/20`이라고 하자.
    2. ISP는 주소 블록을 8개의 동일한 크기의 연속적인 주소 블록으로 나눈다.
    3. 이 블록 중 하나를 조직에게 준다.

![address](/assets/img/cs/network/network04/isp%20address.png)


위 방법은 호스트가 ISP로부터 주소들을 받는 과정이다. 그럼 ISP는 주소 블록을 어떻게 받을까?

- **ICANN (Internet Corporation for Assigned Names and Numbers)**
    - ISP는 **ICANN**으로부터 IP 주소를 할당받는다.
    - ICANN의 역할
        - IP 주소를 할당하는 것
        - DNS 루트 서버를 관리하는 것
        - 도메인 이름을 할당하고 이름 관련 분쟁을 해결하는 것

### 호스트 주소의 획득: DHCP (Dynamic Host Configuration Protocol)

조직이 주소 블록을 획득한 뒤에 호스트와 라우터 인터페이스에 각 IP 주소를 할당한다.

라우터에는 IP 주소를 주로 수동으로 설정하지만 **호스트에는 DHCP (Dynamic Host Configuration Protocol)를 사용**하여 설정한다.

DHCP는 **호스트**가 **자동으로 IP 주소를 획득**하도록 한다.

호스트가 네트워크에 연결할 때마다 전에 쓰던 **같은 IP 주소를 할당**해 주거나, 매번 **임시 IP 주소(temporary IP address)를 할당**할 지는 네트워크 관리자가 설정하는 방식에 따른다.

**plug-and-play** 또는 **zeroconf(zero-configuration)**이라고도 불린다. 네트워크 관련 일들을 자동화해주기 때문이다.

DHCP는 가정, 기업, 무선 LAN 등에서 광범위하게 사용된다.

![Untitled](/assets/img/cs/network/network04/Untitled%2012.png)

DHCP는 **클라이언트-서버 프로토콜**이다.

- 클라이언트 → 네트워크 설정 정보(IP 주소 등)를 얻으려는 새로 들어온 호스트

단순한 경우로는 각 서브넷이 **DHCP 서버를 따로** 갖고있지만, 그렇지 않은 경우도 있다.

위 그림처럼 DHCP 서버가 **한 서브넷에 위치**한 경우

- DHCP 서버의 주소를 **DHCP relay agent(주로 라우터)가 알려줘야** 한다.

![Untitled](/assets/img/cs/network/network04/Untitled%2013.png)

새롭게 호스트가 들어온다면 DHCP 프로토콜은 다음 과정들을 거친다. (위 그림 참고, yiaddr : your Internet address)

1. DHCP 서버 발견(DHCP server discovery)
    - 새롭게 들어온 호스트가 할 일은 소통할 DHCP 서버를 찾는 것이다.
    - 이는 **DHCP 발견 메시지**를 통해 이루어진다.
        - DHCP 발견 메시지(DHCP discover message) : 클라이언트가 67번 포트에 UDP 패킷을 보낸다. UDP 패킷은 IP 데이터그램 안에 캡슐화되어진다.
        - 메시지는 255.255.255.255의 **브로드캐스트 목적지 IP 주소**로 보낸다.
2. DHCP 서버 제공(DHCP server offer(s))
    - DHCP 발견 메시지를 받은 DHCP 서버는 클라이언트에게 **DHCP 제공 메시지**로 응답한다.
        - 이 과정도 IP 브로드캐스트 주소를 사용한다.
    - DHCP 제공 메시지가 포함하는 내용은 다음과 같다.
        - 수신된 발견 메시지에 대한 트랜잭션 ID
        - IP 주소의 제안
        - 네트워크 마스크
        - IP 주소 임대 시간(address lease time : IP 주소가 유효한 시간. 주로 수 시간 ~ 수 일로 설정)
3. DHCP 요청(DHCP request)
    - 새로 들어온 클라이언트는 하나 이상의 IP 제안 중에서 하나를 고른다.
    - 제안을 보낸 곳에 **DHCP 요청 메시지(DHCP request message)**로 응답한다.
    - 설정 관련 매개변수들은 그대로 되풀이한다.
4. DHCP ACK
    - 서버가 받은 DHCP 요청 메시지에 대해 **DHCP ACK 메시지**로 응답한다.

클라이언트가 DHCP ACK를 받으면 소통이 완료된다. 이제 DHCP 할당 IP 주소를 대여한 시간동안 사용할 수 있다.

## 네트워크 주소 변환 (Network Address Translation, NAT)

위에서 살펴본 바와 같이 **모든 IP 지원 장치들은 IP 주소를 필요**로 한다. 소규모 사무실(SOHO, small office/home office)의 서브넷이 늘어나는 것은 ISP가 이러한 SOHO의 IP 기기들을 위해 IP 주소 범위를 할당해야하는 것을 암시한다. 서브넷이 커진다면 더 큰 영역의 주소가 할당되어야 한다.

하지만 ISP가 이미 인접한 영역의 주소를 할당했다면 어떻게 될까? 그리고 집주인이 IP 주소를 관리하기 위해 처음에 알아야 할 내용은 뭐가 있을까?

이러한 상황에서 요즘에 많이 쓰이는 단순한 접근법이 바로 **네트워크 주소 변환(network address translation, NAT)**이다.

![Untitled](/assets/img/cs/network/network04/Untitled%2014.png)

위 그림에서 가운데 라우터는 **NAT 가능 라우터(NAT-enabled router)**이다. 오른쪽의 홈 네트워크를 보면 네 개의 인터페이스 모두 `10.0.0.0/24`로 같은 서브넷 주소를 사용한다.

### 사설 주소 범위 (realm with private addresses)

`10.0.0.0/8`은 **사설망(private network)**을 위해 예약된 IP 주소 공간의 세 부분 중 하나이며 **사설 주소 범위**라고도 부른다.

사설 주소 범위는 **해당 네트워크 안에서만 의미**를 갖는 주소를 의미한다.

하지만 홈 네트워크 밖의 큰 **글로벌 인터넷에서는 해당 주소를 사용할 수 없다.**(출발지, 도착지 주소로도) 같은 주소 블록을 사용하는 네트워크가 굉장히 많기 때문이다. 그렇다면 **주소가 고유해야하는 글로벌 인터넷** 상에서 패킷이 송수신되려면 어떻게 해야할까?

- NAT 라우터의 IP 주소
    - NAT 가능 라우터는 밖에서는 위 그림과 달리 **하나의 IP주소를 가진 하나의 기기**로 동작한다. 해당 라우터를 나가는 모든 트래픽은 `138.76.29.7`의 출발지 IP 주소를 갖고, 들어오는 모든 트래픽도 같은 도착지 주소를 갖는다. 밖에서는 홈 네트워크의 세부사항들을 가리는 셈이다.

WAN에서 NAT 라우터로 들어온 모든 데이터그램이 **같은 목적지 주소**를 갖는다면 라우터는 **패킷을 어디로 포워딩해야 할지** 어떻게 알까?

- NAT 라우터의 포워딩 방법
    - **NAT 변환 테이블(NAT translation table)**을 사용한다.
        - 테이블의 엔트리에 IP 주소와 함께 **포트 번호**도 포함시킨다.
- **예시** (위 그림 참고)
    1. 홈 네트워크의 호스트 10.0.0.1 뒤에 있는 사용자가 특정 웹 서버(80번 포트)에 웹 페이지를 요청한다.
    2. 호스트 10.0.0.1은 **임의의 출발지 포트 번호** 3345번을 할당하여 데이터그램을 LAN에 보낸다.
    3. NAT 라우터는 데이터그램을 받아서 새로운 출발지 포트 번호 5001번을 데이터그램에 생성한다.
        - 이 때의 포트 번호는 NAT 변환 테이블에 없는 포트 번호 중 아무 번호나 선택할 수 있다.
    4. 라우터의 NAT는 테이블에 **엔트리를 추가**한다.
    5. 웹 서버는 목적지 IP 주소가 **NAT 라우터의 주소**이고 목적지 포트 번호가 **5001**인 데이터그램으로 응답을 보낸다.
    6. 해당 데이터그램이 라우터에 들어오면, 라우터는 NAT 변환 테이블을 인덱싱해서 **적절한 IP 주소**와 **목적지 포트 번호**를 찾는다.
    7. 라우터는 **데이터그램의 목적지 주소와 목적지 포트 번호를 수정**하고, 해당 데이터그램을 홈 네트워크에 **포워딩**한다.
- NAT에 대한 논쟁
    1. 포트 번호는 호스트가 아니라 **프로세스의 주소를 지정**하는 것이 본래 목적이다.
        - 실제로 서버의 경우 이 점이 문제가 될 수 있다.
        - 해결책으로는 NAT traversal 도구와 Universal Plug and Play(UPnP)가 있다.
    2. 라우터는 3 계층(네트워크 계층)이라 패킷을 네트워크 계층까지만 처리해야 하지만 NAT는 이 원칙에 위배된다.

## IPv6

1990년대 초기에 IPv4의 뒤를 잇기 위해 **IPv6**의 개발이 시작되었다. 새로운 서브넷들과 IP 노드들을 담기엔 **IPv4의 32비트 주소 공간은 턱없이 부족**했기 때문이다.

### IPv6 데이터그램 형식

IPv6가 **IPv4와 다른 중요한 점**들은 다음과 같다.

- 확장된 주소 지정 능력 (Expanded addressing capabilities)
    - IP 주소의 크기를 32비트에서 128비트로 늘렸다.
    - 유니캐스트와 멀티캐스트에 더해 새로운 주소 형식인 애니캐스트(anycst) 주소를 도입하여 그룹 내 어떤 호스트에게든 데이터그램을 전달할 수 있다.
        - 유니캐스트(Unicast) : 고유 주소로 식별된 하나의 네트워크 목적지에 1:1로 트래픽 또는 메시지를 전송하는 방식
        - 멀티캐스트 (Multicast) : 한 번의 송신으로 메시지나 정보를 목표한 여러 컴퓨터에 동시에 전송하는 방식
- 간결한 40바이트의 헤더 (A streamlined 40-byte header)
    - IPv4가 필드를 선택적으로 둔 것과 달리 **40바이트로 고정된 헤더**는 라우터가 **IP 데이터그램을 더욱 빠르게 처리**하도록 해준다.
- 플로우 레이블링 (Flow labeling)
    - **플로우(flow)** : 송신자는  특별한 처리를 요청하는 특정 흐름에 속하는 패킷에 표시를 할 수 있게 해준다.
    - 예를 들어, 오디오와 비디오 전송은 플로우로 처리될 것이다.
    - 반면 전통적인 애플리케이션(파일 전송, 이메일 등)은 플로우로 처리되지 않을 것이다.
    - 높은 우선순위의 사용자가 보낸 트래픽도 플로우로 처리될 수 있다.(더 나은 트래픽을 위해 비용을 지불하는 사용자 등)

**IPv6에 정의된 필드**는 다음과 같다.(아래 그림 참고)

- Version
    - IP 버전 번호를 나타낸다.
    - IPv6는 여기에 6을 넣는다. 하지만 4를 넣는다고 IPv4 데이터그램으로 만들 수는 없다.
- Traffic class
    - IPv4의 TOS 필드와 유사하다.
    - 플로우 내의 특정 데이터그램에 우선순위를 줄 수 있다.
    - 혹은 특정 애플리케이션의 데이터그램을 다른 애플리케이션의 데이터그램보다 높은 우선순위를 줄 수 있다.
- Flow label
    - 데이터그램의 플로우를 식별하는데에 사용된다.
- Payload length
    - 40 바이트의 헤더 뒤에 올 바이트 수를 나타낸다.
- Next header
    - 해당 데이터그램의 **내용(데이터 필드)이 전해져야 할 프로토콜**을 나타낸다.
    - IPv4의 protocol 필드와 같은 값을 사용한다.
- Hop limit
    - IPv4의 Time-to-live 필드와 유사하다.
    - 해당 필드의 값은 각 라우터를 거칠 때마다 1씩 감소한다.
    - 값이 0이 되면, 데이터그램은 버려진다.
- Source and destination addressses
- Data
    - IPv6 데이터그램의 페이로드 영역이다.
    - 데이터그램이 목적지에 도착하면, IP 데이터그램에서 페이로드가 제거되고 (next header 필드에 명시된)프로토콜에 전달된다.

![Untitled](/assets/img/cs/network/network04/Untitled%2015.png)

IPv4에는 있지만 **IPv6에는 없어진 필드**

- Fragmentation/reassembly
    - IPv6에서는 단편화(fragmentation)와 재조립(reassembly)을 **중간 라우터에서 할 수 없다.**
        - 오로지 **출발지와 목적지에서만 가능**하다.
    - 만약 라우터가 받은 IPv6 데이터그램이 포워딩 되기에 너무 크다면, 해당 데이터그램을 버리고 “패킷이 너무 큼"이라는 ICMP 에러 메시지를 송신자에게 보낸다.
    - 이러한 작업들은 시간이 쓰이는 작업이기 때문에, 라우터에서 해당 기능을 없애서 네트워크의 **IP 포워딩을 빠르게** 한다.
- Header checksum
    - 해당 기능이 전송 계층(TCP, UDP)과 링크 계층(이더넷)의 프로토콜에서 수행되기 때문에 삭제되었다.
- Options
    - 해당 필드가 사라져서 **헤더의 길이가 40바이트로 고정**되었다.

### IPv4에서 IPv6로의 이동

IPv6 가능 시스템들은 하위 호환이 가능한데 반해, IPv4 가능 시스템들은 상위 호환이 불가능하다. 이를 위해 몇 가지의 방법들이 있다.

1. **flag day 선언**
    - 모든 인터넷 기기들이 꺼지고 IPv4에서 IPv6로 업그레이드될 일시를 지정하는 것이다.
    - 현실성이 없다.
2. **터널링 (tunneling)**
    
    ![Untitled](/assets/img/cs/network/network04/Untitled%2016.png)
    
    - IPv6 노드 두 개(위 그림에서 B, E)가 상호작용하려고 하지만 **중간에 IPv4로 연결**되어 있다고 가정하자.
    - 이렇게 **IPv6 라우터 사이에 있는 IPv4 라우터**를 **터널 (tunnel)**이라고 부른다.
    - 송신 측의 IPv6 노드(B)는 IPv6 데이터그램을 IPv4 데이터그램의 데이터 필드에 넣는다.
    - IPv4 데이터그램은 수신 측 IPv6 노드(E)로 주소 지정되고 첫 번째 노드(C)로 전송된다.
    - 터널 중간의 IPv4 라우터들은 IPv4 데이터그램을 다른 것들과 마찬가지로 라우팅한다.
    - 수신 측 IPv6 노드가 IPv6를 완전히 받는다.
        - 해당 노드는 IPv4 데이터그램의 프로토콜 번호가 41번임을 통해 IPv6가 안에 있다는 것을 알 수 있다.
        - IPv6 데이터그램을 수신했을 때와 동일하게 라우팅한다.
