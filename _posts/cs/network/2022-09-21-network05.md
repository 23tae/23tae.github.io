# Introduction

이 챕터에서는 앞에서 살펴본 **포워딩 테이블**과 **플로우 테이블**이 어떻게 **계산**되고 **유지**되며 **설치**되는지에 대해 공부할 것이다.

앞 장에서는 이러한 과정을 위해 두 가지 접근법이 있다고 배웠다.

- **라우터별 제어 (Per-router control)**
    
    ![Untitled](/assets/img/cs/network/network05/Untitled.png)
    
    - 위 그림은 **각 라우터마다 라우팅 알고리즘**을 따로 실행시키는 모습을 나타낸다.
        - 각 라우터가 포워딩과 라우팅 기능을 포함한다.
    - 각 라우터에는 다른 라우터의 구성요소와 통신하는 라우팅 구성요소가 있다.
        - 이들은 통신을 통해 **포워딩 테이블의 값을 계산**한다.
    - OSPF, BGP 프로토콜이 이 방식을 기반으로 한다.
- **논리적으로 중앙 집중된 제어 (Logically centralized control)**
    
    ![Untitled](/assets/img/cs/network/network05/Untitled%201.png)
    
    - 위 그림은 **논리적으로 중앙 집중된 제어 장치(logically centralized controller)**가 각 라우터에서 사용될 **포워딩 테이블의 값을 계산하고 분배**해주는 모습을 나타낸다.
    - 제어 장치는 각 라우터의 **control agent (CA)**와 상호작용하여 **플로우 테이블**을 설정하고 관리한다.
    - 일반적으로 CA는 최소한의 기능만 가지고 있다.
        - 이들의 기능은 제어 장치와 통신하고 제어 장치의 명령을 수행하는 것이다.
    - CA는 다른 CA들과 직접 상호작용하거나 포워딩 테이블을 계산하지 않는다. (주요 차이점)
    - SDN이 이 개념을 채택했다.

# Routing Algorithm

이 섹션에서는 라우터들의 네트워크를 따라 송신자에서 수신자로 가는 **좋은(최소한의 비용이 드는) 경로**를 찾기 위해 **라우팅 알고리즘**에 대해 공부할 것이다.

네트워크 제어 평면이 위의 접근법 중 어떤 것을 사용하던지간에, 패킷이 송신자에서 수신자로 가는 과정에서 라우터를 지나가는 순서가 반드시 정해져 있다는 것에 주목한다.

다양한 라우팅 문제들을 표현하기 위해 그래프를 사용할 것이다.

- `graph G = (N, E)`
    - N : 노드들의 집합. E : 엣지들의 집합
    - 각 엣지 : N 중에 한 쌍의 노드
- 네트워크 계층에서의 의미
    - 노드 : 라우터 (패킷의 포워딩이 일어나는 지점)
    - 엣지 : 라우터 간의 물리적 링크

![Untitled](/assets/img/cs/network/network05/Untitled%202.png)

위 그림에서 엣지의 값 : **비용**

- 비용을 **결정하는 요인**은 아래와 같다.
    - 물리적인 링크의 길이
    - 링크 속도
    - 링크와 관련된 금전적인 비용
- 여기서는 비용이 결정되는 방법은 생각하지 않고 주어져 있다고만 가정할 것이다.
- E 중에 edge (x, y)에 대해 c(x, y) : 노드 x와 y 사이 엣지의 비용
    - pair (x, y)가 E에 속하지 않는다면 c(x, y) = ∞

또한 여기서는 무향 그래프(undirected graph)에 대해서만 고려할 것이다.

- edge (x, y) = edge (y, x)
- c(x, y) = c(y, x)

(x, y)가 E에 속할 때, 노드 y는 노드 x의 **이웃(neighbor)**이라고 부른다.

알고리즘의 목표는 출발지와 목적지 사이의 최소 비용의 경로를 찾는 것이다.

- graph G = (N, E)에서 경로란 노드의 순서(x1, x2, …, xp)를 뜻한다.
- 경로 (x1, x2, …, xp)에서의 비용이란 단순히 모든 엣지 비용을 더한 값이다.
    - c(x1, x2) + c(x2, x3) + … + c(x(p-1), xp)

위 그림에서 출발지 노드 u와 목적지 노드 w 사이의 최저 비용 경로(least-cost path)는 (u, x, y, w)로 경로 비용이 3이다.

- 만약 모든 엣지의 비용이 동일하다면, 최저 비용 경로는 최단 경로가 된다.

## 분류 기준

1. **is Centralized?**
    - **중앙 집중형 라우팅 알고리즘 (centralized routing algorithm)**
        - **네트워크 전체에 대한 완전한 지식**을 사용해서 최소 비용 경로를 계산한다.
            - 지식 : **연결 상태(connectivity), 링크 비용(link costs)**
            - 이 정보들은 알고리즘이 계산을 수행하기 전에 얻어야 한다.
        - 계산은 **한 지점**(논리적으로 중앙 집중된 제어 장치 등)에서 한번에 수행될 수도 있고, **각 라우터의 라우팅 구성요소**에 복제되어 수행될 수도 있다.
        - **link-state (LS)** 알고리즘으로 불린다.
    - **분산형 라우팅 알고리즘 (decentralized routing algorithm)**
        - 최소 비용 경로의 계산은 **라우터들**에 의해 **반복적이고 분산된 방식**으로 수행된다.
        - 어떠한 노드도 모든 네트워크 링크의 비용에 대한 **완전한 정보를 알지 못한다.**
        - 각 노드는 자신과 **직접 연결된 링크의 비용**에 대한 정보만 아는 상태로 시작한다.
        - 노드는 **반복적인 계산**과 **이웃 노드와의 정보의 교환** 과정을 통해 목적지까지의 최소 비용 경로를 **점진적으로 계산**해나간다.
        - **distance-vector (DV)** 알고리즘이 있다.
            - 각 노드는 다른 모든 노드에 대한 **예상 비용을 벡터로 관리**하기 때문이다.
2. **is Static?**
    - **정적 라우팅 알고리즘 (static routing algorithm)**
        - 경로가 시간을 따라 **매우 느리게** 변한다.
        - 종종 사람의 개입이 있다. (예. 사람이 직접 링크 비용을 수정하는 경우)
    - **동적 라우팅 알고리즘 (dynamic routing algorithm)**
        - 네트워크 트래픽 양(network traffic load)이나 토폴로지(topology)가 **변하면 라우팅 경로를 변경**한다.
            - **토폴로지(topology)** : 컴퓨터 네트워크의 요소들(링크, 노드 등)을 물리적으로 연결해 놓은 것
        - 주기적으로 수행될 수도 있고, 변화에 즉각적으로 대응해서 수행될 수도 있다.
        - 장점 : 네트워크 변화에 즉각 대응한다.
        - 단점 : routing loops와 route oscillation과 같은 문제에 더욱 취약하다.
3. **is Load-Sensitive?**
    - **부하 민감 알고리즘 (load-sensitive algorithm)**
        - 링크의 혼잡 상태를 반영하기 위해 경로 비용이 유동적으로 다양해진다.
        - 만약 한 링크가 혼잡하여 높은 비용이 발생한다면, 알고리즘은 그 주변 경로를 선택하는 경향이 있다.
        - 초기 ARPAnet 라우팅 알고리즘에서 사용했다.
    - **부하 둔감 알고리즘 (load-insensitive algorithm)**
        - 링크 비용이 현재의 혼잡 상태를 반영하지 않는다.
        - 현재의 인터넷 라우팅 알고리즘(RIP, OSPF, BFP)에서 사용한다.

## Link-State (LS) 라우팅 알고리즘

- 이 알고리즘에서는 네트워크 토폴로지와 모든 링크 비용이 알려져있다.
    - 각 노드는 자신의 링크 상태 패킷을 모든 노드들에게 브로드캐스팅한다.
    - 이로 인해 모든 노드들이 **네트워크에 대한 동일하고 완전한 모습**을 갖게된다.
- 각 노드는 LS 알고리즘을 실행하고 최소 비용 경로의 집합을 계산할 수 있다.

아래의 예시는 LS 알고리즘의 한 종류인 **다익스트라 알고리즘(Dijkstra’s algorithm)**이다.

이 알고리즘은 한 노드에서 네트워크의 다른 모든 노드로의 최소 비용 경로를 계산한다.

이를 위해 다음의 표기가 사용된다.

- D(v) : 알고리즘의 **반복**에 따른 출발지 노드에서 목적지 v까지의 최소 비용 경로의 **비용**
- p(v) : 출발지에서 v까지의 현재의 최소 비용 경로에서의 이전 노드(v의 이웃 노드)
- N’ : 노드들의 부분 집합. 출발지부터 v까지의 최소 비용 경로를 확실하게 안다면 v는 N’에 속한다.

### 출발지 노드 u에 대한 LS 알고리즘

```c
Initialization:
	N’ = {u}
	for all nodes v
		if v is a neighbor of u
			then D(v) = c(u,v)
		else D(v) = ∞

Loop
	find w not in N’ such that D(w) is a minimum
	add w to N’
	update D(v) for each neighbor v of w and not in N’:
				D(v) = min(D(v), D(w)+ c(w,v))
		/* new cost to v is either old cost to v or known
		least path cost to w plus cost from w to v */
until N’= N
```

위의 그림 5.3의 네트워크를 예로 들어 노드 u에서부터 **모든 가능한 목적지**로의 **최소 비용 경로**를 계산해보자.

아래 표 5.1은 알고리즘의 계산을 표로 정리한 것이다. 각 줄은 반복을 마친 후의 변수들의 값이다.

![Untitled](/assets/img/cs/network/network05/Untitled%203.png)

- **초기화 단계**
    - u에서 이웃인 v, x, w로의 현재 알려진 최소 비용 경로는 2, 1, 5이다.
        - u에서 w로의 직접적인 링크의 비용이 5이기 때문에 w로의 비용은 5가 된다. (이후에 더 낮은 비용이 존재할 지라도 현재로서는 최소이기 때문에)
    - y와 z로의 비용은 무한대로 설정된다.
        - 직접적으로 연결되어있지 않기 때문이다.
    - N’ = {u}
- **첫 번째 반복**
    - 이전 반복에서 최소의 비용을 가지면서 N’에 포함되지 않은 노드를 찾는다.
        - x가 최소의 비용인 1을 갖는다.
        - 따라서 x를 N’에 추가한다.
    - 모든 노드 v에 대해 `D(v) = min(D(v), D(w)+ c(w,v))` 를 수행한다.
        - 표 5.1의 step1과 같이 수행 결과에 따라 **D(v)를 업데이트**한다.
    - v로의 경로에 대한 비용은 변하지 않는다.
    - u에서 w로는 노드 x를 거쳐가는 것이 4로 더 작기 때문에 이게 최소 비용 경로가 된다.
        - w의 u로부터의 최단 경로에서의 predecessor는 x가 된다.
- **두 번째 반복**
    - 노드 v와 y가 최소 비용 경로 (2)를 갖는다는 것을 알았다.
    - 같은 값이기 때문에 임의로 y를 선택해서 N’에 추가한다.
        - N’ = {u, x, y}
    - N’에 없는 노드들(v, w, z)의 비용은 `D(v) = min(D(v), D(w)+ c(w,v))` 를 통해 업데이트된다.
        - 표 5.1의 step2에 추가된다.
- **계속 반복**

LS 알고리즘이 종료되면 각 노드에 대해 (출발 노드부터의 최소 비용 경로 위에 있는)predecessor를 갖게된다. 각 predecessor에 대해 그들의 predecessor도 알 수 있고, 이렇게 해서 출발지부터 모든 목적지까지의 전체 경로를 구성할 수 있다.

노드 u의 경우, u부터 목적지까지의 최소 비용 경로 상의 다음 hop 노드를 저장하여 얻은 정보를 통해 **포워딩 테이블을 구성**할 수 있을 것이다. 아래 그림 5.4를 보면 이렇게 해서 **최소 비용 경로**와 **포워딩 테이블**을 구한 것을 알 수 있다.

### 알고리즘의 계산 복잡도

n개의 노드(출발지 제외)가 있을 때, 최악의 경우 출발지에서 목적지로의 최소 비용 경로를 찾기위해 몇 번의 계산이 필요할까?

- 첫 번째 반복에서 N’에 속하지 않으면서 최소 비용을 가진 노드 w를 찾기 위해 **n개의 노드들을 전부 탐색**해야 한다.
- 두 번째 반복에서 n-1개의 노드들을 탐색해야 한다.
- 세 번째 반복에서는 n-2개의 노드를 탐색하고 같은 방식으로 계속 탐색해나간다.
- 따라서 모든 반복을 하는동안 총 `n(n + 1)/2`개의 노드들을 탐색해야 한다.

따라서 최악의 경우의 복잡도는 `O(n^2)`이다.

### 발생 가능한 문제 : Oscillations

아래 그림 5.5는 **링크 비용**이 **링크에 가해지는 부하**와 동일한 단순한 네트워크 토폴로지이다. (겪게 될 딜레이가 반영됐다고 볼 수 있다.)

이 상황에서는 링크 비용이 비대칭하다. c(u, v)와 c(v, u)는 link (u, v)의 양방향의 부하가 동일한 경우에만 같다.

1. 현재 x, y, z 모두 w로 트래픽을 보내는 상황이다. y는 e만큼의 트래픽을 주입했다. 아래의 a가 이 상황에서의 링크 비용이다.
2. 다음 번 LS 알고리즘이 실행되면, 노드 y는 시계 방향으로의 비용이 1이고, 반시계 방향(현재 사용중인 경로)의 비용은 1+e라는 것을 알게된다. 따라서 y의 w로의 최소 비용 경로는 시계 방향이 된다. 나머지 노드들도 같은 이유로 반시계 방향으로 라우팅하게 된다.
3. 다음 번 LS 알고리즘이 실행되면, x, y, z 모두 트래픽을 시계 방향으로 다시 라우팅한다.
    
    ![Untitled](/assets/img/cs/network/network05/Untitled%205.png)
    

이러한 상황을 **oscillation**이라고 한다.

- Oscillation의 해결책
    1. 링크 비용을 트래픽에 의존하지 않도록 강제하는 방법
        - 라우팅의 목표(혼잡한 링크를 피하는 것)에 어긋난다.
    2. 모든 라우터가 한 번에 LS 알고리즘을 실행하지 않도록 보장하는 방법
        - 주기적으로 LS 알고리즘을 실행하더라도 각 노드끼리 실행이 겹치지 않을 수 있다.
        - 하지만 인터넷은 **자체 동기화(self-synchronize)**가 가능하다.
            - 같은 주기로 다른 시점에 처음 실행되더라도 결국에는 **알고리즘 실행 시점이 라우터에서 동기화**된다.
        - 이를 피하는 법은 각 라우터 별로 LSA를 보내는 시간을 랜덤화하는 것이다.
            - **LSA(Link-State Advertisement, 링크 상태 광고)** : 인접 라우터에 이르는 소요 시간 등을 포함해서 자신의 주변 상태를 네트워크 내의 모든 라우터에게 브로드캐스팅 하는 것

## Distance-Vector (DV) 라우팅 알고리즘

- **분산 (distributed)**
    - 각 노드는 **이웃들**에게서 정보를 받아 계산을 하고 **그 결과**를 이웃들에게 다시 **퍼뜨린다.**
- **반복 (iterative)**
    - 이 과정은 이웃간에 정보의 교환이 없어질 때까지 계속된다.
- **비동기 (asynchronous)**
    - 모든 노드가 같은 시점에 작동할 필요는 없다.
- **자동 종료 (self-terminating)**
    - 계산을 종료하기 위한 시그널이 필요 없다.

### 벨만 포드 방정식 (Bellman-Ford equation)

$min_v$는 x의 모든 이웃들로부터 값을 받는다.

x에서 y로의 최소 비용은 모든 이웃 v로부터의 c(x, v) + $d_v(y)$의 최솟값이다.

노드 u는 이웃이 v, x, w 세 개 있다. 이 노드를 살펴보면 $d_v(z)$ = 5, $d_x(z)$ = 3, $d_w(z)$ = 3 라는 것을 알 수 있다. 또한 c(u, v) = 2, c(u, x) = 1, c(u, w) = 5 이다. 위의 벨만 포드 방정식을 적용해보면 $d_u(z)$ = min {2+5, 5+3, 1+3} = 4라는 것이 확인된다.

- 방정식의 의의
    - 노드 x의 포워딩 테이블의 엔트리를 채울 수 있다.
    - DV 알고리즘에서의 이웃간 통신의 방법을 제안한다.

### 기본 개념

각 노드 x는 $D_x(y)$(x에서 N에 속하는 y까지의 최소 비용 경로의 예상 비용)에서 시작한다. $D_x = [D_x(y): y\in N]$를 노드 x의 거리 벡터(distance vector)라고 하자. 이 벡터는 x에서 다른 모든 (N에 있는)노드 y로의 예상되는 비용이다. DV 알고리즘은 각 노드 x에 대해 다음의 라우팅 정보를 관리한다.

- $c(x,v)$
    - x에서 모든 이웃 노드 v로의 비용
- $D_x = [D_x(y): y\in N]$
    - 노드 x의 거리 벡터
    - x에서 N에 속하는 y로의 예상 비용을 포함한다.
- $D_v = [D_v(y): y\in N]$
    - x의 이웃 노드인 v의 거리 벡터

이러한 분산되고 비동기적인 알고리즘에서 각 노드는 시간마다 **거리 벡터의 사본**을 이웃 노드에게 보낸다. 노드 x가 이웃 노드 w로부터 거리 벡터를 받았다면, 해당 벡터를 저장하고 벨만 포드 방정식을 사용해 자신의 벡터를 다음과 같이 업데이트한다.

- N에 속하는 각각의 노드 y에 대해, $D_x(y)$ = $min_v${c(x,v) + $D_v(y)$}

이렇게 해서 노드 x의 거리 벡터가 변경되면, 노드 x는 해당 벡터를 이웃들에게 보낸다. 이웃들은 이에 따라 자신들의 벡터를 수정한다. 모든 노드가 이러한 교환과정을 계속한다면, 비용의 예상치였던 $D_x(y)$는 최소 비용 경로의 실제 비용인 $d_x(y)$가 된다.

### DV 알고리즘

각 노드 x에 대해 아래 과정을 따른다.

```c
Initialization:
	for all destinations y in N:
		Dx(y)= c(x,y) /* if y is not a neighbor then c(x,y)= ∞ */
	for each neighbor w
		Dw(y) = ? for all destinations y in N
	for each neighbor w
		send distance vector Dx = [Dx(y): y in N] to w

loop
	wait (until I see a link cost change to some neighbor w or
				until I receive a distance vector from some neighbor w)

	for each y in N:
		Dx(y) = minv{c(x,v) + Dv(y)}

if Dx(y) changed for any destination y
			send distance vector Dx = [Dx(y): y in N] to all neighbors

forever
```

- 거리 벡터가 업데이트되는 경우
    - 이웃으로의 비용이 변한 것을 확인할 때
    - 이웃으로부터 거리 벡터의 업데이트를 받았을 때

목적지 y에 대해 자체 포워딩 테이블을 업데이트하려면 y까지의 최단 경로가 아닌, 최단 경로 위에 있는 이웃 노드 $v^*(y)$(다음 홉 라우터)를 알아야 한다.

중앙 집중형이었던 LS 알고리즘의 다익스트라 알고리즘의 경우, 실행 전에 네트워크의 완전한 맵을 모든 노드가 갖고있어야 했다.

반면 DV 알고리즘은 분산형 알고리즘이기 때문에 이러한 전체적인 정보를 사용하지 않는다.

- DV 알고리즘이 필요로 하는 정보
    - 이웃들로의 링크의 비용
    - 해당 이웃들로부터 받은 정보

인터넷의 RIP와 BGP, ISO IDRP, Novell IPX, 원조 ARPAnet 등에서 사용된다.

![Untitled](/assets/img/cs/network/network05/Untitled%207.png)

위 그림은 노드가 세 개인 단순한 네트워크에서의 **DV 알고리즘의 동작**을 보여준다.

제일 왼쪽의 표는 각 노드들의 초기 **라우팅 테이블**이다. 표의 각 줄은 거리 벡터이다.

따라서 노드 x의 초기 라우팅 테이블의 첫 줄은 다음 의미를 갖는다.

$D_x$ = [$D_x(x)$, $D_x(y)$, $D_x(z)$] = [0, 2, 7]

- 두 번째와 세 번째 줄은 노드 y, z로부터 받은 최근 거리 벡터인데, 초기 단계에서 x는 받은게 없기 때문에 무한대로 초기화된다.
- 각 노드는 거리 벡터를 이웃들에게 보낸다.
    - 1열(column)의 표에서 2열의 표로의 화살표를 통해 알 수 있다.
    - 노드 x가 자신의 거리 벡터 $D_x$ = [0, 2, 7]을 y와 z에게 보낸다.
- 해당 업데이트를 받으면, 각 노드는 자신의 자체 거리 벡터를 다시 계산한다.
    - 노드 x는 아래의 계산 과정을 통해 [0, 2, 7]에서 [0, 2, 3]이 되었다.
        - $D_x(x)$ = 0
        - $D_x(y)$ = min{c(x,y) + $D_y(y)$, c(x,z) + $D_z(y)$} = min{2 + 0, 7 + 1} = 2
        - $D_x(z)$ = min{c(x,y) + $D_y(z)$, c(x,z) + $D_z(z)$} = min{2 + 1, 7 + 0} = 3
    - 이렇게 계산한 결과가 2열의 표의 값에 반영된다.
    - 따라서 노드 x에서는 $v^*(y) = y$이고 $v^*(z) = y$이 된다.
- 노드의 재계산과정 이후에, 이들은 업데이트된 거리 벡터를 이웃들에게 다시 전송한다.
    - 2열의 표에서 3열의 표로의 화살표를 통해 알 수 있다.
    - 노드 x와 z만 업데이트를 전송하는 이유는 y의 **거리 벡터에는 변화가 없기 때문**이다.
    - 이 업데이트를 받고 노드들은 자신들의 거리 벡터를 재계산하여 라우팅 테이블을 업데이트한다.
- 이러한 과정은 업데이트 메시지가 전송되지 않을때까지 계속 이어진다.
    - 이 과정이 끝나면 알고리즘은 **조용한(quiescent) 상태**에 들어간다.
    - 모든 노드는 알고리즘 중간의 loop에서 wait한다.
    - **링크 비용(link cost)이 변할 때**까지 조용한 상태가 유지된다.

### DV 알고리즘: 링크 비용이 변하고 링크가 실패한 경우

이러한 노드가 자신에서 이웃으로의 **링크 비용이 변경된 것을 감지**하면, 거리 벡터를 **업데이트**하고, **최소 비용 경로에 변경**이 있다면, **이웃들에게 새 벡터**를 알려준다.

![Untitled](/assets/img/cs/network/network05/Untitled%208.png)

- **링크 비용이 감소한 경우 (위 그림의 a)**
    - 위 그림의 a는 y에서 x로의 링크 비용이 4에서 1로 변경된 상황이다. DV 알고리즘에 의해 다음 과정이 발생한다.
    - 시간이 $t_0$일 때, y가 링크 비용이 변경됨을 감지하고, 거리 벡터를 업데이트하여, 이웃들에게 알려준다.
    - 시간이 $t_1$일 때, z가 y로부터 업데이트를 받아 자신의 표를 수정한다.
        - 자신의 x로의 새로운 최소 비용을 계산하고, 새로운 거리 벡터를 이웃들에게 전송한다.
    - 시간이 $t_2$일 때, y가 z’의 업데이트를 받고 자신의 거리 테이블을 업데이트한다. y의 최소 비용은 바뀌지 않았으므로 z에게 어떠한 메시지도 보내지 않는다.
        - 알고리즘은 조용한(queiescent) 상태가 된다.
    - 이처럼 조용한 상태까지 단 두 번의 반복만이 필요했다.
- **링크 비용이 증가한 경우 (위 그림의 b)**
    - 시간이 $t_0$일 때, y가 링크 비용이 변경됨을 감지하고, x로의 새로운 최소 비용 경로를 아래와 같이 계산하여 $D_y(x)$를 얻는다.
        - $D_y(x)$ = min{c(y,x) + $D_x(x)$, c(y,z) + $D_z(x)$} = min{60 + 0, 1 + 5} = 6
        - 그림을 전체적으로 보면, z를 통한 경로의 비용은 잘못됐다는 것을 알 수 있다. 하지만 y는 x로 바로 가는 비용은 60이고 z는 x까지 5가 든다고 알고있다.(z가 제일 최근에 알려준 내용)
        - 이렇게 되면 **라우팅 루프(routing loop)**에 들어간다.
            - y에서 x로 가기 위해서, y는 z로 라우팅하고, z는 다시 y로 라우팅하는 과정을 반복한다.
        - 노드 y가 z로의 새로운 최소 비용을 계산했으므로, z에게 $t_1$에 새로운 거리 벡터를 알려준다.
    - $t_1$이후 특정 시점에, z는 y의 새 거리 벡터를 받는다.
        - 해당 벡터는 y의 x로의 최소 비용이 6이라는 것을 알려준다.
        - z는 y까지 1이 든다는 것을 알기 때문에 $D_z(x)$ = min{50 + 0, 1 + 6} = 7이 된다.
        - z의 x로의 최소 비용이 증가했으므로, y에게 $t_2$에 새 거리 벡터를 알려준다.
    - 계속 반복
        - 위 과정은 44번이나 반복된다. (이때서야 비로소 z의 y를 거쳐가는 비용이 50을 넘게된다.)
    - 이러한 문제를 **count-to-infinity problem**이라고 한다.

### DV 알고리즘 : poisoned reverse의 추가

위에서 설명한 루프 상황을 피하는 방법은 아래와 같은 **poisoned reverse** 방법을 사용하는 것이다.

- 만약 z가 y를 거쳐서 목적지인 x로 라우팅한다고 하자.
- z는 y에게 $D_z(x)$ = ∞ 라고 알려줄 것이다.
    - $D_z(x)$ = 5 라는 것을 알더라도 말이다.
    - 이러한 거짓말은 z가 x로 가기 위해 y를 거치는 동안은 계속된다.
    - y는 z에게 x로 가는 경로가 없다고 믿기 때문에, x로 라우팅하기 위해 z를 거치지 않을 것이다.

### LS 알고리즘 vs DV 알고리즘

두 알고리즘은 컴퓨팅 라우팅에 대해 보완적인 접근법을 취한다.

DV 알고리즘에서 각 노드는 자신의 인접한 이웃과만 이야기한다. 하지만 이웃에게 자신부터 모든 노드에 대한 예상 최소 비용을 제공한다.

LS 알고리즘은 전체적인 정보를 필요로 한다. 이에 따라 각 라우터의 노드는 모든 노드들과 소통해야한다. 하지만 해당 노드는 자신과 인접한 링크에 대한 비용만을 알려준다.

이들 알고리즘의 주요 차이점은 다음과 같다.

- **메시지 복잡성 (message complexity)**
    - LS
        - 각 노드가 각 링크의 비용을 알도록 한다.
        - 이를 위해 $O(|N||E|)$개의 메시지가 보내져야 한다.
        - 또한 링크 비용이 변하면, 새 링크 비용이 모든 노드에게 전해져야 한다.
    - DV
        - 인접한 이웃과 매 반복마다 메시지를 교환하도록 한다.
        - 링크 비용이 변하면, 새 링크 비용으로 인해 이웃 노드의 최소 비용 경로가 변경되는 경우에만 바뀐 링크 비용의 결과를 전파한다.
- **수렴 속도 (speed of convergence)**
    - LS
        - $O(|N||E|)$개의 메시지가 필요한 $O(|N|^2)$ 알고리즘이다.
    - DV
        - 수렴하는 속도가 느리고, 그 과정에서 라우팅 루프도 발생할 수 있다.
        - 또한 count-to-infinity 문제도 갖고있다.
- **강건성 (robustness)**
    - 라우터가 고장나거나 잘못 작동하거나 방해를 받으면 어떻게 될까?
    - LS
        - 인접한 링크에 잘못된 비용을 전파할 수 있다.
        - LS의 브로드캐스팅 과정에서 노드 또한 손상되거나(corrupt) 패킷을 버릴 수 있다.
        - 하지만 각자 포워딩 테이블을 따로 갖고있다는 점으로 인해, 어느정도의 robustness를 확보할 수 있다. (계산이 독립적으로 이루어지기 때문)
    - DV
        - 노드는 몇몇 목적지에 대한 최소 비용 경로를 잘못 전파할 수 있다.
        - 이렇게 되면 반복을 거듭해서 네트워크 전체에 잘못된 계산이 퍼질 것이다.
    
    어떤 알고리즘도 승자나 패자가 아니다. 실제로는 두 개 모두 인터넷에서 사용되고 있다.
    

# Intra-AS Routing in the Internet: OSPF

라우팅 프로토콜은 라우팅의 범위에 따라 **intra**와 **inter**로 나뉜다. (intra : 내부, inter : 상호간)

- **Inter-AS** 라우팅 프로토콜
    - ex. OSPF
- **intra-AS** 라우팅 프로토콜
    - ex. BGP

라우팅 알고리즘을 공부하면서, 네트워크를 단순히 **상호 연결된 라우터들의 집합**으로 봤었다. 모든 라우터가 동일한 라우팅 알고리즘을 실행해 경로를 계산하기 때문에 각각을 구분할 수 없었다. 이와같이 모델을 단순화한 이유는 다음과 같다.

- **규모 (Scale)**
    - 라우터의 수가 늘어남에 따라, 통신, 계산, 라우팅 정보 저장에 필요한 **오버헤드(overhead)는 막대**해졌다.
- **관리의 자율성 (Administrative autonomy)**
    - 인터넷은 ISP들의 네트워크이고, 각각의 ISP는 자신의 라우터들의 네트워크로 이루어져있다. ISP는 **자신의 네트워크의 내부를 외부로부터 숨기고** 싶어하고, **원하는대로 운영**하고 싶어한다.
    - 한 조직은 네트워크를 외부와 연결하면서도 운영과 관리는 자율적으로 할 수 있어야 한다.

위의 두 문제는 라우터를 **자율 시스템(autonomous system, AS)**들로 운영하여 해결할 수 있다. 각 AS는 동일한 관리를 받는 라우터들의 그룹으로 구성되어있다.

같은 AS에 있는 라우터들은 모두 동일한 라우팅 알고리즘을 실행하고 각자에 대한 정보를 갖는다. 이렇게 자율 시스템에서 동작하는 라우팅 알고리즘을 **intra-AS 라우팅 프로토콜**이라고 부른다.

### Open Shortest Path First (OSPF)

OSPF 라우팅과 IS-IS는 intra-AS 라우팅에 널리 쓰이고 있다.

OSPF는 링크 상태 정보(link state information)의 **플러딩**과 **다익스트라** 최소 비용 알고리즘을 사용한다.

- **플러딩(flooding)**
    - 수신한 패킷을 라우터에 접속되어 있는 **다른 모든 노드로 전달**하는 정적 알고리즘
    - 패킷의 전송이 보장된다.
    - 네트워크 내의 모든 경로를 사용하기 때문에 **최단 경로를 찾아서 활용**하는 것이 보장된다.

각 라우터는 전체 AS에 대한 완전한 토폴로지를 만든다. 각 라우터는 이후에 각자 다익스트라 알고리즘을 실행해서 모든 **서브넷**까지의 최단 경로를 구한다. 개개의 링크 비용은 네트워크 관리자에 의해 설정된다.

라우터는 주기적으로 링크의 상태를 브로드캐스트하고 상태의 변화가 있을 때마다 추가적으로 브로드캐스트한다.

- **광고(advertisement)** : 라우터, 무선노드 등의 장비에서, 일정 주기 또는 필요 시에 자신과 주변 정보를 알리는 과정

OSPF 광고는 IP가 운반하는 OSPF 메시지 안에 들어가 있으며 89번 프로토콜을 사용한다.

OSPF 프로토콜은 신뢰할 수 있는 메시지 전송과 링크 상태 브로드캐스트와 같은 기능을 구현해야 한다.

OSPF에는 다음과 같은 특징이 있다.

- **보안 (Security)**
    - 라우터 간의 정보 교환은 인증 과정을 거칠 수 있다.
        - 이를 통해 악의적인 침입자가 라우터 테이블에 잘못된 정보를 추가하는것을 막는다.
    - 인증에는 두 가지 종류가 있다.
        - simple : 모든 라우터에 동일한 비밀번호가 설정된다.
        - MD5 : 모든 라우터가 공유하는 비밀 키(secret key)가 있다. 리플레이 공격을 막기 위해 시퀀스 번호를 사용하기도 한다.
- **여러 동일 비용 경로 (Multiple same-cost paths)**
    - 목적지에 대해 여러 경로가 동일한 비용을 가지기 때문에, 하나의 경로를 선택할 필요가 없다.
- **유니캐스트와 멀티캐스트 라우팅에 대한 통합 지원 (Integrated support for unicast and multicast routing)**
    - **유니캐스트(unicast)**
        - **하나의 목적지에 1:1로** 트래픽 또는 메시지를 전송하는 방식
        
        ![Untitled](/assets/img/cs/network/network05/Untitled%209.png)
        
    - **멀티캐스트(multicast)**
        - 한 번의 송신으로 **목표한 여러 컴퓨터에 메시지를 동시에** 전송하는 방식
        
        ![Untitled](/assets/img/cs/network/network05/Untitled%2010.png)
        
    - **브로드캐스트(broadcast)**
        - 송신 호스트가 데이터를 **네트워크에 연결된 모든 호스트**에게 전송하는 방식
        - 네트워크의 **성능을 악화**시킬 수 있기 때문에, LAN과 같은 **소규모** 네트워크에서는 쓰이지만 WAN과 같은 대규모 네트워크에서는 사용되지 않는다. IPv6에서도 지원하지 않는다.
        - **Dos (Denial of Service) 공격**에 사용될 수 있는 우려가 있다.
            - 예를 들어, 공격자가 가짜 핑 요청(fake ping request)들을 출발지를 희생자 컴퓨터의 주소로 설정하여 노드들에게 보냈다고 하자.
            - 모든 노드들이 일제히 희생자 컴퓨터의 요청에 응답을 해서 고장을 일으킨다.
        
        ![Untitled](/assets/img/cs/network/network05/Untitled%2011.png)
        
    - **브로드캐스팅 vs 플러딩**
        - 브로드캐스팅
            - 모든 호스트에게 **동시에** 패킷을 보낸다.
            - 링크를 따라 패킷을 한 번만 전송한다.
            - 플러딩과 달리 패킷에 특수 브로드캐스트 주소를 지정하여 전송한다.
        - 플러딩
            - 모든 호스트에게 동시에 패킷을 보내지 않는다.
            - 동일한 패킷을 동일한 링크를 통해 여러 번 전송할 수 있다.
- **단일 AS 내에서 계층을 지원 (Support for hierarchy within a single AS)**
    - OSPF 자율 시스템(AS)은 여러 영역들로 계층적으로 구성될 수 있다.
    - 각 영역은 자신의 OSPF 링크 상태 라우팅 알고리즘을 실행한다.
        - 영역 내의 한 라우터는 다른 라우터들에게 자신의 링크 상태를 브로드캐스팅한다.
    - 각 영역의 **영역 경계 라우터(area border router)**는 바깥 영역으로 패킷을 라우팅한다.
    - AS 영역 중 하나는 백본 영역(backbone area)으로 설정된다.
        - 백본 영역 : 다른 영역들간의 트래픽을 라우팅하는 역할

# Routing Among the ISPs: BGP

위에서 살펴본 OSPF는 intra-AS 라우팅 프로토콜로 동일한 AS  내의 출발지와 목적지 사이를 패킷이 오간다.

하지만 패킷이 **여러 AS 사이**를 오가기 위해서는 inter-AS 라우팅 프로토콜이 필요하다. 이 과정을 위해 모든 AS가 동일한 inter-AS 라우팅 프로토콜을 실행한다. 인터넷에서 사용하는 동일한 inter-AS 라우팅 프로토콜은 **Border Gateway Protocol(BGP)**이다.

BGP는 **분산형**의 **비동기** 프로토콜이다.

## BGP의 역할

한 패킷의 목적지가 동일한 AS 내에 있을 경우 → 라우터의 포워딩 테이블의 엔트리는 intra-AS 라우팅 프로토콜에 의해 결정된다.

**목적지가 AS 밖에** 있는 경우? → BGP

BGP에서  패킷은 특정한 목적지 주소로 라우팅되지 않는다. 대신 CIDR된 prefix로 라우팅된다. 각 prefix는 서브넷이나 그 집합을 의미한다.

ex. 목적지가 138.16.68/22와 같은 형태로 설정된다. 해당 prefix는 1024개의 IP 주소를 포함한다.

포워딩 테이블은 (x, I)의 형태이다. (x : prefix, I : 각 라우터 인터페이스의 번호)

BGP는 각 라우터들에게 아래의 사항들이 가능하도록 해준다.

- 인접한 AS로부터 **prefix 도달 가능성(reachability)** 정보를 얻는다.
    - BGP는 각 서브넷이 나머지 인터넷에 자신의 존재를 광고하도록 해준다.
- prefix들로의 **최고의 경로**를 결정한다.
    - 라우터는 특정 prefix로의 경로를 하나 이상 알 수 있다.
    - 라우터는 이 중 최고의 경로를 찾기위해, BGP 경로 선택 과정을 각자 수행한다.
        - 이 과정에서 BGP로부터 받은 prefix 도달 가능성 정보를 사용한다.

## BGP 경로 정보의 광고 (Advertising BGP Route Information)

![Untitled](/assets/img/cs/network/network05/Untitled%2012.png)

위의 그림을 보면 자율 시스템(autonomous system)이 세 개가 존재한다는 것을 알 수 있다. 왼쪽부터 AS1, AS2, AS3.

각 라우터는 **게이트웨이 라우터**이거나 **내부 라우터**이다.

- **게이트웨이 라우터(gateway router)** : AS의 끝자락에 위치하며 다른 AS의 라우터들과 직접 연결되는 라우터
- **내부 라우터(internal router)** : 자기 자신의 AS 내에 있는 호스트와 라우터와만 연결되는 라우터

예를 들어, prefix x에 대한 도달 가능성 정보를 광고하는 과정은 다음과 같다.

1. AS3이 AS2에게 다음과 같은 BGP 메시지를 전송한다.
    - `"AS3 x"`
    - x가 AS3에 존재한다는 의미
2. AS2가 AS1에게 BGP 다음과 같은 메시지를 보낸다.
    - `"AS2 AS3 x"`
    - AS2를 거쳐 AS3로 가면 x에 도달할 수 있다는 의미

실제로는 AS가 아니라 라우터가 메시지를 주고받는다.

라우터쌍은 179번 포트를 사용하는 반 영구적인 TCP 연결을 통해 라우팅 정보를 교환한다. 이러한 연결은 **BGP 연결**이라고 불린다.

BGP 연결의 종류는 다음과 같다. (그림 5.9 참고)

![Untitled](/assets/img/cs/network/network05/Untitled%2013.png)

- 외부 BGP (external BGP, eBGP)
    - 두 개의 AS를 걸쳐서 있는 연결
- 내부 BGP (internal BGP, iBGP)
    - 같은 AS 내의 라우터간의 연결

## 최고 경로의 결정 (Determining the Best Routes)

라우터가 BGP 연결을 통해 prefix를 광고할 때, 여러 **BGP 속성**들도 함께 보낸다. 이 속성들을 **route**라고 부른다. 중요한 속성들로는 AS-PATH와 NEXT-HOP이 있다.

- AS-PATH
    - 광고가 거쳐간 AS들의 리스트
- NEXT-HOP
    - **AS-PATH의 시작** 부분의 라우터 인터페이스의 IP 주소
    - inter-AS와 intra-AS 라우팅 프로토콜 사이에 중요한 링크를 제공한다.

![Untitled](/assets/img/cs/network/network05/Untitled%2014.png)

위 그림에서 AS1에서 x까지의 경로가 `"AS2 AS3 x"` 에 대한 **NEXT-HOP** 속성은 **라우터 2a의 왼쪽 인터페이스의 IP 주소**이다. 반면에 AS1에서 x까지의 경로가 `"AS3 x"` 라면 **NEXT-HOP** 속성은 **라우터 3d의 왼쪽 인터페이스의 IP 주소**일 것이다.

이로 인해 AS1의 각 라우터는 prefix x로의 BGP 경로 두 개를 알게 된다.

- 라우터 2a의 왼쪽 인터페이스의 IP 주소; AS2 AS3; x
- 라우터 3d의 왼쪽 인터페이스의 IP 주소; AS3; x

위와 같이, 각 **BGP 경로**는 `NEXT-HOP; AS-PATH; 목적지 prefix` 와 같은 형태를 갖는다.

### 뜨거운 감자 라우팅 (Hot Potato Routing)

가장 단순한 라우팅 알고리즘 중 하나이다.

여러 경로 중에 **NEXT-HOP 라우터까지의 비용이 최소**인 경로를 고른다.

라우터 1b가 prefix x까지의 경로 두 개를 받은 상황을 가정해보자.

1. intra-AS 라우팅 정보를 통해 NEXT-HOP 라우터 2a와 3d 각각까지의 비용을 계산해본다.
2. 비용이 거치는 링크의 개수라고 한다면 2a까지는 2, 3d까지는 3이기 때문에 라우터 2a가 선택될 것이다.
3. 라우터 1b는 포워딩 테이블에서 라우터 2a로의 최소 비용 경로 상에 있는 인터페이스 I를 찾는다.
4. 그 후에 (x, I)를 테이블에 추가한다.

![Untitled](/assets/img/cs/network/network05/Untitled%2015.png)

[외부-AS 목적지를 포워딩 테이블에 추가하는 과정]

위 그림에서 볼 수 있듯이 이를 위해 inter-AS(BGP)와 intra-AS(OSPF) 라우팅 프로토콜 모두 필요하다.

**자신의 AS에서의 비용을 최소화하는 데에만 집중**할 뿐, 끝에서 끝(end-to-end)까지의 비용은 무시하는 이기적인 알고리즘이다.

### 경로 선택 알고리즘 (Route-Selection Algorithm)

실제로는 뜨거운 감자 라우팅만 사용하지 않고 더 복잡한 알고리즘에 이걸 혼합해서 사용한다.

목적지 prefix까지의 경로가 **한 개** → 해당 경로를 택한다.

목적지 prefix까지의 경로가 **여러 개** → 하나가 남을 때까지 아래의 **제거 규칙(elimination rule)**들을 순서대로 호출한다.

1. **로컬 선호도(local preference)** 값이 가장 높은 경로가 선택된다.
    - 경로에는 AS-PATH, NEXT-HOP 속성 외에도 로컬 선호도 값이 할당된다.
    - 해당 라우터나 AS 내의 다른 라우터에 의해 값이 설정된다.
2. 남아있는 경로(로컬 선호도가 동일)에 대해, **AS-PATH가 제일 짧은 경로**가 선택된다.
3. 남아있는 경로(로컬 선호도, AS-PATH가 동일)에 대해, **뜨거운 감자 라우팅**이 사용된다.
    - NEXT-HOP 라우터까지의 거리가 제일 가까운 경로가 선택된다.
4. 아직도 두 개 이상의 경로가 남았다면, **BGP 식별자(identifier)**를 사용해서 경로를 선택한다.

## IP-애니캐스트 (IP-Anycast)

- **애니캐스트(anycast)**
    - 송신자가 데이터그램을 **가장 가까운 노드**로 연결시키는 네트워크 어드레싱 및 라우팅 방식
    - 임의 캐스트라고도 부른다.
    
    ![Untitled](/assets/img/cs/network/network05/Untitled%2016.png)
    

BGP는 IP-애니캐스트 서비스를 구현하는데에 자주 사용된다.

애플리케이션에 있어서 **주요 관심사**는 다음과 같다.

1. 여러 **분산된 지역**에 있는 다른 서버에서 같은 **컨텐츠를 복제**하는 것
2. 각 사용자가 **가까운 서버**에서 컨텐츠에 **액세스**하는 것

그림 5.12는 CDN이 IP-애니캐스트를 사용하는 과정을 나타낸 것이다.

1. 우선 CDN 회사는 자신의 서버들에게 모두 동일한 IP 주소를 할당한다.
    - 그 후 각 서버가 표준 BGP를 사용해 IP 주소를 광고한다.
2. BGP 라우터가 해당 IP 주소에 대해 여러 경로 광고를 받게 된다.
    - 해당 라우터는 **동일한 물리적 위치로의 경로를 여러 개 수신**한 것으로 여긴다. (실제로는 서로 다른 물리적 위치이다.)
3. 라우팅 테이블을 설정할 때, 각 라우터는 각자 BGP 경로 선택 알고리즘을 사용해서 해당 IP 주소로의 최고의 경로를 선택한다.

# ICMP: The Internet Control Message Protocol

ICMP (Internet Control Message Protocol)는 호스트와 라우터간에 네트워크 계층 정보를 주고받는데 사용된다. 주된 사용은 오류 보고(error reporting)이다.

ICMP 메시지가 IP 데이터그램 안에 들어가는 형태이다. TCP나 UDP 세그먼트와 같은 방식으로 IP payload에 위치한다.
