다중 프로그래밍 환경에서는 여러 스레드들이 한정된 수의 자원(resource)을 두고 경쟁할 수 있다. 스레드가 자원을 요청할 때, 자원이 사용가능하지 않다면 스레드는 대기 상태에 돌입한다. 가끔씩 대기하는 스레드가 요청한 자원을 **다른 대기하는 스레드가 소유**하고 있는 경우, **대기 상태에서 영원히 벗어나지 못할 수 있다.** 이 상황을 데드락(deadlock, 교착상태)이라고 부른다. **데드락**은 프로세스 집합의 모든 **프로세스**가 집합의 **다른 프로세스에서만 발생할 수 있는 이벤트**를 기다리는 상황을 말한다.

이 장에서는 애플리케이션 개발자와 운영 체제 프로그래머가 **교착 상태를 방지**하거나 **처리**하기 위해 사용할 수 있는 방법에 대해 설명한다.

## 챕터 목표

- 뮤텍스 락을 사용할 때 교착 상태가 어떻게 발생할 수 있는지 설명한다.
- 교착 상태를 결정하는 네 가지 필요 조건을 정의한다.
- 리소스 할당 그래프(resource allocation graph)에서 교착 상태를 식별한다.
- 교착 상태를 방지하기 위한 네 가지 접근 방식을 평가한다.
- 교착 상태 회피를 위해 뱅커 알고리즘을 적용한다.
- 교착 상태 감지 알고리즘을 적용한다.
- 교착 상태를 복구하기 위한 접근 방식을 평가한다.

# System Model

시스템은 다수의 경쟁하는 스레드들에 분배될 한정된 자원들로 이루어져 있다. 리소스는 여러 유형(또는 클래스)으로 분할될 수 있으며, 각각은 몇 개의 동일한 인스턴스로 구성된다.

- 리소스 타입 : CPU 주기, I/O 장치(네트워크 인터페이스, DVD 드라이브) 등

6장에서 봤던 뮤텍스 락, 세마포어와 같은 동기화 도구들도 시스템 리스소이다.

잠금장치(lock)는 일반적으로 특정 자료 구조와 관련된다. —한 잠금은 큐에 대한 접근을 막기 위해 쓰이고, 다른 잠금은 연결 리스트에 대한 접근을 막기 위해 쓰인다. 이러한 이유에서 각 잠금 인스턴스에는 일반적으로 자체 리소스 클래스가 할당된다.

스레드는 자원을 사용하기 전에 요청해야 하고, 다 사용하면 해제해야한다. 스레드는 지정된 작업을 수행하는데 필요한 자원의 양만큼 리소스를 요청할 수 있다. 물론 시스템에서 사용할 수 있는 리소스의 양보다는 적게 요청한다.

일반적으로 스레드는 자원을 다음의 순서로 사용한다.

1. **요청(Request)** : 스레드가 자원을 요청한다. 만약 뮤텍스 락이 다른 스레드에 걸려있는 등의 이유로 요청이 바로 처리될 수 없다면 요청한 스레드는 자원이 사용가능해질 때까지 대기한다.
2. **사용(Use)** : 스레드가 자원을 사용한다. (뮤텍스 락이 있는 자원의 경우 스레드는 임계구역에 들어간다.)
3. **해제(Release)** : 스레드가 자원을 해제한다.

# Deadlock in Multithreaded Applications

우선 Pthread 프로그램에서 데드락이 발생하는 과정을 살펴보자. `pthread_mutex_init()` 함수가 잠금해제된 뮤텍스를 초기화한다. 뮤텍스 락은 `pthread_mutex_lock()` 과 `pthread_mutex_unlock()` 을 사용해서 잠기고 해제된다. 스레드가 잠긴 뮤텍스를 획득하려고 시도하면 뮤텍스 락의 주인이 `pthread_mutex_unlock()` 을 호출할 때까지 호출이 차단된다.

두 개의 뮤텍스 락은 아래와 같이 생성되고 초기화된다.

```c
pthread_mutex_t first_mutex;
pthread_mutex_t second_mutex;
pthread_mutex_init(&first_mutex,NULL);
pthread_mutex_init(&second_mutex,NULL);
```

스레드 두 개(`thread_one`, `thread_two`)가 생성되고 둘 다 뮤텍스 락에 접근한다. 각각 아래와 같이 `do_work_one`, `do_work_two` 함수를 실행한다. 여기서 `thread_one`은 `first_mutex`, `second_mutex`의 순서로 뮤텍스 락을 획득하려고 시도하고, 같은 시간에 `thread_two`는 `second_mutex`, `first_mutex`의 순서로 획득하려고 시도한다. 만약 `thread_one`이 `first_mutex`를 획득하는 동안 `thread_two`가 `second_mutex`를 획득한다면 데드락이 발생할 수 있다.

```c
/* thread one runs in this function */
void *do_work_one(void *param)
{
	pthread_mutex_lock(&first_mutex);
	pthread_mutex_lock(&second_mutex);
	/**
	* Do some work
	*/
	pthread_mutex_unlock(&second_mutex);
	pthread_mutex_unlock(&first_mutex);

	pthread exit(0);
}

/* thread two runs in this function */
void *do_work_two(void *param)
{
	pthread_mutex_lock(&second_mutex);
	pthread_mutex_lock(&first_mutex);
	/**
	* Do some work
	*/
	pthread_mutex_unlock(&first_mutex);
	pthread_mutex_unlock(&second_mutex);

	pthread exit(0);
}
```

[데드락의 예시]

하지만 `thread_two`가 락을 획득하려고 시도하기 전에 `thread_one`이 `first_mutex`와 `second_mutex`를 획득하고 해제한다면 데드락이 발생하지 않는다. 이런 식의 스레드의 동작 순서는 CPU에 의해 스케줄된다.

위의 예시에서 봤듯이 데드락 처리에는 문제가 있는데, 특정한 스케줄링 상황에서만 발생하는 데드락을 식별하고 테스트하기가 어렵다는 점이다.

# Deadlock Characterization

## 데드락의 필요 조건

데드락이 발생하기 위해서는 다음 **네 개의 조건**이 **모두 충족**되어야 한다.

1. 상호 배제 (mutual exclusion)
    - 최소 하나의 자원은 공유 불가능 모드로 있어야 한다. 즉, 한 번에 하나의 스레드만 자원을 사용할 수 있다. 다른 스레드가 해당 자원을 요청하면 리소스가 해제될 때까지 요청한 스레드가 지연돼야 한다.
2. 점유 대기 (hold and wait)
    - 스레드가 최소 한 개의 자원을 들고 있고 다른 스레드가 가지고 있는 자원을 추가적으로 얻기위해 기다려야 한다.
3. 비선점 (no preemption)
    - 스레드가 자신의 작업을 마무리 한 다음에야 해당 자원을 사용할 수 있다.
4. 순환 대기 (circular wait)
    - 스레드의 집합이 있을 때(T0, T1, … Tn), T0은 T1이 갖고있는 자원을 기다리고, T1은 T2가 갖고있는 자원을, …, Tn-1은 Tn이 가진 자원을, Tn은 T0이 가진 자원을 기다린다.

## 자원 할당 그래프

데드락은 다음과 같은 **시스템 자원 할당 그래프(system resource-allocation graph)**라는 이름의 유향 그래프로 정확하게 표현할 수 있다. 이 그래프는 꼭짓점(vertices) V의 집합과 엣지(edge) E의 집합으로 이루어져 있다. V는 두 가지의 노드로 나뉘는데, T = {T1, T2, …, Tn}는 시스템 내의 활성화 된 모든 **스레드**의 집합이고, R = {R1, R2, …, Rn}은 시스템에 있는 모든 **자원**의 집합이다.

- **그래프의 구성** : 그래프는 꼭짓점(vertex), 교점(node), 점(point)으로 구성되며 이것들은 변(edge)로 연결된다.

# Methods for Handling Deadlocks

데드락 문제를 처리하는 데에는 세 가지 방법이 있다.

1. 다 같이 문제를 무시하고 데드락이 시스템에 발생하지 않는 것처럼 모른체한다.
2. 데드락 상태에 절대 빠지지 않도록 보장하는 프로토콜을 사용한다. → deadlock prevention, deadlock avoidance
3. 시스템이 데드락에 빠지도록 하고, 그것을 탐지하여 복구한다. → deadlock detection, recovery from deadlock

1번 방법은 리눅스, 윈도우를 포함한 다수의 운영체제에서 사용되는 방법으로, 커널과 개발자가 데드락의 처리를 맡게된다. 3번 방법은 데이터베이스 등의 시스템에서 사용되는 방법이다.

# Deadlock Prevention

위에서 필요조건에서 살펴봤듯이, 데드락은 네 개의 조건이 모두 충족되어야만 발생한다. 따라서 데드락을 방지하기 위해서는 최소 한가지 조건이 발생하지 않게끔 해주면 된다.

## Mutual Exclusion

이 조건은 **반드시 충족**된다. 자원을 여러 스레드에게 동시에 접근 가능하도록 허용해야 하지만 대부분의 애플리케이션은  일부 자원이 **본질적으로 공유 불가능**하기 때문 적용할 수 없다. (예를 들어 뮤텍스 락의 경우 여러 스레드에게 공유될 수 없다.)

## Hold and Wait

이 조건이 발생하지 않도록 보장하려면 **한번에 모든 자원을 할당**하거나, 스레드가 자원을 요청할 때 **다른 자원을 들고있지 않도록** 하면 된다. 하지만  대부분의 애플리케이션에서 비현실적이다. 우선 **자원 활용도가 떨어질** 수 있고 **기아가 발생**할 수 있기 때문이다.

## No Preemption

프로토콜이 선점을 보장하도록 해서 사용하면 된다.

이 조건이 발생하지 않도록 하려면 다음과 같이 프로토콜을 사용하면 된다.  스레드가 어떤 자원을 가진 상태에서 다른 바로 할당되지 못하는 자원을 요청했다면, 스레드가 현재 가지고 있는 모든 자원이 해제(선점)된다. 이렇게 선점된 자원들은 스레드가 기다리는 자원 리스트에 포함된다. 이후 스레드는 옛 자원과 새로 요청한 자원을 모두 얻을 수 있을 때 비로소 재시작된다.

![Untitled](/assets/img/cs/os/os08/Untitled%203.png)

이러한 프로토콜은 상태가 쉽게 저장되고 복구될 수 있는 자원에 주로 적용된다. 그 예로는 CPU 레지스터와 데이터 트랜잭션 등이 있다. 반면 뮤텍스 락이나 세마포어와 같이 데드락이 가장 일반적으로 발생하는 자원에는 적용할 수 없다.

## Circular Wait

앞의 세 가지 방법은 대부분의 상황에서 실용적이지 않지만 이 방법은 실용적으로 문제를 해결할 수 있다. 이 조건을 무효화하는 방법은 전체 리소스 타입에 순서를 적용시키고 각 스레드가 높아지는 순서대로 자원을 요청하도록 하는 것이다.

리소스 타입의 집합인 R = {R1, R2, R3, …, Rn}이 있다고 가정하자. 이 **각각의 리소스 타입**에 유일한 **정수를 부여**하여 이후에 자원간에 비교를 할 때 어떤 자원이 **더 높은 순서**인지를 파악할 수 있도록 한다. 이를 위해 일대일 함수인 F: R → N (N은 자연수 집합) 을 정의한다. 이렇게 하면 위의 **Deadlock in Multithreaded Applications**에서 예시로 살펴본 각 뮤텍스 자원은 다음과 같이 순서가 부여된다.

```c
F(first_mutex) = 1
F(second_mutex) = 5
```

만약 맨 처음에 스레드가 Ri라는 자원의 인스턴스를 요청했다고 하면 이후에는 `F(Rj) > F(Ri)`를 만족하는 Ri의 인스턴스만을 요청할 수 있다. `first_mutex`와 `second_mutex` 를 도이세 사용하길 원하는 스레드가 있다면 `first_mutex`를 먼저 요청하고 `second_mutex` 를 요청해야 한다. 또 `F(Ri) ≥ F(Rj)`를 만족하는 Ri의 인스턴스를 요청하는 스레드는 Rj를 해제해야 한다.

하지만 이런 순서를 만드는 것 만으로는 데드락이 방지되지 않는다. 순서에 따라 프로그램을 짜는 것은 개발자의 몫이기 때문이다. 또한 **락이 동적으로 얻어지는 경우**에는 락에 순서를 적용하는 것으로 **데드락의 방지를 보장할 수 없다.**

예를 들어, 아래와 같이 두 계좌 간에 자금을 이체하는 함수를 가정하자. 경쟁 상태를 막기 위해 각 계좌에는 `get_lock()` 함수를 통해 얻어진 뮤텍스 락이 있다. 만약 두 개의 스레드에서 동시에 서로의 계좌에 돈을 옮기도록 `transaction()` 함수를 호출했다면, 한 스레드는 `transaction(checking_account, savings_account, 25.0)` 를 호출할 것이고 다른 스레드는 `transaction(savings_account, checking_account, 50.0)` 를 호출할 것이다.

```c
void transaction(Account from, Account to, double amount)
{
	mutex lock1, lock2;
	lock1 = get_lock(from);
	lock2 = get_lock(to);

	acquire(lock1);
		acquire(lock2);

			withdraw(from, amount);
			deposit(to, amount);

		release(lock2);
	release(lock1);
}
```

# Deadlock Avoidance

위에서 살펴본 교착 방지 알고리즘(Deadlock-prevention algorithms)은 요청 방식을 제한하여 네 가지 조건 중 일부가 발생하지 않도록 해서 교착상태를 방지하였다. 하지만 이런 방식은 **낮은 장치 활용률(device utilization)**과 **시스템 처리량(throughput) 감소**라는 부작용을 갖는다.

이 대신 자원이 어떻게 요청될 지에 대한 추가적인 정보를 요청해성 데드락을 회피할 수 있다. 예를 들어 자원 R1, R2가 있는 시스템에서는 스레드 P가 R1→R2의 순서로 요청하고 스레드 Q가 R2→R1의 순서로 요청할 것이라는 사실을 알고있어야 한다.

이러한 접근법에 사용되는 알고리즘은 저마다 필요로 하는 정보가 다르다. 가장 간단한 알고리즘을 예로 들면 스레드가 필요로 하는 **최대 자원의 양**을 정보로 받는다. 그 후에 이 정보를 토대로 시스템이 절대 데드락에 빠지지 않을 알고리즘을 만든다. 알고리즘은 자원 할당 상태가 절대 순환대기가 발생하지 않는지를 검사한다.

- **자원 할당 상태(resource-allocation state)**의 구성 : 사용 가능한 자원, 할당된 자원, 스레드의 최대 요구량

## 안전 상태(Safe State)

시스템이 자원을 각 스레드에 특정 순서로 할당하면서도 데드락이 발생하지 않는 상태를 의미한다.

안전 상태가 되려면 **안전 순서열(safe sequence)**이 존재해야만 한다.

## 은행원 알고리즘(Banker’s Algorithm)

위의 자원 할당 그래프는 각 자원의 인스턴스가 여러 개라면 적용할 수 없다. 이 때 사용하는 방식이 **은행원 알고리즘(banker’s algorithm)**이다.

스레드가 처음 시스템에 들어오면 자신이 필요로 하는 각 자원 유형 별 인스턴스의 최대량(시스템의 최대 자원을 초과하면 안됨)을 알려줘야 한다. 사용자가 자원을 요청하면 시스템은 **자원의 할당이 시스템을 안전 상태에 있게 하는지**를 고려한다. 안전 상태에 있다면 자원을 할당하고, 아니라면 다른 스레드가 충분히 자원을 해제할 때까지 기다린다.

은행원 알고리즘을 구현하기 위해서는 다음과 같은 자료구조가 필요하다.(n과 m은 각각 스레드와 자원 유형의 수)

- Available : 각 유형별로 가용한 자원의 개수를 나타내는 벡터로 길이가 m이다.
    - ex) `Available[j] = k` 는 현재 Rj의 인스턴스를 k개 사용가능하다는 의미
- Max : 각 스레드가 최대로 필요로 하는 자원의 개수를 나타내는 행렬로 크기가 n x m이다.
    - ex) `Max[i][j] = k` 는 스레드 Ti가 Rj를 최대 k개까지 요청할 수 있다는 의미
- Allocation : 각 스레드에 현재 할당된 자원의 개수를 나타내는 행렬로 크기가 n x m이다.
    - ex) `Allocation[i][j] = k` 는 현재 스레드 Ti가 Rj를 k개 사용 중이라는 의미
- Need : 각 스레드가 향후 요청할 수 있는 자원의 개수를 나타내는 행렬로 크기가 n x m이다.
    - ex) `Need[i][j]` 는 스레드 Ti가 향후 Rj를 k개까지 더 요청할 수 있다는 의미
    - `Need[i][j] = Max[i][j] - Allocation[i][j]` 의 관계를 지님

이 자료 구조는 시간이 지남에 따라 그 크기와 값이 달라진다.

여기에 아래의 표기법을 추가한다. X, Y를 길이가 n인 벡터라고 하자. X ≤ Y는 모든 i (i = 1, 2, …, n)에 대해 X[i] ≤ Y[i] 관계가 있음을 의미한다.

### 안전성 알고리즘(Safety Algorithm)

다음은 시스템이 안전한지를 확인하는 알고리즘이다.

1. 길이가 각각 m, n인 벡터 Work와 Finish를 가정하자. Work = Available로 초기화하고 Finish[i] = false (i = 0, 1, …, n - 1)로 초기화 한다.
2. 아래의 조건을 만족하는 i를 찾는다. i가 존재하지 않으면 4번으로 간다.
    a. `Finish[i] == false`
    b. `Need(i) ≤ Work`
3. 아래 연산을 수행하고 2번으로 간다.
    a. `Work = Work + Allocation(i)`
    b. `Finish[i] = true`
4. 모든 i에 대해 `Finish[i] == true` 를 만족한다면 시스템은 안전 상태에 있다.

### 자원 요청 알고리즘(Resouce-Request Algorithm)

다음은 자원 요청이 안전하게 들어줄 수 있는지를 확인하는 알고리즘이다.

Request(i)는 스레드 Ti의 요청 벡터라고 하자. `Request(i)[j] == k`라면 Ti가 Rj를 k개까지 요청하고 있음을 뜻한다. Ti가 자원을 요청하면 아래의 조치가 취해진다.

1. `Request(i) ≤ Need(i)` 이면 2번으로 간다. 아니라면 최대 요청 개수를 초과했으므로 오류로 처리한다.
2. `Request(i) ≤ Available` 이면 3번으로 간다. 아니라면 자원이 없다는 뜻이므로 기다린다.
3. 다음과 같이 상태를 변경해서 자원을 할당한 것처럼 한다.
    
    ```c
    Available = Available - Request
    Allocation(i) = Allocation(i) + Request(i)
    Need(i) = Need(i) - Request(i)
    ```
    

만약 이렇게 해서 바뀐 상태가 안전하다면 이 정보대로 Ti에 자원을 할당한다. 하지만 새로운 상태가 위험하다면, 자원 할당 상태는 원상태로 복원되고 Ti는 Request(i)가 만족할 때까지 기다린다.

# Deadlock Detection

시스템이 데드락 방지나 데드락 회피 알고리즘을 채용하지 않는다면 데드락이 발생할 수 있으므로 다음의 방식을 제공해야 한다.

- 데드락이 **발생했는지를 인지**하는 알고리즘(**deadlock detection**)
- 데드락으로부터 **복구**하는 알고리즘(**deadlock recovery**)

## 자원 유형 별로 한 개의 인스턴스가 있는 경우

이 경우 리소스 할당 그래프의 변형인 **대기 그래프(wait-for graph)**를 사용해 데드락 감지 알고리즘을 정의할 수 있다. 대기 그래프에서 Ti에서 Tj로 향하는 엣지는 Ti가 Tj가 자신이 필요로 하는 자원을 해제하기를 기다린다는 의미이다. 대기 그래프에서 Ti→Tj 엣지는 자원 할당 그래프 상의 Ti→Rq와 Rq→Tj가 모두 있어야만 존재한다.

## 자원 유형 별로 여러 인스턴스가 있는 경우

이 경우에 사용되는 알고리즘은 **은행원 알고리즘**과 유사하게 **시간 변동(time-varying) 자료 구조**를 사용한다.

- Available : 각 종류의 자원이 현재 몇 개가 가용한지를 나타내는 벡터. 길이가 m.
- Allocation : 각 스레드에 현재 할당된 자원의 개수를 나타내는 행렬. 크기가 n x m.
- Request : 각 스레드가 현재 요청 중인 자원의 개수를 나타내는 행렬. 크기가 n x m.
    - `Request[i][j] == k` 이면 현재 Ti가 Rj를 k개 요청 중이라는 의미.

1. Work와 Finish를 길이가 m과 n인 벡터라고 하자. Work = Available로 초기화 한다. `Allocation(i) != 0` (i = 0, 1, …, n-1)이면, `Finish[i] = false` 이고 아니라면 `Finish[i] = true` 이다.
2. 다음을 만족하는 i를 찾는다.
    a. `Finish[i] == false`
    b. `Request(i) ≤ Work`
3. 아래 연산을 수행하고 2번으로 간다.
    a. `Work = Work + Allocation(i)`
    b. `Finish[i] = true`
4. 어떤 i에 대해서 `Finish[i] == false` (0 ≤ i < n)라면 시스템이 교착상태에 빠진 것이다. 또한 Ti가 데드락에 빠진 것이다.

# Recovery from Deadlock

만약 데드락 탐지 알고리즘으로 데드락이 존재한다고 판명됐다면 다음 과정을 거친다.

1. 작동자에게 데드락의 발생을 알리고 **작동인이 수동으로 데드락을 해결**하도록 한다.
2. 시스템이 데드락으로부터 **스스로 복구**한다.

데드락을 복구하는 두 가지 방법은 다음과 같다.

1. 하나 이상의 스레드를 종료해서 순환 대기를 깨뜨린다.
2. 하나 이상의 교착상태에 빠진 스레드로부터 자원을 선점한다.

## 프로세스와 스레드 종료(process and thread termination)

- 데드락 된 모든 프로세스를 종료.
    - 확실하게 데드락 사이클을 깨뜨리지만 비용(여태까지 계산했던 것들을 버림)이 많이 드는 단점이 존재한다.
- 데드락 사이클이 제거될 때까지 한 프로세스씩 종료.
    - 각 프로세스가 중단된 후에 상당한 오버헤드(데드락 탐지 알고리즘을 호출해서 데드락을 확인)를 발생시킨다.
    - 오버헤드 : 어떤 처리를 하기 위해 들어가는 간접적인 처리 시간이나 메모리 등을 의미한다.

## 자원 선점(resource preemption)

이 방법은 프로세스로부터 자원을 연속적으로 선점해서 다른 프로세스에게 주는 과정을 데드락 사이클이 깨질 때까지 반복한다.

이를 위해 아래 세 가지 사항을 고려해야 한다.

1. 희생자 찾기(selecting a victim)
    - 프로세스 종료에 있어서, **비용을 최소화**하는 방향으로 선점의 순서가 결정되어야 한다.
2. 롤백(rollback)
    - 자원을 프로세스로부터 선점했다면 해당 프로세스는 실행되기 전의 **안전 상태**로 롤백되어야 한다.
    - 일반적으로 **전체 롤백**(total rollback, 프로세스를 종료시키는 것)한다.
3. 기아(starvation)
    - **비용 요인**에 주로 근거해서 희생자를 찾는다면 같은 프로세스가 항상 희생자로 선택되어 기아가 발생할 수 있다.
    - 한 희생자 당 **유한한 횟수**만큼 선정되어야 하는데, 제일 흔한 방법은 **롤백의 횟수**를 **비용 요인에 포함**시키는 것이다.
