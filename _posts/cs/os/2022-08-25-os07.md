# Classic Problems of Synchronization

이 섹션에서는 동시성 제어 문제의 예로 많은 동기화 문제를 제시한다. 이런 문제들은 거의 모든 새로 나온 동기화 도구들을 검증하는데 사용되고있다. 이 문제의 해결책으로 우리는 세마포어를 사용하지만 실제 구현에서는 이진 세마포어 대신 뮤텍스 락을 사용할 수 있다.

## 한정 버퍼 문제(bounded-buffer problem)

### **한정 버퍼 문제(생산자-소비자 문제)**

- 데이터를 보관하는 버퍼가 있을 때 생산자가 저장할 공간이 없거나 소비자가 소비할 데이터가 없는 문제

여기에서 생산자와 소비자는 다음의 자료구조를 공유한다.

```c
int n;
semaphore mutex = 1;
semaphore empty = n;
semaphore full = 0;
```

mutex 변수는 상호배제를 위한 변수, empty와 full 은 버퍼의 상태를 나타내는 변수.

```c
while (true) {
		...
	/* produce an item in next produced */
		...
	wait(empty);
	wait(mutex);
		...
	/* add next produced to the buffer */
		...
	signal(mutex);
	signal(full);
}
```

[생산자 프로세스의 구조]

```c
while (true) {
	wait(full);
	wait(mutex);
		...
	/* remove an item from buffer to next consumed */
		...
	signal(mutex);
	signal(empty);
		...
	/* consume the item in next consumed */
		...
}
```

[소비자 프로세스의 구조]

위 구조를 살펴보면 생산자와 소비자가 대칭구조를 이룬다는 것을 알 수 있다.

## 독자-저자 문제(readers-writers problem)

### **독자-저자 문제**

- 다수의 독자와 다수의 저자가 하나의 공유 데이터베이스를 사용할 때 발생할 수 있는 문제
- 저자 + (저자 또는 독자)가 동시에 데이터베이스에 접근하면 문제가 발생
- **독자** : 데이터를 읽기만 하는 프로세스
- **저자** : 데이터를 읽고 수정하는 프로세스

이 문제를 방지하려면 저자가 DB 접근을 독점적으로 하게 해야한다.

이 문제에는 두가지 변형이 있다.

## 첫 번째 변형

저자가 기다리고 있어도 독자가 기다려주지 않는다.

독자 프로세스가 계속해서 들어오면 **저자 프로세스는 기아(Starvation) 상태**에 빠질 수 있다.

## 두 번째 변형

저자가 객체에 접근하려고 기다리고 있으면 독자는 더 이상 읽을 수 없다.

저자 프로세스가 계속해서 들어올 경우 **독자 프로세스가 기아 상태**에 빠질 수 있다.

첫번째 문제를 해결하기 위해선 아래와 같은 자료구조를 독자 프로세스끼리 공유한다.

```c
semaphore rw_mutex = 1;
semaphore mutex = 1;
int read_count = 0;
```

```c
while (true) {
	wait(rw_mutex);
		...
	/* writing is performed */
		...
	signal(rw_mutex);
}
```

[저자 프로세스의 구조]

```c
while (true) {
	wait(mutex);
	read_count++;
	if (read_count == 1)
		wait(rw_mutex);
	signal(mutex);
		...
	/* reading is performed */
		...
	wait(mutex);
	read_count--;
	if (read_count == 0)
		signal(rw_mutex);
	signal(mutex);
}
```

[독자 프로세스의 구조]

### reader-writer lock

이러한 문제의 해결책들은 일반적으로 **락**을 제공한다. read, write 접근 중에 원하는 것에 따라 mode를 구체화해서 락을 획득한다. 예를 들어 데이터를 읽기만 하고싶은 프로세스는 read모드로 락을 요청하고, 데이터를 수정하고 싶은 프로세스는 write 모드로 락을 요청한다.

read 모드로 락을 획득하는 것에는 프로세스 개수의 제한이 없지만 write 모드는 한번에 한 프로세스만 획득할 수 있도록 제한된다.

락의 제공은 아래와 같은 경우에 유용한다.

- 어떤 프로세스가 읽기만 하고 어떤 프로세스가 쓰기만 하는지를 알기 쉬운 애플리케이션.
- 저자보다 독자가 많은 애플리케이션.

## 식사하는 철학자들 문제(dining-philosophers problem)

### 식사하는 철학자들 문제

자신의 일생을 생각하고 먹는데 쓰는 다섯 명의 철학자가 있다고 가정하자. 이들은 원형 테이블에 있는 다섯 개의 의자에 둘러앉아있다. 가운데에는 밥그릇이 있고 젓가락은 다섯 개가 있다. 이들이 생각할 때는 다른이들과 상호교류하지 않는다. 시간이 지날수록 철학자들은 배가 고파지고 제일 가까운 두 개의 젓가락을 집어들려고 한다. 한 철학자는 한번에 오직 한 젓가락만 집어들 수 있다. 물론 옆사람의 손에 들린 젓가락을 가져올 수도 없다. 배고픈 철학자가 동시에 두 개의 젓가락을 가졌다면 젓가락을 내려놓지 않고 밥을 먹는다. 다 먹었다면 두 젓가락을 내려놓고 다시 생각하기 시작한다.

이 문제는 대규모의 동시성 제어 문제의 예시가 되기 때문에 고전적인 동기화 문제로 여겨진다. 이 문제는 여러 자원을 여러 프로세스에 deadlock-free하고 starvation-free하게 할당할 필요성에 대해 간단하게 설명한다.

### 세마포어 해결책

단순한 해결책 중 하나는 각 **젓가락**을 **세마포어**로 표현하는 것이다. 철학자는 `wait()` 호출을 통해 젓가락을 집으려 시도하고 `signal()` 호출을 통해 젓가락을 내려놓는다. 따라서 공유 데이터는 `semaphore chopstick[5]` 이고 모든 chopstick의 요소들은 1로 초기화된다. 철학자 i의 구조는 아래와 같다.

```c
while (true) {
	wait(chopstick[i]);
	wait(chopstick[(i+1) % 5]);
		...
	/* eat for a while */
		...
	signal(chopstick[i]);
	signal(chopstick[(i+1) % 5]);
		...
	/* think for awhile */
		...
}
```

[철학자 i의 구조]

- 해결책의 문제점

이 방법은 인접한 두 사람이 동시에 먹지 않는다는 것을 보장하지만 교착상태가 발생할 수 있다는 문제가 있다. 다섯 철학자가 동시에 배가 고파서 왼쪽 젓가락을 들게되면 chopstick 요소는 0이 되고 이들은 오른쪽 젓가락을 잡으려고 영영 대기하게 된다.

- 해결책의 개선방법
    - 최대 4명의 철학자가 동시에 앉을 수 있도록 한다.
    - 한 철학자는 두 젓가락이 모두 사용가능할 때에만 젓가락을 들도록 한다. (젓가락 드는 과정은 임계 구역 안에서 행해져야 한다.)
    - 비대칭 해결법 : 홀수번 철학자는 왼쪽 젓가락부터 들고 오른쪽을 들게 하고, 짝수번 철학자는 그 반대의 순서로 들게 한다.

6.7장 모니터 파트에서는 데드락에서 자유로운 철학자 문제의 해결책이 제시되었다. 하지만 이 문제의 해결책은 반드시 철학자가 굶어죽을 가능성을 막아야한다는 것을 기억하자. **데드락에서 자유롭다**고 **굶어죽지 않는 것은 아니다.**

### 모니터 해결책

철학자가 두 젓가락이 모두 사용가능할 때만 젓가락을 집어들도록 한다. 이 해결책을 구현하려면 철학자들의 상태를 **세 단계(생각하는 상태, 배고픈 상태, 먹는 상태)**로 구분해야 한다. 이렇게 하면 철학자가 배고프지만 젓가락을 얻을 수 없을 때 지연하도록 한다. 이를 위해 다음과 같은 자료 구조를 사용한다.

`enum {THINKING, HUNGRY, EATING} state[5];`

젓가락의 분포는 `DiningPhilosophers` 모니터에 의해 다음과 같이 제어된다.

```c
monitor DiningPhilosophers
{
	enum {THINKING, HUNGRY, EATING} state[5];
	condition self[5];

	void pickup(int i) {
		state[i] = HUNGRY;
		test(i);
		if (state[i] != EATING)
			self[i].wait();
	}
	void putdown(int i) {
		state[i] = THINKING;
		test((i + 4) % 5);
		test((i + 1) % 5);
	}
	void test(int i) {
		if ((state[(i + 4) % 5] != EATING) &&
			(state[i] == HUNGRY) &&
			(state[(i + 1) % 5] != EATING)) {
				state[i] = EATING;
				self[i].signal();
		}
	}
	initialization code() {
		for (int i = 0; i < 5; i++)
			state[i] = THINKING;
	}
}
```

[철학자 문제의 모니터로의 해결법]

각 철학자는 먹기 전에 `pickup()` 을 반드시 호출해야한다.이 작업은 철학자의 진행과정을 막을 수 있다. 동작이 성공적으로 완료되면, 철학자는 먹는다. 그 후에 철학자는 `putdown()` 을 호출한다. 정리하면, 철학자 i는 `pickup()` 과 `putdown()` 을 아래와 같이 순서대로 호출한다.

```c
DiningPhilosophers.pickup(i);
			...
			eat
			...
DiningPhilosophers.putdown(i);
```

위의 내용들을 볼 때, 이 해결책이 어떤 인접한 두 사람도 동시에 먹지 않고, 데드락도 발생하지 않는다는 것을 보장한다는 사실은 쉽게 알 수 있다. 하지만 철학자들이 굶어죽을 가능성은 여전히 존재한다.

# Alternative Approaches

멀티코어 시스템의 등장으로 멀티프로세싱 코어를 활용하는 **동시 애플리케이션**(concurrent application)을 개발하라는 압력이 세지고 있다. 하지만 동시 애플리케이션은 경쟁 상황과 라이브니스 위험의 리스크가 높다. 전통적으로 뮤텍스 락, 세마포어, 모니터가 이런 문제를 해결하기 위해 사용되었으나 프로세싱 코어가 늘어남에 따라 경쟁 상황과 데드락으로부터 자유로운 애플리케이션을 디자인하는 것은 점점 어려워지고 있다. 이 섹션에서는 프로그래밍 언어와 하드웨어 모두에서 **thread-safe**한 동시 애플리케이션의 설계에 도움되는 기능들을 소개한다.

## Transactional Memory

이 개념은 데이터베이스 이론에서 유래됐지만 프로세스 동기화 전략으로 활용된다.

**메모리 트랜잭션**(memory transcation)이란 순차적인 **메모리 읽기-쓰기** 작업을 **원자적으로 처리**되게 한 것을 말한다. 모든 트랜잭션 작업이 완료되면 커밋되지만 그렇지 않다면 작업들은 중단되고 롤백된다.

예를 들어 `update()` 라는 공유 데이터를 수정하는 함수가 있다고 할 때, 전통적으로는 뮤텍스 락(또는 세마포어)을 통해 다음과 같이 구현된다.

```c
void update()
{
	acquire();

	/* modify shared data */

	release();
}
```

하지만 뮤텍스락과 세마포어같은 동기화 메커니즘의 사용은 **데드락** 같은 잠재적인 문제점을 수반한다. 또한 스레드의 수가 늘어남에도 전통적인 락은 그렇게 규모를 늘릴 수 없는데, 잠금 소유권을 위한 스레드 간의 경쟁이 매우 세지기 때문이다.

이런 문제점 때문에 대안으로 **트랜잭셔널 메모리(transactional memory)**가 프로그래밍 언어에 추가되었다.

예를 들어, `atoic{s}` 구조를 추가하여 작업 S가 트랙잭션으로 실행되는 것을 보장한다고 하자. 이렇게 되면 `update()` 함수는 다음과 같이 새로 쓰여진다.

```c
void update ()
{
	atomic {
		/* modify shared data */
	}
}
```

이런 메커니즘을 락 대신 사용하게 되면 **원자성을 보장할 책임**이 개발자가 아닌 **트랜잭셔널 메모리 시스템**에 있다는 장점이 있다. 또한 락을 사용하는 것이 아니기 때문에 **데드락이 발생할 일도 없다**.

이러한 트랜잭셔널 메모리는 **두 가지 방법**으로 구현될 수 있다.

### 소프트웨어 트랜잭셔널 메모리 (Software Transactional Memroy, STM)

STM은 트랜잭션 블록 안에 **인스트루멘테이션 코드를 삽입**해서 작동한다. 이 코드는 컴파일러에 의해 삽입되고 코드의 어디서는 병렬로 실행하고, 어디서는 저수준의 락이 필요한지를 검사하는 방법으로 각 트랜잭션을 관리한다.

- **인스트루멘테이션(Instrumentation)**
    - 오류를 진단하거나 추적 정보를 쓰기 위해 제품의 성능 정도를 모니터하거나 측정하는 기능

### 하드웨어 트랜잭셔널 메모리 (Hardware Transactional Memroy, HTM)

HTM은 **하드웨어 캐시 계층**(hardware cache hierarchies)과 **캐시 일관성 프로토콜**(cache coherency protocols)을 사용해서 **프로세서의 캐시에 개별적으로 위치**한 공유 데이터와 관련한 **충돌을 관리하고 해결**한다.

## OpenMP

- **OpenMP** : **암묵적 스레딩**의 방법 중 하나로, 병렬로 처리되기를 원하는 **부분을 지정**해서 **병렬적으로 실행**한다.

이전에 배웠던 OpenMP 개념의 장점은 **스레드 생성과 관리**의 책임이 개발자에게 있지 않고 **OpenMP 라이브러리**에게 있다는 것이었다.

OpenMP에서는 코드를 병렬로 실행할 수 있게 하는 전처리 지시문이었던 `#pragma omp parallel` 와 함께 `#pragma omp critical` 이라는 지시문도 제공하는데, 이걸 사용한다면 코드의 일부 구역을 임계 구역으로 특정하여 한 개의 스레드만 한번에 활성화할 수 있도록 한다.

이 예시에 앞서서 공유 변수인 `counter` 이 `update()` 함수에서 수정될 수 있다고 가정하자.

```c
void update(int value)
{
	counter += value;	
}
```

위의 코드가 병렬 구역 내에 있거나 병렬 구역에서 호출된다면 `counter` 변수에 대해 경쟁상황이 발생할 수 있다. 이를 해결하기 위해 임계 구역 컴파일러 지시문(critical-section compiler directive)이 다음과 같이 사용될 수 있다.

```c
void update(int value)
{
	#pragma omp critical
	{
		counter += value;
	}
}
```

이 지시문은 이진 세마포어나 뮤텍스락과 같이 동작하는데, 구역 내에 한 번에 한 스레드만이 활성화되도록 보장한다. 구역 내에 한 스레드가 있는 상황에 다른 스레드가 진입하려고 시도한다면 내부의 스레드가 나갈 때까지 밖의 스레드는 차단된다.

이건 뮤텍스 락보다 사용하기 간편하다는 장점이 있다. 반면 단점으로는 경쟁상황이 발생할 수 있기 때문에 개발자가 직접 지시문을 사용해서 공유 데이터를 적절히 보호해야 한다는 점이 있다. 또한 뮤텍스 락과 비슷하게 동작하기 때문에 데드락의 가능성이 여전히 존재한다는 점도 있다.

위의 두 방법 외에도 **함수형 프로그래밍 언어(functional programming language)**를 사용하는 방법도 있다.
