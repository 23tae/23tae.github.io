협력 프로세스는 시스템에서 실행되는 다른 프로세스에 영향을 미치거나 영향을 받을 수 있는 프로세스이다. 협력 프로세스는 논리적 주소 공간(코드와 데이터)을 직접 공유하거나 공유 메모리 또는 메시지 전달을 통해서만 데이터를 공유할 수 있다. 그러나 공유 데이터에 동시에 액세스할 경우 데이터 불일치(data inconsistency)가 발생할 수 있다. 이 장에서는 데이터 일관성이 유지되도록 논리적 주소 공간을 공유하는 협력 프로세스의 순차적인 실행을 보장하기 위한 다양한 메커니즘에 대해 논의한다.

## 챕터 목표
- **임계 구역 문제**와 **경쟁 상태**에 대해 설명한다.
- 임계 구역 문제에 대한 하드웨어적인 해결책을 **메모리 장벽**, **compare-and-swap 작업**, **원자 변수(atomic variables)**를 사용하여 설명한다.
- 임계 구역 문제를 해결하기 위해 **뮤텍스 락**, **세마포어**, **모니터**, **조건 변수**가 어떻게 사용되는지를 설명한다.
- 임계 구역 문제를 해결하는 도구들을 다양한 경합 상황(contention scenarios)에서 평가한다.

# Background

## 협력 프로세스

서로 영향을 주거나 받는다

**논리적 주소 공간**을 공유하거나(스레드 등), **데이터**를 공유한다.(shared memory, message passing 등)

- 물리적 주소 : 메모리 장치 안에서의 위치
- 논리적 주소 : CPU가 생성한 가상 주소
    - 논리적 주소 공간 : 프로그램의 논리적 주소의 집합

### 문제점

shared data에 동시에(concurrent) 접근한다면 **데이터 불일치**가 발생할 수 있다.

따라서 협력 프로세스들을 **순서대로 실행**시켜서 데이터 지속성을 유지해야 한다.

### 데이터의 불일치

전체 데이터가 여러 프로세스(또는 스레드)에게 **공유**되는 경우 **불일치**가 발생할 수 있다.

- 동시에 실행되는 경우(Concurrent execution)
    - 프로세스는 명령 스트림(instruction stream) 도중 인터럽트될 수 있다.
    - 프로세싱 코어가 다른 프로세스에게 할당될 수 있다.
- 병렬로 실행되는 경우(Parallel execution)
    - 여러 명령 스트림이 각각의 프로세싱 코어에서 동시에 실행된다.

두 프로세스가 개별적으로는 정확할지라도, 동시에 실행할 경우 올바르게 작동하지 않을 수 있다.

예시) **생산자-소비자 문제**

**생산자 프로세스**가 다음과 같고,
```c
while(true) {
	/* produce an item in next_produced */
	while(count == BUFFER_SIZE)
    	; /* do nothing */
    
    buffer[in] = next_produced;
    in = (in + 1) % BUFFER_SIZE;
    count++;
}
```

**소비자 프로세스**가 다음과 같다고 하자.
```c
while(true) {
	while (count == 0)
 	;	/* do nothing */
        
    next_consumed = buffer[out];
    out = (out + 1) % BUFFER_SIZE;
    count --;
    
    /* consume the item in next_consumed */
}
```

int형의 count 변수를 5로 초기화한 뒤에 생산자와 소비자 프로세스를 동시에 실행시켰다고 가정하자.
위의 코드에서 한쪽은 count를 1씩 더해나가고 한쪽은 1씩 빼나가기 때문에 항상 5가 되어야 할 것 처럼 보인다. 하지만 실제로 count의 값은 4, 5, 6 모두 될 수 있다.

이러한 결과가 발생한 이유는 `count++`, `count--`가 기계어에서 어떻게 처리되는지와 관련있다.

![Untitled](/assets/img/cs/os/os06/Untitled.png)

위 과정을 보면 count에 바로 1이 더해지지 않고, 레지스터를 사용해 기존의 값을 저장한 뒤 레지스터에 1을 더하고, 다시 count에 해당 레지스터를 저장하게 된다.

register1에 count를 저장하고 인터럽트가 발생한다면? context switch가 발생하고 register2에 1을 빼고 count에 4를 저장한 뒤에 다시 돌아와 해당 count에 6을 저장하게 된다. 결과적으로 1이 아닌 2가 더해진 꼴이 된다.

이와 같이 각 실행 단계가 뒤죽박죽될 수 있기 때문에 이러한 결과가 발생한다.

## 경쟁 상황 (Race Condition)

- 개념

여러 프로세스(또는 스레드)가 같은 데이터에 동시에 접근하여 조작하고, 그 실행 결과가 접근한 순서에 의존하는 상황

- 해결책

한번에 하나의 프로세스만 공유 데이터를 조작할 수 있도록 보장해야한다.
이를 위해 **프로세스(또는 스레드)의 동기화**가 필요하다.

# The Critical Section Problem

**임계 구역 문제(CSP)**

여러개의 프로세스로 이루어진 시스템이 있다고 할 때, 각 프로세스는 **임계 구역**(crtitical section)이라고 부르는 코드 조각을 가지고 있다. 임계 구역 안에서는 하나 이상의 다른 프로세스와 공유하는 데이터에 접근하고 업데이트할 수 있다.

- **코드 구역**
    
    일반적인 코드의 구성은 다음과 같다.
    
    - **진입 구역 (entry-section)** : critical-section에 진입하기 위해 허가를 요청한다.
	- **임계 구역 (critical-section)** : 공유 데이터에 접근하여 데이터를 수정할 수 있다.
    - **퇴출 구역 (exit-section)** : entry-section에서 대기하는 다른 프로세스가 critical-section으로 진입할 수 있도록 한다.
	- **나머지 구역 (remainder-section)** : 위의 세 구역에 해당하지 않는 나머지 코드들이 여기에 속한다.    
    ```c
    while (true) {
    	/* entry section */
    		/* critical section */
    	/* exit section */
    		/* remainder section */
    }
    ```

이 시스템의 중요한 특징은 **한 프로세스**가 자신의 임계구역에서 **실행**되는 동안에 **다른 프로세스들**은 각자의 임계구역에서 **실행될 수 없다**는 것.

프로세스가 작업을 동기화하는데 사용할 수 있는 **프로토콜을 설계**하여 데이터를 올바르게 공유할 수 있게 설정해야 한다.

## 해결책의 조건
다음 사항들을 만족해야 한다.

- **상호 배제 (Mutual Exclusion)**
    - 프로세스 $P_i$가 자신의 임계 구역에서 실행중이라면 다른 프로세스는 각자의 임계 구역에서 실행될 수 없다.
- **진행 (Progress)** (**deadlock**을 방지한다.)
    - 자신의 임계구역에서 실행중인 프로세스가 없고, 자신들의 임계구역에 진입하려는 프로세스들이 있을 때, 나머지 구역에 있는 실행 중이지 않은 프로세스들만 다음에 누가 그 임계구역으로 진입할 수 있는지를 결정하는 데 참여할 수 있다. 이 결정은 무기한 연기될 수 없다.
	- 임계 구역이 비어있을 때, 항상 접근이 가능하게 해준다.
- **한정 대기 (Bounded Waiting)** (**starvation**을 방지한다.)
    - 프로세스가 자신의 임계구역에 진입하려는 요청을 한 후 요청이 허용될 때까지 다른 프로세스들이 그들의 임계구역에 진입하도록 허용되는 횟수에 한계가 있다.
	- 자원을 요청한 스레드가 무기한으로 대기하는 것을 방지한다.

## 싱글 코어로의 해결책

가장 **쉬운** 해결책이지만 멀티 프로세서 환경에서는 **실현 불가능**한 방법.

- 공유 변수를 수정하는 동안 인터럽트 발생을 억제한다.
- 명령 리스트가 선점되는 현상 없이 순서대로 실행될 것이라는 확신을 얻을 수 있다.
- 다른 명령이 지시되지 않음으로서 공유 데이터를 예기치 않게 수정하는 것을 방지할 수 있다.

## 두가지 접근법

- **비선점형 커널**
    - 프로세스가 커널모드를 나가거나, 차단되거나, 자발적으로 놓기 전까지 커널 모드 프로세스가 동작한다.
	- 커널 자료구조에서의 **경쟁 상황**으로부터 본질적으로 **자유로움**
- **선점형 커널**
    - 프로세스가 커널 모드로 작동중에 다른 프로세스에 의해 CPU를 빼앗길 수 있다.
    - **설계하기 까다롭다**는 단점이 있지만 **응답성이 좋다**는 장점으로 인해 선호된다.

# Peterson’s Solution

임계 구역 문제에 대한 고전적인 소프트웨어 기반 해결책.

현대 컴퓨터 아키텍처가 기계어 명령어를 수행하는 방식 때문에 이 해결책이 제대로 동작한다는 보장은 없다. 하지만 임계 구역 문제를 해결하고 mutual exclusion, progess, bounded waiting를 다루는 프로그램을 디자인하는데 좋은 알고리즘적 설명을 해준다.

피터슨의 알고리즘은 **임계구역**과 **나머지 구역**을 번갈아 가며 실행하는 두 개의 프로세스로 한정된다.

프로세스가 번갈아 실행될 때, 피터슨 알고리즘은 다음과 같은 turn, flag 변수를 사용하여 문제를 해결한다. **turn**은 **임계 구역에 진입할 차례인 프로세스**의 이름을, **flag** 배열은 **프로세스가 임계구역에 진입할 준비가 되었는지**를 나타냄. 예를들어 `turn == i` 라면 프로세스 P(i)가 임계구역에 진입할 차례이고 `flag[i] == true` 이면 P(i)가 임계구역에 진입할 준비가 되었다는 의미이다.

```c
int turn;
boolean flag[2];
```

```c
while(true){
	flag[i] = true;
	turn = j;
	while(flag[j] && turn == j)
		;
	
		/* critical section */

	flag[i] = false;

		/* remainder section */
}
```

이 방식은 mutual exclusion, progress, bounded-waiting 모두 증명가능한 해결책이다.

- mutual exclusion : Pi는 `flag[j] == false` 이거나 `turn == i` 일 때만 임계 구역에 진입한다.
위에서 언급했다시피 피터슨 알고리즘은 최신 컴퓨터 아키텍처에서 제대로 동작한다는 보장이 없다. 그 이유는 시스템 성능 향상을 위해 프로세서나 컴파일러가 종속성이 없는 읽기 쓰기 작업을 계속해서 재주문하기 때문이다. 이 점은 싱글 스레드일때는 문제가 되지 않지만 멀티 스레드에서는 부정확한 결과를 불러올 수 있다.

# Hardware Support for Synchronization

앞에서는 임계 구역 문제에 대해 소프트웨어 기반의 해결책을 살펴봤다. 이번에는 해결책으로 세 가지의 하드웨어 명령어를 살펴볼 것이다. 이 기본 작업은 동기화 도구로 직접 사용될 수도 있고, 추상적인 동기화 메커니즘을 구성하는데 사용될 수도 있다.

- 메모리 장벽(memory barriers or fences)
- 하드웨어 명령어(hardware instructions)
- 원자적 변수(atomic variables)

## 메모리 장벽

앞서 피터슨 알고리즘에서는 시스템이 명령을 재정렬할 수 있다는 것을 살펴봤다. 컴퓨터 아키텍처가 어떤 메모리를 응용 프로그램에 제공할 것인지를 결정하는 방법을 **메모리 모델**이라고한다.

### 메모리 모델

- **강한 순서(Strongly ordered)** : 한 프로세서의 메모리 수정 사항이 다른 프로세서에게 바로 보인다
- **약한 순서(Weakly ordered)** : 한 프로세서의 메모리 수정 사항이 다른 프로세서에게 바로 보이지 않을 수 있다.

메모리 모델은 **프로세서 유형에 따라 다르므로** 커널 개발자는 공유 메모리 멀티프로세서의 메모리 수정의 가시성(다른 프로세서에게 보일지 안보일지)에 대해 어떠한 가정도 할 수 없다. 이 문제를 해결하기 위해 컴퓨터 아키텍처는 메모리의 **모든 수정 사항**이 다른 프로세서로 **전파**되도록 강제할 수 있는 **명령어**를 제공하여 메모리 수정 사항이 다른 프로세서에서 실행되는 스레드에 보이도록 한다. 이러한 명령어들을 **메모리 장벽**이라고 한다.

## Atomic Variables

원자적 조작(atomic operation) : 간섭받지 않는 최소 작업 단위

원자적 명령(atomic instruction)은 우리가 단어의 내용을 테스트하고 수정(test&modify)하거나 두 단어의 내용을 테스트하고 교환(test&swap)할 수 있도록 한다.
- 원자적 명령의 두 종류
    - `test_and_set()`
    - `compare_and_swap()`

### test_and_set()

```c
boolean test_and_set(boolean *target){
	boolean rv = *target;
  *target = true;
    
  return rv;
}
```

```c
do{
	while(test_and_set(&lock))
    ; /* do nothing */
    
    /* critical section */
    
    lock = false;
    
    /* remainder section */
    
}while(true);
```

### compare_and_swap()

```c
int compare_and_swap(int *value, int expected, int new_value){
	int temp = *value;
    
  if(*value == expected) *value = new_value;
    
  return temp;
}
```

```c
while (true) {
	while (compare_and_swap(&lock, 0, 1) != 0)
		; /* do nothing */

		/* critical section */

	lock = 0;

		/* remainder section */
}
```

일반적으로 `compare_and_swap()` 명령은 원자적 변수같은 다른 도구를 만드는 데 사용된다. 원자적 변수는 int형과 bool형과 같은 기본 데이터 타입에 대한 원자 연산을 제공한다.

CSP(Critical Section Problem)를 해결하기 위한 고수준의 소프트웨어 도구로는 다음의 것들이 있다.

- Mutex Locks : 동기화를 위한 가장 간단한 도구
- Semaphore : 더 강력하고, 편리하며, 효과적인 도구
- Monitor : mutex와 semaphore의 단점들을 해결한 도구
- Liveness : 프로세스의 진행을 보장하는 도구

# Mutex Locks

Mutex (**mut**ual **ex**clusion, 상호 배제)

앞선 하드웨어 기반의 해결책들은 복잡할 뿐 아니라 개발자들이 일반적으로 접근할수도 없다. 대신 OS 설계자들은 CSP를 해결하기 위해 고수준의 소프트웨어 도구를 개발한다. 그중 가장 간단한 도구가 **mutex lock**.

## 사용 목적

mutex lock을 사용하면 임계 구역(critical section)을 보호하고 이로 인해 경쟁 상태(race conditon)를 막을 수 있다.

## 작동 원리

- 프로세스가 임계 구역에 진입하기 전에 lock을 획득해야 한다. `acquire()` 함수 사용.
- 임계 구역을 빠져나갈 때 lock을 다시 돌려준다. `release()` 함수 사용.

```c
while (true) {

	acquire lock

		/* critical section */

	release lock

		/* remainder section */

}
```

- `acquire()`
    
    ```c
    acquire() {
        while (!available)
            ; /* busy wait */
        available = false;
    }
    ```
    
- `release()`
    
    ```c
    release() {
        available = true;
    }
    ```
    

위의 두 함수는 원자적으로(atomically) 호출되어야 한다. → mutex lock은 위의 하드웨어 해결책 중 CAS(compare and swap)으로 구현될 수 있다.

`available` 이라는 이름의 bool 변수도 사용하는데, 락(lock)을 사용할 수 있는지의 여부를 나타냄. 잠금이 사용 가능하면, `acquire()` 호출에 성공한뒤에 변수를 사용 불가능으로 바꿔놓는다. 락이 풀릴(release) 때까지 바깥 프로세스의 락 획득은 차단된다.

## 주요 단점

**busy waiting**이 필요하다는 단점이 있다. 하나의 프로세스가 임계 구역에 있으면, 임계 구역에 들어가려는 다른 프로세스들은 락을 획득(`acquire()`)할 때까지 무한 루프를 돌아야 한다.

이 점은 멀티프로그래밍 시스템(한개의 CPU로 여러 프로세스가 공유하는 시스템)에서는 확실히 문제가 된다. busy waiting하지 않았다면 유용하게 사용되었을 CPU 사이클을 낭비하게 된다.

## 주요 장점

busy waiting을 다른 말로 spin-lock이라고도 한다. 프로세스가 락이 사용가능해 질때까지 회전(루프를 돎)하기 때문. 위의 단점과 별개로 장점도 존재하는데, 이 방법은 상당한 시간이 소요되는 문맥 교환이 일어나지 않는다는 이점이 있다.

이는 멀티 코어 시스템의 특정 상황에서 선호되어지는 방법이다. 예를 들어 락이 짧은 시간동안만 되어야 한다면 한 스레드는 다른 스레드가 다른 코어에서 임계 구역을 진행하는 동안 스핀을 돌 수 있다. 이 외에도 현대 멀티 코어 시스템에서 spin lock은 다양한 OS에서 사용된다.

# Semephores

semaphore : 신호라는 뜻

세마포어(semaphore)는 뮤텍스 락과 유사하게 동작할 수 있지만, 동기화를 위해 더욱 정교한 방법을 제공할 수도 있다.

## 관련 함수

**semaphore S** : 초기화와 별개로, `wait()`, `signal()`이라는 두 개의 원자적 조작에 의해서만 접근될 수 있는 int형 변수.

wait과 signal은 P와 V로도 불림. (최초 개발자인 Edsger Dijkstra의 국가인 네덜란드어 proben, verhogen에서 유래되었다.)

- `wait()`
    
    ```c
    wait(S) {
        while (S <= 0)
        	; // busy wait
        S--;
    }
    ```
    
- `signal()`
    
    ```c
    signal(S) {
        S++;
    }
    ```
    
    위의 두 함수의 연산에서 S값의 수정은 원자적으로 실행되어야 한다. (=한 프로세스가 세마포어 값을 수정할 때 다른 프로세스는 동일한 세마포어 값을 동시에 수정할 수 없다.)
    
    `wait()` 에서, S값을 체크하는 것(`S <= 0`)과 S값의 수정(`S--`)은 인터럽트 없이 실행되어야 한다.
    

## 주요 종류

- 카운팅 세마포어(counting semaphore)
    - 범위가 제한되지 않는다.
- 이진 세마포어(binary semaphore)
    - 범위가 0과 1 사이이다.
    - mutex lock과 동일하게 동작한다.(두 개 사이에서 동작)

## 세마포어의 사용

카운팅 세마포어는 한정된 인스턴스로 구성된 **리소스**에 대한 **접근을 제어**하는데 사용될 수 있다.

- 세마포어는 사용 가능한 리소스의 수로 초기화 된다.
- 리소스를 사용하고자 하는 프로세스는 세마포어에어 `wait()` 연산을 수행한다.(카운트 감소)
- 프로세스가 리소스를 놓아주면 `signal()` 연산을 수행한다. (카운트 증가)
- 세마포어의 카운트가 0이 되면 모든 리소스가 사용되었다는 뜻
    - 카운트가 0보다 커질 때까지 리소스를 사용하려는 프로세스들이 차단된다.

카운팅 세마포어를 통해 다양한 동기화 문제를 해결할 수 있다.

예를 들어 동시에 실행(concurrent)되는 프로세스 P1, P2와 그 statement인 S1, S2가 있다고 하자. S2가 S1이 끝나야만 실행되도록 하려면 아래와 같이 구현해야 한다.

```c
S1;
signal(synch);
```

```c
wait(synch);
S2;
```

이렇게 하면 P1이 `signal(synch)`를 호출(S1이 실행됐다는 의미)해야만 P2가 S2를 실행할 것이다.

## 세마포어 구현

뮤텍스 락이 busy waiting이라는 단점이 있었던 것처럼, `wait()`, `signal()` 세마포어 연산의 정의도 같은 문제를 가진다.

이 문제를 해결하기 위해 두 연산을 다음과 같이 수정할 수 있다.

- 프로세스가 `wait()`을 실행하고 세마포어의 값이 양수가 아닌 것을 발견한다.
- 프로세스가 자신을 멈춤(원래대로면 busy waiting에 들어감)
- suspend 작업을 통해 프로세스를 세마포어와 관련된 **대기 큐**로 이동시킴
- 프로세스의 상태를 **대기 상태**로 변경.
- 해당 프로세스의 제어권이 CPU 스케줄러(다음에 실행시킬 프로세스를 고르는 역할)에게로 넘어감
- 다른 프로세스가 `signal()` 을 실행시킴
- 대기중인 프로세스가 `wakeup()` 작업에 의해 재시작된다.
- 프로세스의 상태: 대기 상태 → **준비 상태**
- 프로세스가 **레디 큐**로 이동된다.

```c
typedef struct {
	int value;
	struct process *list;
} semaphore;
```

```c
wait(semaphore *S) {
	S->value--;
	if (S->value < 0) {
		add this process to S->list;
		sleep();
	}
}
```

```c
signal(semaphore *S) {
	S->value++;
	if (S->value <= 0) {
		remove a process P from S->list;
		wakeup(P);
	}
}
```

# Monitors

뮤텍스나 세마포어를 사용해도 타이밍 오류는 여전히 발생할 수 있다. 그 예로는 `wait()`과 `signal()` 연산의 순서가 뒤바뀌는 경우나 Critical Section이 끝나고 `signal()`대신 wait()이 호출되는 경우가 있다.

이런 타이밍 에러를 해결하는 방법으로는 단순한 동기화 도구들을 혼합해서 **고급 언어 구조**로 만드는 것이 있다. 그 예로 여기서 다뤄볼 것은 **모니터형(monitor type)**이다.

## 모니터의 사용

추상 자료형(ADT)는 데이터를 캡슐화 한다. 모니터형은 모니터 내부에서 프로그래머가 정의한 **상호배제가 보장**되는 동작들의 집합을 포함하는 ADT이다. 또한 모니터형은 인스턴스의 상태를 정의하는 변수들과 이를 조작할 수 있는 프로시저 또는 함수들의 본체도 같이 포함한다. 모니터형의 표현은 다양한 프로세스에서 직접적으로 사용될 수 없다. 따라서 모니터 내에서 정의된 함수는 모니터 내에서 로컬로 선언된 변수와 그 형식 매개 변수에만 접근할 수 있다. 마찬가지로 모니터의 로컬 변수도 로컬 함수에서만 접근할 수 있다.

# Liveness

프로세스가 실행 주기동안 진행되는 것을 보장하기 위해 만족해야하는 시스템의 속성

프로세스가 무한정 대기하는 상황을 **라이브니스 실패(liveness failure)**라고 부름. 대부분의 경우 낮은 성능과 낮은 반응속도를 보인다. 대표적으로 무한루프가 있다. 라이브니스 실패를 일으키는 두가지 상황은 **데드락**과 **우선순위 역전**이 있다.

## 데드락(Deadlock)

대기 큐가 있는 세마포어를 구현하면 두 개 이상의 프로세스들이, 오로지 대기 중인 프로세스들 중 하나에 의해서만 야기될 수 있는 `signal()`연산을 **무한정 기다리는 상황**이 발생할 수 있다. 이러한 상태에 있는 프로세스들을 **교착 상태(deadlock)**라고 한다.

아래와 같이 두 개의 프로세스 P0, P1이 있을 때, 이들이 두 개의 세마포어 S, Q에 접근하여 값을 1로 설정하게된다. P0이 `wait(S)`를 실행하고 P1이 `wait(Q)`를 실행한다고 가정하자. P0이 wait(Q)를 실행할 때, 프로세스는 P1이 `signal(Q)`를 실행할 때까지 기다려야한다. 비슷하게 P1도 P0가 `signal(S)`을 실행할 때까지 기다려야한다. 이처럼 `signal()` 연산은 실행될 수 없는 상황일 때 P0과 P1이 **교착상태**에 있다고 한다.

## 우선순위 역전(Priority Inversion)

우선순위가 가장 높은 프로세스가 제일 늦게 실행되는 상황

예를들어 우선순위가 L < M < H 의 순서인 프로세스가 있다고 하자. H가 필요로 하는 세마포어 S를 현재 L이 쓰고 있다. H는 L이 끝날 때까지 기다리지만 중간에 M이 실행가능해져서 프로세스 L을 선점하게 된다. 이 경우 M은 H보다 우선순위가 낮지만 H가 계속 대기하도록 한다. (M이 끝나고 L이 끝나야 H가 S를 쓸 수 있기 때문이다)

이런 상황은 두 개 이상의 우선순위가 있는 시스템에서만 발생할 수 있다.

**우선순위 상속**(priority inheritance protocol)을 통해 방지된다.

## 우선순위 상속(Priority Inheritance)

우선순위가 높은 프로세스가 필요로 하는 리소스에 액세스하는 모든 프로세스는 해당 리소스가 끝날 때까지 높은 우선순위를 상속받는다.

위의 예를 다시 보면, 이 경우 우선순위 상속을 통해 L이 H의 우선순위를 잠시 상속받아서 M으로부터 실행이 선점되지 못하도록 한다. L이 리소스 S를 다 쓰면 원래의 우선순위로 돌아오고 리소스 S가 사용가능해져서 H가 직후에 실행될 것이다.
