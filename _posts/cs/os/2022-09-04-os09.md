> Operating System Concepts, 10th Edition를 정리한 글입니다.

5장에서의 **CPU 스케줄링** 덕분에 CPU 사용률과 컴퓨터 응답속도를 향상시킬 수 있었다. 하지만 이러한 성능 향상을 체감하려면 많은 프로세스를 메모리에 위치(**메모리 공유**)시켜야 한다.

이 장에서는 메모리를 관리하는 다양한 방법에 대해 이야기한다. 메모리 관리 알고리즘은 원시적인 베어머신 접근법부터 페이징을 사용한 전략 등 다양하다.

## 챕터 목표

- **논리 주소**와 **물리 주소**의 차이와 주소를 변환하는 데 있어 **MMU(Memory Management Unit)**의 역할을 설명한다.
- **메모리를 연속적으로 할당**하기 위한 전략인 first-fit, best-fit, worst-fit을 적용한다.
- **내부·외부 단편화(internal·external fragmentation)**에 대해 설명한다.
- **TLB(Translation Look-Aside Buffer)**가 포함된 **페이징 시스템**에서 논리 주소를 물리 주소로 변환한다.
- **계층적 페이징(hierarchical paging)**, **해시 페이징(hashed paging)** 및 **역 페이지(inverted page)** 테이블을 설명한다.
- IA-32, x86-64 및 ARMv8 **아키텍처의 주소 변환**을 설명한다.

# Background

**메모리**는 현재 컴퓨터 시스템 운영의 핵심이다. 각 메모리는 고유한 주소를 가진 큰 **바이트의 배열(array of bytes)**로 구성된다. CPU는 프로그램 카운터의 값에 따라 메모리에서 명령어들을 fetch한다. 이러한 명령어들이 특정한 메모리 주소로/로부터 저장될 때 추가적인 로딩이 있을 수 있다.

여기서는 메모리 관리와 관련한 몇 가지 문제(기본 하드웨어, 가상 주소를 실제 물리 주소로의 바인딩, 논리 주소와 물리 주소의 차이점)에 대해 다룰 것이다.

## 기본 하드웨어

CPU는 **메인 메모리**와 각 프로세싱 코어에 내장된 **레지스터**에만 직접 접근할 수 있다. 따라서 실행 중인 모든 **명령어**들과 그것들이 사용하는 모든 **데이터**는 이러한 직접 접근 가능한 **스토리지(메인 메모리, 레지스터)에 있어야 한다.** 만약 메모리에 해당 데이터가 없다면 CPU가 작동하기 전에 그곳으로 옮겨져야 한다.

### 레지스터

각 CPU 코어에 내장된 레지스터는 일반적으로 **CPU 클럭**의 한 주기안에 접근가능하다. 일부 CPU의 경우 한 클럭동안 명령어를 해독하고, 해당 동작을 수행하는 과정을 한 번 이상 수행할 수 있다.

- **클럭(clock)** : CPU의 속도를 나타내는 단위로, 1초 동안 파장이 한번 움직이는 시간을 의미한다.

반면 **메인 메모리**는 메모리 버스(memory bus)에서 트랜잭션을 통해 접근하기 때문에 이처럼 빠르게 할 수 없다. 메모리 접근은 CPU 클럭 주기가 많이 걸리기 때문에 **프로세서가 멈출 수 있다.** 이 문제를 해결하기 위해 CPU와 메인 메모리 사이에 **빠른 메모리(캐시(cache) 등)를 추가**한다.

시스템이 올바르게 동작하기 위해 각 프로세스가 다른 프로세스의 **메모리 영역에 침범하기 않도록 보호**해야 한다.  이를 위해 프로세스가 유일하게 접근하도록 허용된 메모리 영역을 지정한다. 이때, 아래와 같은 두 레지스터인 **기준 레지스터(base register)**와 **상한 레지스터(limit register)**를 사용한다.

## 주소 바인딩(address binding)

- 바인딩 : 프로그램의 어떤 기본 단위가 가질 수 있는 구성요소의 **구체적인 값, 성격을 확정하는 것**을 말한다.

대부분의 시스템은 유저 프로세스가 물리 메모리 상의 어느 곳에서든 위치할 수 있도록 한다. 따라서 주소가 굳이 00000으로 시작할 필요는 없다.

유저 프로그램이 실행되기 까지 많은 과정을 거치는데, 이 과정에서 주소는 다양한 방법으로 표현된다.

- 소스 프로그램 상 : 심볼릭 주소(예. count 변수)
- **컴파일러** : 심볼릭 주소→ **재배치 가능한(relocatable) 주소**(예. 이 모듈의 시작 위치로부터 14바이트 떨어진 곳)
- **링커/로더(loader)** : 재배치 가능한 주소 → **절대(absolute) 주소**(예. 74014)

전통적으로, 명령어와 데이터를 메모리 주소 상에 바인딩하는 작업은 아래 과정 중 어떤 단계에서든 수행될 수 있다.

- **컴파일 시점(compile time)**
    - 프로세스가 메모리 내에 위치할 곳을 컴파일 시점에 알고있다면 **절대코드(absolute code)**가 생성될 수 있다.
    - 이후에 시작지점이 변경된다면 재컴파일을 해야한다.
- **로드 시점(load time)**
    - 프로세스가 위치할 곳을 컴파일 시점에 알지 못한다면, 컴파일러는 **재배치 가능한 코드**를 생성해야한다.
    - 시작 주소가 변경되면 코드를 다시 로드하기만 하면 된다.
- **실행 시점(execution time)**
    - 프로세스가 실행되는 동안 메모리 상에서 이동한다면, 바인딩은 **실행 시점까지 연기**된다.

## 논리 주소 공간 vs 물리 주소 공간

- 논리 주소(logical address) : CPU에 의해 생성된 주소
- 물리 주소(physical address) : 메모리 유닛이 볼 수 있는 주소. 메모리 상의 메모리 주소 레지스터(memory-address register)에 로드된다.

주소 바인딩이 **컴파일 시점**이나 **로드 시점**에 이루어지는 경우 논리 주소와 물리 주소는 동일하지만, **실행 시점에 이루어지는 경우는 다르다.** 이처럼 두 주소가 다른 경우 논리 주소를 **가상 주소(virtual address)**라고 부른다.

- 논리 주소 공간 : 프로그램에 의해 생성된 모든 논리 주소들의 집합
- 물리 주소 공간 : 논리 주소에 대응되는 모든 물리 주소들의 집합

**런 타임에 이루어지는 가상 주소에서 물리 주소로의 매핑**은 **MMU(memory-management unit)**라는 하드웨어 장치에 의해 이루어진다.

## 동적 적재(dynamic loading)

위에서 살펴본 내용에 따르면 프로세스를 실행하기 위해서는 **프로그램 전체와 프로세스의 모든 데이터가 물리 메모리에 위치**해있어야 한다. 이 때문에 프로세스의 크기도 최대 물리 메모리의 크기로 한정된다. 이처럼 낮은 **메모리 공간 활용도**를 높이기 위해서 **동적 적재**라는 방법을 사용한다.

동적 적재를 사용하면, 하나의 루틴은 호출되기 전까지 적재(load)되지 않는다. 모든 루틴은 디스크 상에 **재배치 가능한 적재 형식(relocatable load format)으로 존재**한다.

1. 메인 프로그램이 메모리에 적재되어 실행된다.
2. 한 루틴이 다른 루틴을 호출한다. 호출된 루틴이 이전에 적재되었는지를 확인한다.
    - 아직 적재되지 않은 경우
		1. relocatable linking loader를 호출해 해당 루틴을 메모리에 적재한다.
		2. 프로그램 주소 표에 해당 내용을 반영한다.

이러한 방식은 필요한 경우에만 루틴이 로드된다는 이점이 있다. 에러 루틴과 같이 자주 일어나진 않지만 많은 양의 코드가 필요한 경우에 유용하다.

## 동적 연결(dynamic linking) 및 공유 라이브러리

**동적 연결 라이브러리(Dynamically linked libraries, DLLs)**는 프로그램이 실행될 때 사용자 프로그램에 연결되는 시스템 라이브러리이다.

동적 적재에서는 적재가 실행 시까지 미뤄졌지만, 동적 연결에서 **연결은 실행 시기까지 미뤄진다.**

주로 C 라이브러리 같은 시스템 라이브러리에 사용된다.

- 장점
    - 동적 연결 기술이 없다면, 각 프로그램은 해당 언어 라이브러리의 복사본을 포함해야하는데, 이는 실행 이미지의 크기를 늘리면서도 메인 메모리의 낭비도 초래한다.
    - DLL 라이브러리는 여러 프로세스 간에 공유되기 때문에 메인 메모리에는 DLL의 인스턴스가 하나만 존재하게 된다.

# Contiguous Memory Allocation

메인 메모리에는 OS와 함께 다양한 유저 프로세스들이 위치해야하므로 최대한 효율적으로 메모리를 할당해야 한다.

메모리는 주로 두 파티션(partition, 부분)으로 나뉘는데, 하나는 **운영 체제(OS)**이고 다른 하나는 **유저 프로세스**이다. OS 파티션은 낮은 메모리 주소나 높은 메모리 주소에 위치시키는데, 대부분의 OS는 높은 메모리 주소에 위치시킨다.

연속 메모리 할당(contiguous memory allocation)에서, 각 프로세스는 다음 프로세스가 포함된 섹션에 인접한 섹션에 위치한다.

## 메모리 보호

**재배치 레지스터(relocation register)**는 **가장 작은 물리 주소**를 가지고 있고, **상한 레지스터(limit register)**는 **논리 주소의 범위**를 가지고 있다.

MMU 는 **논리 주소**에 **재배치 레지스터**의 값을 더해서 주소를 변환한다.

## 메모리 할당

메모리를 할당하는 가장 쉬운 방법은 **가변 분할(variable-partition)** 방식으로, 가변 크기 파티션에 프로세스를 하나씩 할당한다. OS는 메모리의 어느 부분이 사용가능하고 어느 부분이 사용중인지를 주시한다.

처음에는 모든 메모리가 가용한 하나의 큰 블럭(또는 구멍(hole))이지만 계속해서 사용하다보면 아래 그림처럼 다양한 크기의 여러 구멍을 가질 것이다.

![Untitled](/assets/img/cs/os/os09/Untitled%206.png)

[가변 분할]

프로세스가 시스템에 들어왔을 때, OS는 프로세스의 메모리 요구량과 가용 메모리를 고려해서 어떤 프로세스에게 메모리를 할당할지를 결정한다. 만약 프로세스가 요구하는만큼의 메모리가 가용하지 않다면 아래의 과정을 거친다.

- 프로세스를 거부하고 에러 메시지를 띄운다.
- 프로세스를 대기 큐로 옮긴다. 이후에 메모리가 해제되면 대기 큐에서 대기하는 프로세스를 확인한다.

이러한 가변 분할 방식에서 크기가 n인 프로세스가 들어갈 가장 적절한 구멍을 찾는 문제를 **Dynamic Storage-Allocation Problem**이라고 한다.

이 문제를 해결하는 세 가지 방법 다음과 같다.

- **최초 적합(first-fit)** : 최초로 발견한 알맞은 크기의 구멍에 할당한다.
- **최적 적합(best-fit)** : 크기가 맞는 구멍 중 제일 작은 곳에 할당한다.
- **최악 적합(worst-fit)** : 크기가 맞는 구멍 중 제일 큰 곳에 할당한다.

first, best가 시간, 활용률 면에서 worst보다 낫다. first와 best는 거의 성능이 비슷한데, 일반적으로는 first가 빠른 편이다.

## 단편화

메모리 할당의 최초, 최적 적합 방식 모두 외부 단편화 문제를 겪는다.

### 외부 단편화(external fragmentation)

프로세스가 메모리에 로드됐다가 없어지면 메모리 공간은 작은 조각들로 쪼개진다. **외부 단편화**란 메모리의 총량은 요청 메모리에 부합하지만 **가용 공간이 연속적이지 않는**(여러 조각으로 쪼개져있음) 현상을 말한다.

• 50% 규칙 : 통계적으로, 최초 적합의 경우 메모리의 1/3을 단편화 문제로 쓸 수 없게 된다.

### 내부 단편화(internal fragmentation)

프로세스가 **요구하는 공간보다 크게 할당**함으로써 내부에서 **사용하지 않는 공간이 발생**하는 현상을 의미한다.

### 외부 단편화의 해결책

- **메모리 압축(compaction)**
    - 메모리 내용을 섞어서 가용한 메모리를 하나의 큰 블록으로 만든다.
    - 재배치가 정적으로 어셈블리나 적재 시점에 이루어지면 불가능하다.
    - 재배치가 동적으로 실행 시점에 이루어져야 가능하다.
    - 비용이 많이드는 문제가 있다.
- **페이징(paging)**
    - 논리 주소 공간의 불연속을 허용해서 사용 가능한 모든 공간에 물리 메모리를 할당하도록 한다.

# Paging

페이징(paging)은 프로세스의 **물리 주소 공간의 불연속**을 허용하는 방법이다. 이 방법은 **외부 단편화**를 피하고 **메모리 압축의 도움을 받지 않는** 등 연속 메모리 할당의 주요 문제로부터 자유롭다. 이러한 큰 이점들로 인해 오늘날 대형 서버부터 모바일 기기까지 다양한 형태로 대부분의 OS에 사용되고 있다. 페이징은 **운영체제와 하드웨어 간의 협력**을 통해 구현된다.

## 기본 방법

- 물리 메모리를 **고정된 크기의 블록(프레임, frame)들**로 쪼개고, 가상 메모리를 **같은 크기의 블록(페이지, page)들**로 쪼갠다.
- 프로세스가 실행되면 페이지들은 소스(파일 시스템 또는 보조 기억 장치(backing store))에서 **가용한 프레임들에 로드**된다.

위 방법으로 인해 **가상 주소 공간이 물리 주소 공간과 완전히 분리**됐기 때문에 시스템의 물리 메모리가 2^64바이트보다 적어도 프로세스는 가상 64-bit 주소 공간을 가질 수 있다.

CPU가 생성한 모든 주소는 **페이지 번호(p)**와 **페이지 오프셋(d)**으로 나뉜다.

![Untitled](/assets/img/cs/os/os09/Untitled%207.png)

**페이지 번호**는 아래 그림처럼 **페이지 테이블의 인덱스**로 사용된다. **페이지 테이블**은 **물리 메모리의 각 프레임의 기준 주소(base address)**를 포함한다. **페이지 오프셋**은 **참조된 프레임에서의 위치**를 가리킨다.

따라서 **물리 메모리 주소**는 **프레임의 기준 주소(페이지 테이블)**와 **페이지 오프셋**으로 정의된다.

![Untitled](/assets/img/cs/os/os09/Untitled%208.png)

[페이징 하드웨어]

![Untitled](/assets/img/cs/os/os09/Untitled%209.png)

[가상·물리 메모리의 페이징 모델]

MMU는 CPU가 생성한 **논리 주소를 물리 주소로 변환**하기 위해 다음 과정을 거친다.

1. **페이지 번호 p**를 구해서 페이지 테이블의 인덱스로 사용한다.
2. 페이지 테이블에서 p에 대응되는 **프레임 번호 f**를 구한다.
3. 가상 주소의 **페이지 번호 p**를 **프레임 번호 f**로 교체한다.

페이지의 크기는 프레임의 크기처럼 하드웨어가 결정하는데, 2의 거듭제곱의 크기를 갖는다.

논리 주소 공간의 크기가 2^m 바이트 이고 페이지 크기가 2^n 바이트라면,

논리 주소의 상위 m-n 비트는 페이지 번호를 지정하고 하위 비트 n개는 페이지 오프셋을 지정한다.(아래 그림 참고)

페이징은 **동적 재배치(dynamic relocation)** 개념과 흡사하다. 모든 논리 주소는 페이징 하드웨어에서 물리 주소로 바인딩되어있다. 페이징을 사용하는 것은 각 프레임별로  기준 레지스터의 테이블을 사용하는 것과 유사하다.

페이징을 사용하면 어떤 free한 프레임도 할당될 수 있기 때문에 **외부 단편화**는 발생하지 않지만 **내부 단편화는 발생**할 수 있다. 만약 프로세스의 메모리 요구량이 페이지의 범위를 넘어가면 추가적으로 할당한 마지막 프레임은 완전히 차지 않을 것이다. 이를 막기 위해 페이지의 크기를 줄여야 할 것 같지만 실제로는 부하를 적게 받기 위해 페이지의 크기를 크게 가진다.

1. 프로세스가 시스템에 실행되기 위해 들어오면 그 크기와 페이지를 검사한다.
    - 프로세스가 n개의 페이지를 필요로 한다면 메모리에는 최소 n개의 프레임이 있어야 한다.
2. 프로세스의 첫 페이지가 프레임에 로드된다.
3. 프레임 번호가 해당 프로세스에 대한 페이지 테이블에 기록된다.
4. (끝까지 반복)

## 하드웨어 지원

페이지 테이블은 **프로세스 별**로 갖는 자료구조이기 때문에 해당 포인터도 **PCB에 저장**된다. 이후에 **CPU 스케줄러**가 실행할 프로세스를 선택할 때, 사용자 레지스터와 적합한 하드웨어 페이지 테이블을 저장된 사용자 페이지 테이블에서 다시 불러와야 한다. 이를 위해 페이지 테이블을 하드웨어적으로 구현하는 여러 방법이 있다.

### 고속의 하드웨어 레지스터로 구현하는 방법(fast register)

- 가장 간단한 방법이다.
- 페이지 주소 변환을 효율적으로 할 수 있다.
- 문맥 교환 시간이 증가하는 단점이 있다. (이 레지스터 또한 교환해줘야 하기 때문이다.)

이 방법은 테이블이 충분히 작을 때는 괜찮지만 현대 CPU들은 테이블이 크기 때문에 고속 레지스터로 구현할 수 없다. 대신 현대에는 **페이지 테이블을 메인 메모리에** 두고, **페이지 테이블 기준 레지스터(page-table base register, PTBR)**가 테이블을 가리키도록 한다. 이렇게 하면 문맥 교환 시에도 PTBR 하나만 교환하면 되기 때문에 시간을 줄일 수 있다.

### 변환 색인 버퍼(Translation Look-Aside Buffer, TLB)

페이지 테이블을 주기억장치에 저장해두면 문맥 교환은 빨라질 수 있어도, **메모리 접근 시간이 느려**질 수 있다. 이 해결책 **TLB**라고 불리는 특수하고 작은 **고속 검색 하드웨어 캐시**를 사용하는 것이다. TLB는 고속의 **연상 메모리(associative memory)**이다.

각 TLB의 입구는 두 부분(키(or 태그), 값)으로 이루어진다.

- **동작 과정** (아래 그림 참고)
    - 연상 메모리와 아이템이 주어진다.
    - 해당 아이템을 모든 키와 비교해 존재여부를 파악한다.
    - 존재 O
        - 대응되는 값을 반환한다.
    - 존재 X(TLB miss)
        - 페이지 테이블에 접근하여 프레임 번호를 알아낸다.
        - 해당 페이지 번호와 테이블 번호를 TLB에 저장하여 다음번 검색때 쓸 수 있도록 한다.

![Untitled](/assets/img/cs/os/os09/Untitled%2012.png)

[TLB를 통한 페이징 하드웨어]

- **주소 공간 식별자 (address-space identifier, ASIDs)**
    - 각 프로세스를 고유한 식별자로 구분을 해주어 프로세스 별로 주소 공간을 보호해준다.
    - 일부 TLB가 자신들의 입구에 저장한다.
- **적중률 (hit ratio)**
    - 원하는 페이지 번호가 TLB에서 발견되는 비율

## 보호 (protection)

**페이지 환경**에서는 각 프레임에 대한 **보호 비트**를 통해 **메모리를 보호**할 수 있다.

- 비트 1 (숫자는 구분을 위해 임의로 부여함)
    - 페이지가 읽기-쓰기용인지/읽기 전용인지를 구분하기 위해 사용된다.
    - 읽기 전용 페이지에 쓰려고 시도하면 하드웨어 트랩(또는 메모리 보호 위반)이 발생해서 OS로 보내진다.

 

- 비트 2 (**유효 무효 비트, valid-invlid bit**)
    - OS가 이 비트를 설정해서 페이지의 허용 여부를 확인한다.
    - 각 페이지 테이블의 입구에 붙는다.
    - 해당 비트가 valid → 해당 페이지는 프로세스의 논리 주소 공간에 있다.(=허용된 페이지)
    - 해당 비트가 invalid → 해당 페이지는 프로세스의 논리 주소 공간에 있지 않다.

## 공유 페이지 (shared pages)

페이징의 장점은 공동으로 사용하는 코드를 **공유**할 수 있다는 점이다.

만약 libc 라이브러리를 사용하려는 유저 프로세스가 40개 있다면 메모리는 총 80MB가 필요하다. 하지만 코드가 **재진입 코드 (reentrant code)**라면 공유가 가능해진다. 재진입 코드는 자신을 수정하지 않기 때문에 두 개 이상이 하나를 동시에 실행할 수 있다. 따라서 물리 메모리에는 하나의 libc만 올려두고, 각 프로세스들에 대한 페이지 테이블은 **같은 libc 물리적 사본**을 가리킨다. 이렇게 해서 80MB 대신 2MB만이 필요하게 된다.

# Structure of the Page Table

이 섹션에서는 페이지 테이블을 구조화하는 가장 흔한 방법들에 대해 알아볼 것이다.

- (참고) 메모리 단위 별 크기
    - KB = 2 ^ 10
    - MB = 2 ^ 20
    - GB = 2 ^ 30

대부분의 현대 컴퓨터 시스템은 매우 큰(2^32 ~ 2^64) 논리 주소 공간을 지원한다. 이런 환경에서는 페이지 테이블 자체도 상당히 커진다. 이 때문에 페이지 테이블을 작은 부분으로 나눠야 한다.

## 계층적 페이징(Hierarchical Paging)

이  방법은 **2단계 페이징 알고리즘**을 사용하는데, 페이지 테이블 또한 하나의 페이지가 된다.

![Untitled](/assets/img/cs/os/os09/Untitled%2015.png)

[2단계 페이지 테이블 구조]

32비트 논리 주소 공간과 페이지 크기가 4KB인 시스템이 있을 때, 논리 주소는 20비트의 **페이지 번호**와 12비트의 **페이지 오프셋**으로 나뉘어짐. 본 방법은 **페이지 테이블도 페이지화**하기 때문에 페이지 번호는 10비트의 페이지 번호와 10비트의 페이지 오프셋으로 또 나뉜다.

![Untitled](/assets/img/cs/os/os09/Untitled%2016.png)

p1은 바깥 페이지 번호의 인덱스이고 p2는 안쪽 페이지 테이블에서의 간격이다.(아래 그림 참고)

![Untitled](/assets/img/cs/os/os09/Untitled%2017.png)

[2단계 32-bit 페이징 구조에서의 주소 변환]

이 방법을 사용하면 바깥 테이블부터 안쪽 테이블 방향으로 주소 변환이 이루어지기 때문에 **forward-mapped** 페이지 테이블이라고도 한다.

## 해시 페이지 테이블(Hashed Page Tables)

이 방법은 **주소공간이 32-bit를 초과**하는 경우에 사용한다.

**해시 값**이 **가상 페이지 번호**가 된다.

- **해시 값** : 해시 함수에 의해 얻어지는 값
- **해시 함수(or 알고리즘)** : 임의의 길이의 데이터를 고정된 길이의 데이터로 매핑하는 함수

해시 테이블의 각 입구에는 같은 공간에 해시되는 요소들의 연결 리스트가 포함되어 있다. 각 요소는 세 개의 필드를 갖는다.

1. 가상 **페이지 번호**
2. 매핑된 **페이지 프레임의 값**
3. 연결 리스트 상의 다음 요소를 가리키는 **포인터**

### 동작 과정 (아래 그림 참고)

1. 가상 주소에서의 **가상 페이지 번호**가 해시 테이블에 해시된다.
2. 해시 페이지 테이블에서 가상 페이지 번호와 연결 리스트의 **첫 번째 요소의 1번 필드를 비교**한다.
3. 일치 O → 그에 대응되는 페이지 프레임(2번 필드)을 통해 물리 주소를 구한다.
4. 일치 X → 연결리스트의 다음 입구를 검색하여 매칭되는 가상 페이지 번호를 찾는다.

[해시 페이지 테이블]

# Swapping

프로세스 명령어들은 실행되기 위해 메모리에 있어야 한다. 하지만 프로세스 또는 그 일부분이 **보조 기억 장치(backing store)에 임시적으로 보내졌다가(swap)** 메모리에 불러와져서 계속 실행될 수 있다.(아래 그림 참고)

스와핑은 프로세스로 하여금 **시스템의 실제 물리 메모리 보다 큰 물리 주소 공간**을 가질 수 있도록 하여 **멀티프로그래밍**의 정도를 증가시킨다.

- 멀티프로그래밍의 정도(degree of multiprogramming) : 현재 메인 메모리에 존재하는 활성화 된 일(job)의 개수

![Untitled](/assets/img/cs/os/os09/Untitled%2020.png)

[디스크를 보조 기억 장치로 사용하는 두 프로세스 간의 표준 스와핑]

## 표준 스와핑

**모든 프로세스**를 메인 메모리와 보조 기억 장치(backing store) 간에 이동한다.

- 보조 기억 장치
    - 주로 빠른 저장소를 사용한다.
    - 프로세스의 모든 부분이 저장됐다가 회수될 수 있도록 충분히 커야한다.
    - 이 메모리 이미지로 직접 접근이 가능해야 한다.
- 장점
    - 물리 메모리를 확장해주어 시스템이 실제 물리 메모리보다 더 많은 프로세스를 처리할 수 있도록 한다.
    - 놀고 있는 프로세스가 주요 스와핑 후보이다.

## 페이징을 통한 스와핑

전통적인 UNIX 시스템에서는 표준 스와핑이 사용됐지만 현대 OS에서는 더 이상 사용되지 않는다. 전체 프로세스들을 메모리와 보조 기억 장치 사이에 스와핑하는데 걸리는 시간이 너무 오래 걸리기 때문이다.

이 방식은 프로세스 전체가 아니라 프로세스의 **일부 페이지들만을 스와핑**한다.

- **page out** : 메모리에서 보조 기억 장치로 페이지를 옮긴다.
- **page in** : 보조 기억 장치에서 메모리로 페이지를 옮긴다.

## 모바일 시스템에서의 스와핑

모바일 시스템은 PC나 서버와 달리 대부분 스와핑을 지원하지 않는다. 모바일 기기는 비휘발성 저장장치로 용량이 큰 하드 디스크 대신 플래시 메모리를 사용한다. 이러한 공간상의 제약 때문에 OS 디자이너가 스와핑을 꺼린다. 다른 이유로는 쓰기 횟수의 제한과 낮은 처리량 등이 있다.

- 대안 1 (IOS의 사례)
    - 사용 가능한 메모리가 특정 임계값 밑으로 내려가면 애플리케이션이 할당받은 메모리를 자발적으로 해제하도록 요청한다.
    - 읽기 전용 데이터(코드 등)는 메인 메모리에서 제거되고 이후에 필요할 때 플래시 메모리로부터 재로드된다.
    - 충분한 메모리를 확보하지 못한 애플리케이션은 운영 체제에 의해 종료될 수 있다.
- 대안 2 (안드로이드의 사례)
    - 전반전인 방식은 IOS와 유사하다.
    - 차이점은 메모리 확보를 실패한 프로세스가 종료되기 전에 애플리케이션의 상태를 플래시 메모리에 기록해둬서 이후 재시작을 빨리 할 수 있도록 한다.
