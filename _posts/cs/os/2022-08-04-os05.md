# Basic Concepts

## CPU 스케줄링

- 멀티프로그래밍 OS에서 기본적으로 사용하는 방식
- 멀티프로그래밍의 목표는 항상 일부 프로세스를 실행하여 **CPU 사용량을 증대**하는 것이다.

## CPU-I/O Burst Cycle

CPU 스케줄링의 성공 여부 프로세스의 속성에 달려있다.

### 프로세스의 실행

**CPU 실행**과 **I/O 대기**의 사이클로 구성되며 두 상태를 왔다갔다한다.

- CPU burst
- I/O burst
- CPU burst
- I/O burst
- …
- 최종 CPU burst → 시스템 요청(실행 종료)
    - **CPU burst** : CPU만 연속적으로 쓰면서 명령을 실행하는 단계
    - **I/O burst** : I/O를 실행하는 단계

![Untitled](/assets/img/cs/os/os05/Untitled.png)

CPU burst의 지속시간은 아래와 같은 도수 곡선의 형태로 분포한다.

- **I/O-bound 프로그램** : 짧은 CPU burst를 가진다. 긴 I/O burst를 가진다.
- **CPU-bound 프로그램** : 긴 CPU burst를 가진다.

![Untitled](/assets/img/cs/os/os05/Untitled%201.png)

## CPU 스케줄러

- 메모리에 있는 프로세스 중 실행 준비중인 프로세스를 선택하여 CPU를 할당하는 역할을 한다.
- 프로세스를 선택하는 다양한 기준 (= 스케줄링 알고리즘)
    - FIFO Queue (First-In, First-Out)
    - 우선순위 큐(Priority Queue)
    - 트리(tree)
    - 연결 리스트(linked list)

## 선점 / 비선점 스케줄링

### CPU 스케줄링 결정이 내려지는 경우

1. 프로세스가 실행(running) 상태에서 대기(waiting) 상태로 변경된 경우. (I/O 요청, `wait()` 호출 등)
2. 프로세스가 실행 상태에서 준비(ready) 상태로 변경된 경우. (인터럽트 발생 등)
3. 프로세스가 대기 상태에서 준비 상태로 변경된 경우. (I/O의 완료 등)
4. 프로세스가 종료된 경우.

1과 4의 경우 : 스케줄링의 선택의 여지가 없다. 새 프로세스가 선택되어 실행하게 된다. → **비선점**(non-preemptive) / 협력적 스케줄링  
비선점 스케줄링 : 프로세스가 종료되거나 대기 상태로 바뀔 때까지 CPU를 점유한다. 나머지 프로세스는 대기한다.  
2와 3의 경우 : 선택 사항이 있다. → **선점**(preemptive) 스케줄링  
선점 스케줄링 : 스케줄러에 의해 프로세스가 실행 도중에 CPU에서 빠져나올 수 있다.

## 디스패처

### 디스패처(dispatcher)

CPU 스케줄러에 의해 선택된 프로세스에게 CPU의 코어를 제어해주는 모듈

- 스케줄러 : 바꿀 프로세스를 선택만 한다.
- 디스패처 : 실제로 프로세서를 넘겨주는 역할을 한다.

### 디스패처의 작업

- 하나의 프로세스에서 다른 프로세스로 문맥(context)을 교환한다.
- 사용자 모드로 전환한다.
- 프로그램을 이어가기 위해 사용자 프로그램의 알맞은 위치로 이동한다.

### 디스패처 대기시간(dispatcher latency)

- 하나의 프로세스를 멈추고 다른 프로세스를 실행하기까지 걸리는 시간
- 디스패처는 매 문맥 교환시마다 호출되기때문에 **최대한 빨라야 한다.**

# Scheduling Criteria

## 스케줄링의 기준

- **CPU 효율(CPU utilization)** : CPU를 최대한 바쁘게 유지하도록 한다.
- **처리율(throughput)** : 단위 시간별로 종료된 프로세스의 개수
- **턴어라운드 시간(turnaround time)** : 프로세스를 실행시키는데 걸리는 시간(프로세스 제출부터 완료까지의 시간)
- **대기 시간(waiting  time)** : 레디 큐에서 대기한 시간의 합.
- **응답 시간(response time)** : 응답이 시작되는데까지 걸리는 시간.

# Scheduling Algorithm

CPU 스케줄링은 레디 큐에 있는 어떤 프로세스에서 CPU의 코어를 할당할 지를 결정하는 문제를 다룸.

## 스케줄링 문제의 해결책

- **FCFS**(First-Come, First-Served)
- **SJF**(Shortest Job First), **SRTF**(Shortest Remaining Time First)
- **RR**(Round-Robin)
- **Priority-based**
- **MLQ**(Multi-Level Queue)
- **MLFQ**(Multi-Level Feedback Queue)

## FCFS 스케줄링

First-Come, First Served. 먼저 도착하면 먼저 처리하는 방식. 가장 간단한 알고리즘이다.

- CPU를 먼저 요청한 프로세스가 CPU에 먼저 할당된다.
- FIFO 큐로 쉽게 구현할 수 있다.

- Burst Time : 쉬지않고 CPU를 할당했을 때 예상되는 처리시간

![Untitled](/assets/img/cs/os/os05/Untitled%203.png)

- 도착순서 : P1, P2, P3
    
    간트 차트로 나타내면 다음과 같다.
    
    ![Untitled](/assets/img/cs/os/os05/Untitled%204.png)

위와 같이 순서에 따라 평균 대기 시간이 크게 달라진다. FCFS는 평균 대기 시간이 최적의 경우가 아닐 뿐더러 프로세스의 CPU-burst time 간 차이가 클 수록 예측하기 힘들어진다.  
FCFS 스케줄링 알고리즘은 **비선점형**  
**CPU-bound** 프로세스가 1개, **I/O-bound** 프로세스가 여러 개가 있으면 **Convoy effect**가 발생한다.

- **Convoy effect**
    - 모든 프로세스가 하나의 큰 프로세스가 CPU에서 나올 때까지 기다리는 현상.
    - 짧은 프로세스가 먼저 할당됐을 때에 비해 낮은 CPU 효율, 낮은 디바이스 효율을 보인다.

## SJF 스케줄링

Shortest Job First. 다음 CPU-burst가 짧으면 먼저 실행하는 방식.  
각 프로세스의 다음 CPU-burst의 길이와 관련된다. CPU에 여유가 생기면 다음 CPU-burst가 최소인 프로세스에게 할당한다. 복수의 프로세스의 다음 CPU-burst가 동일하다면 FCFS로 해당 순서를 정한다.(그 중 먼저 들어온 프로세스를 먼저 할당한다.)

## RR 스케줄링

Round-Robin. 먼저 들어온 작업을 먼저 실행하되, 일정 시간(time quantam)마다 작업을 변경하는 방식. 선점형.  
스케줄러는 레디큐를 돌면서 각 프로세스를 CPU에 최대 1 **타임 퀀텀**동안 할당한다.

- 레디 큐는 원형 큐처럼 처리된다.
- **타임 퀀텀**(time quantum. 또는 time slice) : 10~100ms 사이의 길이로 된 시간단위.

### 2가지 경우의 수

- 프로세스의 CPU burst < 타임 퀀텀
    - 프로세스가 자발적으로 CPU를 내려놓다.
    - 스케줄러는 레디 큐에 있는 다음 프로세스로 이동한다.
- 프로세스의 CPU burst > 타임 퀀텀
    - 타이머가 작동해서 OS에 인터럽트를 발생시킴
    - 문맥 교환(context switch)가 일어남
    - 기존 프로세스는 레디 큐의 끝으로 옮겨진다.

## **Priority-based 스케줄링**

우선순위에 따라 작업을 실행하는 방식.  
CPU는 **높은 우선순위**(priority)를 가진 프로세스에게 할당된다. 우선순위가 같다면 FCFS 사용.  
SJF도 Priority-based 스케줄링의 일종이다. 여기서의 우선순위는 **다음 CPU burst의 역수**가 된다.

### 문제점

- Priority 스케줄링은 선점형, 비선점형 모두 가능하다.
- 문제점 : **Starvation** (또는 infinite blocking)
    - blocked 프로세스 : 실행할 준비가 됐지만 CPU를 기다리는 프로세스
    - 낮은 우선순위를 가진 일부 프로세스는 **영원히 대기**할 수 있다.
- 해결책 : **Aging**
    - 시스템에서 오래 대기하는 프로세스의 **우선순위를 점차 높여준다.**

### Priority 와 RR의 조합

## **MLQ** 스케줄링

Multi-Level Queue. **우선순위**별로 각각 **다른 큐**를 두고, 큐 별로 다른 스케줄링을 진행하는 방식.

![Untitled](/assets/img/cs/os/os05/Untitled%2021.png)

**프로세스의 타입**에 따라 다른 큐를 두는 프로세스 파티셔닝에 MLQ를 사용할 수도 있다. 예를 들어, foreground(interactive) 프로세스와 background(batch) 프로세스로 나누는 것이 일반적이다. 이 두 프로세스는 다른 응답시간을 필요로 하고 그에따른 스케줄링의 필요성도 다름. MLQ는 이를 해결해준다.

![Untitled](/assets/img/cs/os/os05/Untitled%2022.png)

## **MLFQ** 스케줄링

Multi-Level Feedback Queue.  
위의 MLQ의 경우 프로세스가 하나의 큐에 배정되면 이후에 바뀌지 않다. 이로 인해 스케줄링 비용은 줄어들지만 **유연성이 떨어진다**는 문제가 있다.  
반면 MLFQ는 프로세스의 큐간 이동을 허용한다. 또한 CPU burst 시간이 높을수록 우선순위를 낮게 준다. CPU-bound 프로세스와 같이 우선순위가 낮은 큐에 오래 머무는 프로세스는 높은 우선순위의 큐로 이동한다.(aging). 이로인해 starvation이 방지된다.

### MLFQ의 매개변수

- 큐의 개수
- 각 큐 별 스케줄링 알고리즘
- 프로세스를 더 높은 우선순위로 승격하는 방법
- 프로세스를 더 낮은 우선순위로 강등하는 방법
- 프로세스를 서비스해야할 때 어떤 큐에 배치할 지를 결정할 방법

# Thread Scheduling

스레드는 **사용자 스레드**와 **커널 스레드**로 나뉨.  
대부분의 현대적인 운영체제는 프로세스가 아니라 **커널 스레드**가 OS에 의해 스케줄링된다. 사용자 스레드는 스레드 라이브러리에서 관리하고, 커널은 이 과정을 모름. 사용자 스레드가 CPU에서 실행되려면 어떤 방식(간접적이거나 LWP(lightweigth process)를 사용하는 등)으로든 반드시 관련된 **커널 스레드에 매핑**되어야 한다.

- **LWP**
    - 각 커널 스레드마다 1개의 LWP가 매핑되어 있다.
    - 사용자 프로세스와 커널을 연결하는 인터페이스 역할을 한다.
    - 스레드 라이브러리는 많은 사용자 스레드로 부터 실행될 스레드를 선택하고, 스레드의 정보(우선 순위, PID 등)를 가지고 LWP를 구성한다. LWP는 다시 커널에 의해 스케줄링될 수 있는 커널 스레드로 매핑된다

## 경쟁 범위 (Contention Scope)

사용자 스레드와 커널 스레드의 차이점으로는 스케줄되는 방식이 있다.

### 프로세스 경쟁 범위 (PCS, Process Contention Scope)

프로세스 내에서 유저 스레드들 간의 경쟁 범위.  
시스템이 다대일, 다대다 모델을 구성할 때, 스레드 라이브러리는 사용자 스레드를 여유로운 LWP에서 실행되도록 스케줄한다. 이 과정에서 CPU에 대한 경쟁은 **동일한 프로세스 내**에 있는 스레드 간에 발생하기 때문에 이러한 개념을 **프로세스 경쟁 범위**라고 한다.

### 시스템 경쟁 범위 (SCS, System Contention Scope)

시스템 내에서 커널 스레드들 간의 경쟁 범위.  
CPU에 스케줄 될 **커널 스레드**를 선택하기 위해 커널은 **시스템 경쟁 범위**를 이용한다.

# Real-Time CPU Scheduling

리얼 타임 스케줄링은 2가지로 구분된다.

- Soft real-time systems
    - 중요한 실시간 프로세스가 스케줄 되는 시점에 관해 어떤 보장도 하지 않다.
    - 중요한 프로세스가 중요하지 않은 프로세스들에 비해 우선권을 갖는다는 것만 보장한다.
- Hard real-time systems
    - 작업은 반드시 마감시간까지 서비스되어야 한다.
    - 마감시간이 만료되면 전혀 서비스 받지 않은 것으로 된다.
